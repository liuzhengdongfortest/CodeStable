# cs-plan

把已经聊清楚的需求，从 `talks/` 推到 `epics/` 或 `issues/`。

## 背景

`cs-talk` 负责把模糊想法聊清楚，但它不创建出口。否则讨论会很容易一边澄清、一边过早承诺实现路径。

`cs-plan` 站在它后面：读取讨论整理，回到当前系统和 requirements，判断这件事是一个可关闭 issue，还是大到需要先进入 epic 拆分。它不是做代码的技能，而是把“想清楚了”变成“可以被执行和关闭”的事项。

## 原则

### 先校准上下文

开始前确认四类东西：

- `cs` 技能：如果当前对话/执行上下文里还没读过，就先读一次，理解 CodeStable 的实体、技能边界和 `cs-plan` 在 `cs-talk` 与 `epics/issues` 之间的位置；如果已经读过，就复用这张体系地图。
- `.cs/facts.md`：如果当前对话/执行上下文里还没读过，就读一次；如果已经读过且没有迹象表明它刚被修改，就复用已知事实。
- `.cs/talks/`：优先读取用户指定的 talk；没指定时找最近或最相关的 talk，但要向用户说明判断依据。
- `.cs/requirements/`：读和本次需求相关的当前需求、约束、领域词汇和设计取舍。

如果仓库里还没有 `.cs/`，提醒用户先接入 CodeStable。不要自己发明一套临时目录。

### 判断出口

能直接进 `issues/` 的需求，应该同时满足：

- 目标能用一句话讲清。
- 范围能分清包含和不包含。
- 做完以后有明确验证方式。
- 不依赖一组未定的架构取舍。
- 可以由一次执行闭环完成。

进入 `epics/` 的需求，通常满足任一条：

- 会拆出多个 issue，且 issue 之间有依赖关系。
- 横跨多个模块或用户工作流，需要先定边界和接口。
- 关键取舍还会影响拆分方式。
- 直接写成一个 issue 会把目标、设计、验收全搅在一起。

如果连真问题、约束或边界都还说不清，回到 `cs-talk`，不要硬拆。

### 让事项可关闭

issue 和 epic 都要服务关闭，不要写成愿望清单。

- issue 写清：类型、目标、范围、当前证据、方案判断、验证、关闭回写。
- epic 写清：目标、背景、关键取舍、子 issue、依赖、整体验收、关闭回写。
- 每个 issue 只承载一条能独立验证的变更；不要把“顺手做掉”的内容塞进去。
- 从 talk 里搬来的内容要改写成事项语言：问题、约束、证据、验证，而不是会议纪要。

计划时可以查看相关代码来确认边界和模块归属。看到架构取舍时，优先判断能力归属、接口边界、模块是否足够深，以及接缝是否真实；但不要在 plan 阶段实施改动。

### 落到本地事项

事项默认落在项目本地的 `.cs/` 里：小需求写入 `.cs/issues/YYYY-MM-DD-{slug}.md`；大需求写入 `.cs/epics/YYYY-MM-DD-{slug}.md`，并列出子 issues。

落完后回写源 talk 的“下一步”或追加“计划出口”：链接到生成的 issue / epic，说明为什么这样分流。稳定下来的需求、约束和领域词汇，只在关闭事项时再回写 `requirements/`。

## 应用场景

- **talk 已经整理完，用户要进入落地。** 读取 talk，把它转成 issue 或 epic。
- **用户说“这个太大，帮我拆一下”。** 先判断是不是 epic，再拆成可关闭 issues。
- **用户说“直接建个 issue”。** 仍然检查目标、范围、验证是否足够清楚；不清楚就先补问。
- **用户拿一段对话或想法要求规划。** 先整理成计划判断；必要时要求补一个 talk，再继续。

不适用：

- **不负责澄清真实诉求。** 想法还模糊时回到 `cs-talk`。
- **不写代码。** 计划落完后等用户调用执行技能。
- **不长期保存需求真相。** requirements 只接收关闭事项后毕业的约束和取舍。
- **不主动扫描仓库找活。** 没有用户给定的 talk、需求或事项，就不凭空规划。
