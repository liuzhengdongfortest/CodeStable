# cs-issue

遵循 `.codestable/convention.md`。

issue = 一件可关闭的变更。类型用 tag 分（`bug` / `refactor` / `feature` / `chore`），流程同一套，差别只在记录详略。载体（GitHub issue / 本地 md）由项目配置决定。

写 / 改代码本身的纪律归 `cs-code`，设计形状归 `codebase-design`。本技能管的是**变更的闭环**。

## 闭环四步（按需压缩，不是四个阶段技能）

1. **记清楚**——这件事是什么、怎么复现 / 现状为何不对。脑子里改完就忘 = 三个月后无凭证
2. **定位**——根因 / 影响范围。bug 找真因不是表面现象；refactor 圈死"行为不变"的边界
3. **改 + 验证**——按定位定点改（走 `cs-code`），验证真的好了 / 行为没变
4. **关闭回写**——若改变了某个取舍，回写 `context/` 对应取舍说明（走 `cs-context`）。**这是 DoD 一条，不是可选**

## 快速通道（简单变更直接走）

同时满足就压缩成「读代码 → 说根因 + 方案 → 用户确认 → 改 → 验证 → 收」：① 根因 / 边界一眼确定 ② 改动小（1-2 处）③ 无跨模块影响。

**不能快**：根因有多个候选 / 跨多模块 / 需先复现才能定位 / 用户要完整存档。一旦进标准路径默认不再二次改判，避免各步对路径各说各话。

## 记录详略

- 简单（快速通道）：一份收尾记录就够——改了什么、为什么、怎么验证的
- 复杂：把"记清楚 / 定位"也落下来作回溯凭证，下次类似问题不必从 git log 反推

记录形态（issue 正文各段 / 本地 md）由载体定。

## 类型边界

- **bug**：本该好的坏了（已有代码的错误行为 / 文档错 / 性能问题）
- **refactor**：行为不变的内部改善。圈死"行为不变"边界，别夹带功能改动
- **feature**：新增能力；大到塞不下 → `cs-epic` 先拆
- **灰色**：修 bug 中发现要加能力 → 先把 bug 闭环，再开 feature issue。不在一条 issue 里偷偷混范围

## 退出条件

通用骨架见 convention，另加：

- [ ] 验证通过（bug：复现消失；refactor：行为不变）
- [ ] 改变了取舍的话已回写 context
- [ ] 没夹带范围外 / 顺手改（顺手发现记一条留后续）
- [ ] 收尾记录留下（简单一份，复杂含定位）
