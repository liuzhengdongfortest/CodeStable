# cs-epic

遵循 `.codestable/convention.md`。

epic = 变更轴上颗粒最大的可关闭事项：一个父变更，孩子是一串带依赖 DAG 的子 issue。它和普通 issue 只差两点——**有子项 DAG**、**扛着前瞻性的接口契约**。

载体形态（GitHub 父 issue + sub-issue，还是本地 `epics/{slug}/`）由项目配置决定；本技能只管**怎么拆**。

## 为什么先架构再拆

架构不清楚就硬拆子 issue，结果是每条各自发明轮子、接口对不齐。顺序硬性：

1. **模块拆分**（概设）——拆成哪几个模块 / 组件，各自职责一句话
2. **接口契约**（详设）——模块间签名 / 数据结构 / 协议字段 / 错误码，写到子 issue 能拿来当**硬约束**的程度；无跨模块接口就明确写"无"
3. **才把方案拆成子 issue**

## 接口契约 = provisional context

接口契约是"还没为真的现状"——它活在 epic 打开期间，是所有子 issue 的硬约束输入（不能违反，要改先回来改 epic）。**epic 关闭时，稳定下来的契约 / 取舍毕业进 `context/`**（走 cs-context）。所以它不写进 context，留在 epic 里，直到为真。

子 issue 发现契约不合理 → 回来改 epic 再继续，不在子 issue 里偷偷绕开（绕开会让下一条同模块 issue 接到老契约二次冲突）。

## 拆解纪律

- **每条子 issue 能独立关闭**——能单独实现 / 验证。跑不下来颗粒度不对
- **依赖图是 DAG**——A 依赖 B 写清理由（"B 提供 XX 表结构"），不循环、不"A 先做"式空话
- **先标一条最小闭环**——做完能端到端跑通最窄路径的标第一条
- **明确不做的边界**——用户脑子里的"权限系统"可能含审计日志 / 脱敏，不做就写进"明确不做"
- **不替用户排优先级**——技术依赖之外的顺序用户拍板

## 接口

- 子 issue → 交给 `cs-issue` 一条条消费
- 关闭 → 契约 / 取舍毕业进 context（`cs-context`）
- 已有 epic 加子 issue / 改依赖 / 标 drop → update；drop 不删，留理由

## 退出条件

通用骨架见 convention，另加：

- [ ] 模块拆分：每个模块职责一句话讲清
- [ ] 接口契约写到可当硬约束级，或明确"无跨模块接口"（不许"待定"）
- [ ] 子 issue 依赖图是 DAG，最小闭环已标
- [ ] "明确不做"写了
- [ ] 没把单条子 issue 的内部实现细节写进 epic
