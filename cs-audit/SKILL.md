# cs-audit

遵循 `.codestable/convention.md`。

`cs-issue` 等你报 bug、`cs-context` 等你记取舍——但"我也不知道哪有问题，你先扫一遍"没人接。cs-audit 补这块。

**只发现，不定修，不立实体。** 清单是一次性透镜：用户挑哪条 → 升级成 issue（载体见 `attention.md`），没挑的不落常驻产物（否则噪音堆积）。

## 两类扫描

### 1. 通用隐患（读代码）

- **bug**：空值 / 边界 / 竞态 / 吞异常 / 无保护断言
- **安全**：注入 / 敏感数据暴露 / 权限校验缺失 / 不安全依赖
- **性能**：N+1 / 重复计算 / 无缓存热点 / 全量加载
- **可维护性**：超长函数 / 高复杂度 / 重复块 / 神秘常量 / 循环依赖

### 2. 对账 context（代码 ↔ 现状轴）

cs-audit 的独门价值——只有它系统查两轴一致性：

- **代码偏离取舍**：context 取舍说明写的约束，代码违反了？
- **抽象缺正当性**：代码里有抽象层，context 讲不出"这根轴为什么要变"？→ 过度设计嫌疑
- **取舍已过期**：context 描述的现状，代码已经不是这样？→ 提示回写

## 工作流

**范围收敛（不盲扫全仓）**：问用户关键词 / 模块目录 / 一段描述，有一样就起步。说"全扫"→ 推回最常改或最近出问题的区域。收敛后确认："扫 X，约 N 文件，看 Y 几维，OK？"

**扫描**：逐维真实 Glob/Grep/Read，每条发现必带证据（`文件:行号` + 片段 + 为什么是问题）。**每维上限 5 条**，逼你挑最值得报的，不 dump。对账维度必须真读 context，不凭记忆。

**triage + 升级**：每条打 性质 + 严重度（P0 必修 / P1 应修 / P2 可修）+ 置信度（high/medium/low）+ 建议动作。给优先级建议，用户选 → 升级成 issue（tag 对应性质）。cs-audit 自己不修、不建 `audits/` 目录。

## 守护规则

- 不盲扫全仓——先收敛范围
- 每条必有证据——禁"感觉不好 / 可能有问题"
- 置信度必标——不准全标 high
- 每维 ≤ 5 条
- 只发现不定修——"顺便修了"算越界
- 对账类必读 context——不凭记忆判断代码该长什么样

## 退出条件

通用骨架见 convention，另加：

- [ ] 范围已和用户确认
- [ ] 各维扫完（零发现也明说）
- [ ] 每条 file:line + 证据 + 置信度，每维 ≤ 5
- [ ] 给了按优先级的升级建议（选中的 → issue）
