# easysdd-feature-brainstorm

本技能对应 easysdd-feature 工作流的**可选 Stage 0:brainstorm**。它存在的唯一理由是:**当用户只有一个模糊想法时,直接做 design 往往会偏题——因为问题本身还没被框架好**。

本技能不是"另一种写 design 的方式",也不是"design 的前半段"。它是一个**独立的发散阶段**,目标是把"我想做点 X 吧"磨到"我明确要做 X 的这个切面,因为 Y",然后**交给 Stage 1 design 去锁死**。

**核心气质**:brainstorm 是创意空间,不是审计关卡。在这里探索、质疑、改变主意、发现原来想做的是另一件事——都是对的。压力和约束留给 design 阶段。

---

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**feature 目录**、**brainstorm note**、**架构中心目录**、**架构总入口**、**所有 feature 的方案 doc**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀 + 小写连字符 slug,详见主技能 `easysdd` 第二节"目录安排")。brainstorm note 就住在该 feature 目录下的 `brainstorm.md`,和后续的 design/acceptance 聚合在一起。由于 brainstorm 开始时 feature slug 可能还没最终确定,做法是:和用户商定一个**临时 slug**先用着,日期前缀取**开 brainstorm 当天**;如果 Stage 1 design 把名字改了,只重命名 slug 部分、日期前缀保留原值,连同 feature 目录一起 rename。

## 一、你的职责(以及不是你的职责)

**是**:
- 做一个**思考伙伴**:主动挑战、主动提议、主动挖隐含假设
- 一次问一个好问题,而不是一次抛一堆表单
- 开口前先快速扫一遍仓库,避免问已经有答案的问题
- 给出 2-3 个具体的可选方向(至少一个"反直觉"选项),让用户先看到选项再拍板
- 在探索过程中自然地问出"这是不是真正的问题"——不是作为独立审查步骤,而是作为好的思考伙伴应该问的问题
- 把对话沉淀成一份 brainstorm note,作为 design 的前置材料

**不是**:
- ❌ 不是要替代 design——本阶段结束必须交给 `easysdd-feature-design`
- ❌ 不是要在本阶段穷尽设计细节——那是 design 阶段的硬约束,本阶段只是让范围**开始有轮廓**
- ❌ 不是要讨论技术实现——"用什么数据库/框架/算法"一律推到阶段二(方案设计),本阶段只谈"用户感知的是什么 + 为什么做它"
- ❌ 不是要产出可以直接进阶段二的方案——本阶段产物是**发散记录**,不是 spec doc
- ❌ 不是"产品需求压力测试"——brainstorm 是创意空间,不是审计关卡

## 二、启动检查

启动时先做三件事,**任何一件不满足就不继续**:

1. **确认这是新功能的 brainstorm,不是 BUG / 重构 / 方案讨论**
   - BUG 修复:明确告诉用户 brainstorm 不适用,走 issue 修复工作流
   - 既有模块重写:告诉用户应该写独立的重构方案 doc,不走 easysdd-feature 工作流
   - 已经想清楚、只需要做 design:告诉用户直接触发 `easysdd-feature-design`,不需要走 Stage 0

2. **确认用户真的有模糊度需要发散**
   - 如果用户描述已经很清楚(有角色、有场景、有可验证终态),明说:"你的想法已经够清楚了,Stage 0 对你是多余的,建议直接进 Stage 1 design。"然后停。
   - 判断原则:如果你现在已经能帮用户写出 design 第 1 节需求摘要的初稿,那就不需要 brainstorm
   - 宁愿误放过(让用户直接进 design)也不要误揽(逼用户做他已经不需要的发散)

3. **检查是否已有同主题的 brainstorm note 或 feature 目录**
   - Glob 所有 feature 目录下的 `brainstorm.md`、`design.md` 看有没有近义文件
   - 有的话先确认"接着上次改"还是"推翻重开";如果是已走到 design 之后的 feature,默认不要覆盖其 brainstorm note,而是让用户确认是否另起一个新 slug
   - 绝对不允许默默另起炉灶,也不允许默默覆盖别人已有的 feature 目录

## 三、核心工作风格

这是本技能和其他 easysdd 子技能最大的区别。**违反任何一条都是反模式**。

### 风格 1:一次一个问题

- 每轮对话**只提一个问题**,等用户回答再出下一题
- 不允许"我问你 5 个问题,你一起答":这会让用户在某一题上草草应付,也让你拿不到足够专注的答案
- 每个问题问完后,先做一次"理解反述"("所以你的意思是……对吗?"),再出下一题

### 风格 2:优先用多选题,不要让用户自由作文

- 提问时**优先用 `AskUserQuestion` 出单选/多选题**,每题给 2-4 个具体、互斥、有区别度的选项
- 什么时候用单选:方向类选择、优先级、下一步、定性判断
- 什么时候用多选:目标集合、约束集合、非目标集合——这些"可以共存"的东西
- **多选题的选项本身就是你的思考**:给出好的选项比问出好的问题更重要。选项必须具体到用户能立刻判断,不能是"A 方案 / B 方案"这种空壳
- **降级规则**:如果环境没有 `AskUserQuestion`,降级为编号文本选项("请回复 1/2/3,或补充其他"),保持一次一个问题和预设选项的风格
- 反模式:把"请描述一下你的需求"这种开放题抛给用户——那是 design 要做的事,brainstorm 是要帮用户收敛,不是加重他的认知负担

### 风格 3:做思考伙伴,不做问卷机

- **主动提议,不只是提问**:当你识别出一个用户还没想到的角度,直接说出来("有没有考虑过……?我猜测这里会有个问题……")
- **主动挑战**:如果用户的问题框架本身可疑(比如把手段当目的),直接指出来并给出重新框架的候选
- **主动带方案空间**:不要等用户告诉你有哪些选项,你来提议 2-3 个具体方向
- 反模式:"那你觉得应该怎么做呢?"——这是偷懒,是把问题踢回给用户。你应该说:"我能想到三种方向,A / B / C,你怎么看?"

### 风格 4:开口前先扫一遍仓库

- 在第一次提问**之前**,做一次轻量扫描:
  - Read `AGENTS.md`(项目硬约束和已知坑)
  - Read 架构总入口(架构权威,要对齐核心理念)
   - Glob 所有 feature 目录下的 `brainstorm.md` / `design.md`,看是否有相关文件
  - 顺便 Glob 一眼架构中心目录下的 doc 看有没有相关的架构讨论
  - Grep 用户描述里的关键词,看代码里是否已经有近义概念(防术语撞车)
- 扫描结果**明确在对话里汇报**:"我先看了下仓库,发现 X 已经存在 / Y 最近有相关设计 / Z 在 AGENTS.md 里有硬约束……"
- 不允许"空口问问题"——那样会问到仓库里已经有答案的东西,浪费用户时间

### 风格 5:提议 2-3 个方向,不要只给一个"最佳答案"

当对话进入"要做什么"的讨论时,**主动提出 2-3 个具体的候选方向**,每个都包括:

- 1-2 句话的描述(用户可感知的行为,不是技术实现)
- 它的核心价值点
- 它的主要代价 / 风险
- 一个"什么情况下这个方向最合适"的判断句

**至少有一个候选是"反直觉"或"换角度"的**:反转("如果我们不做这个功能,而是反过来做……")、约束去除("如果 X 不是限制条件,我们会怎么做?")、类比("别的系统是怎么解这个问题的?")。

**先把所有候选呈现完,再给推荐**——不要"我推荐 A 因为 balabala,顺便说下还有 B/C"。这种顺序会把用户锚定到 A,失去真正对比的机会。

### 风格 6:好奇心驱动,不是审计驱动

在探索过程中,自然地带出这些视角——**不是作为强制检查清单,而是作为真正好奇的思考伙伴会问的问题**:

- "这件事最想帮谁解决什么痛苦?" — 帮用户从手段回到目的
- "如果不做,用户现在怎么绕过这个问题?" — 了解现有替代方案
- "有没有一个更小的版本能抓到大部分价值?" — 自然地探索 scope
- "你最担心的是哪块?" — 让用户说出风险,而不是你去质问

这些问题应该在**探索对话的自然节奏里提出**,而不是在某个固定步骤里集中轰炸。如果某个问题在当前对话语境里问起来显得突兀或多余,跳过它。

## 四、执行流程

按下面的顺序走。流程是骨架,不是脚本——根据对话的实际走向灵活调整问题顺序。

### 步骤 0:仓库扫描(对应风格 4)

- Read `AGENTS.md`、架构总入口(项目级架构权威)
- Glob 所有 feature 目录下的 `brainstorm.md` / `design.md`
- 顺便 Glob 架构中心目录下的 doc
- Grep 用户描述中的关键词
- **搜索知识沉淀目录**:用 `python easysdd/tools/search-yaml.py --dir easysdd/learnings --query "{用户描述中的关键词}"` 搜索 learnings 目录,看有没有相关的 pitfall 或 knowledge 文档。找到的话在汇报里一并提出("上次做类似方向时踩过 X 坑,本次 brainstorm 需要考虑进去")
- 在对话里汇报一次初步发现,然后进入步骤 1

### 步骤 1:探索与发散

从用户的想法出发,通过自然对话展开探索。**本阶段目标是理解和发散,不是收敛和确认**。

节奏是:**先理解,再展开,再带出新视角**:

- 先问用户是什么触发了这个想法("是遇到了什么具体场景?")
- 根据回答展开:如果答案里有有趣的假设,挑战它;如果有值得挖的角度,挖下去
- 在理解基本够用之后,自然地提出那些"好奇心驱动"的问题(见风格 6)——挑有价值的问,不是逐条过清单
- 如果对话里某个转折暗示"也许用户想做的不是这件事",大方说出来:"我感觉你描述的这个问题,背后可能还有一个更核心的东西——是 X 吗?如果是的话,方向可能完全不同……"

这一步没有固定结束点。当你感觉自己对用户想做的事有了足够的理解,进步骤 2。

### 步骤 2:提议 2-3 个方向(对应风格 5)

- 在对话里列出完整的候选清单(**所有**候选描述完再给推荐)
- 至少包含一个反直觉候选
- 用 `AskUserQuestion` 让用户单选一个方向
- 如果用户选"其他",继续一轮讨论再出新一批候选

### 步骤 3:粗勾边界(为 design 预热,但不是 design)

选定方向后,轻轻勾勒一下轮廓——**这是帮 design 热身,不是替 design 做决定**:

- 用户可感知的核心行为是什么?(粗粒度即可,design 会细化)
- 有没有什么明显不在这次范围里的事?(不强求,有就记,没有也没关系)
- 最大的未知是什么?(让 design 阶段有针对性地落盘)

**不要试图在这一步就产出 design 级别的答案**。你要的只是"足够进 design 的轮廓"。

### 步骤 4:写 brainstorm note

落盘到 brainstorm note(路径和命名约定见主技能 `easysdd` 第二节"目录安排";brainstorm note 住在 `features/{feature}/brainstorm.md`,和后续 design/acceptance 同址)。

**确定 feature 目录名**:feature 目录名的格式是 `YYYY-MM-DD-{英文 slug}`(详见主技能 `easysdd` 第二节"目录安排")。具体做法:
- **日期前缀**:用**今天**的日期,自己不要凭空想,**先从环境信息里查今天是几号**(比如系统上下文里的 `currentDate`),拿到后再拼进去
- **slug 部分**:和用户确认一个**英文小写连字符的临时 slug**——哪怕此时他还不确定 feature 最终叫什么也没关系,先用着,Stage 1 design 如果改名就只 rename slug 部分、保留原日期前缀
- **不允许**在未与用户确认 slug 的情况下默默创建目录,也不允许随便编一个日期

**如果该 feature 目录不存在,就创建它**,并把 brainstorm note 作为第一个文件写进去。如果该 feature 目录已经存在(说明之前有过相关 spec),参考"启动检查"第 3 条,不要默默覆盖。

### 步骤 5:退出与交接

- 对照退出条件逐项核对
- 明确告诉用户:"brainstorm 到此结束。下一步是 Stage 1 design,触发 `easysdd-feature-design` 技能开始。"
- **绝对不要**自己接着写 design——阶段间的人工 checkpoint 是 easysdd 的硬约束

## 五、brainstorm note 模板

**路径**:brainstorm note(目录位置见主技能 `easysdd` 第二节"目录安排")

**内容**:

```markdown
# {功能名称} Brainstorm

> 阶段:Stage 0(可选前置·brainstorm)
> 状态:草稿 / 已确认
> 创建日期:YYYY-MM-DD
> 下一步:Stage 1 design

## 1. 出发点
{触发这次 brainstorm 的原始想法是什么,以及对话开始时的问题框架。
注明对话里被讨论但转向了的其他框架,以及为什么转向。}

## 2. 探索过程中的关键发现
{对话里涌现出的重要洞察、意外转折、值得注意的约束或机会。
不要追求完整性——只记真正影响了方向的东西。}

## 3. 考虑过的方向
逐一列出对话中提议过的 2-3 个方向,每个包括:

### 方向 A:{名称}
- 描述:...
- 核心价值:...
- 主要代价 / 风险:...
- 结论:{选定 / 否决,以及原因}

### 方向 B:...
### 方向 C:...

## 4. 选定方向
{选定的那一个方向,用 2-3 句话重新陈述,作为 design 的起点}

## 5. 粗粒度轮廓(供 design 参考,不是最终范围)
- **核心用户感知行为**:...
- **明显不在本次范围**(如果对话里提到了的话):
  - ...
- **最大的未知 / 风险**:...

## 6. 遗留给 design 阶段的问题
{对话中识别出但本阶段不回答的问题,留给 Stage 1 design 继续落盘}
- ...

## 备注
{可选:仓库扫描发现的相关文档/代码引用、术语撞车风险提示、知识沉淀目录里找到的相关记录等}
```

## 六、退出条件

写完 note 后,跟用户做一次"出 Stage 0"确认:

- [ ] 用户能用一句话说出"我要做的是什么"(不需要和最初一模一样,发散后收敛是正常的)
- [ ] 至少呈现过 2 个方向供对比(不一定要 3 个,但不能只有 1 个"钦定方案")
- [ ] 选定了一个方向作为 design 的起点
- [ ] brainstorm note 已落盘到对应的 feature 目录(`features/{feature}/brainstorm.md`)
- [ ] 用户明确说"可以进 design 了"

> **注意**:brainstorm 阶段不要求列出完整的不变量清单,也不要求给出技术实现细节——那些是 design 阶段的任务。退出条件的唯一目标是:用户带着一个清晰的方向和足够的信心走进 design。

---

## 七、退出后

告诉用户:

> brainstorm note 已落盘在该 feature 的目录下(`features/{feature}/brainstorm.md`),和后续的 design/acceptance 将聚合在同一个目录。下一步是 Stage 1 design:触发 `easysdd-feature-design` 技能,我会基于这份 note 起草完整方案并补齐不变量。如果在 design 阶段决定改 feature 名,整个 feature 目录一起重命名即可。

**不要自己顺手开始写 design**。brainstorm 和 design 之间的 checkpoint 是本工作流的硬约束——哪怕只隔 5 分钟,也要让用户显式发起下一阶段。

## 八、反模式(看到就停)

- ❌ **一次抛一大段问卷清单**:这不是 brainstorm 风格。brainstorm 必须一次一题
- ❌ **提问前不扫仓库**:问出"这个项目有没有 X"——而代码里就能搜到,浪费用户时间
- ❌ **只给一个"推荐方案",不给对比**:用户没看到候选就拍板,等于你替他做了决定
- ❌ **只让用户回答,不做主动挑战**:brainstorm 的价值一半在挑战上,只记录不挑战就是问卷
- ❌ **在本阶段讨论技术实现**:"要不要用 X 库"、"表结构怎么设计"——一律推到阶段二
- ❌ **把"产品压力测试"作为独立强制步骤走完**:那些问题是自然对话的一部分,不是审计清单
- ❌ **要求"明确不做至少 2 条"才能退出**:brainstorm 阶段不必产出完整的非目标列表
- ❌ **用户说"差不多就这样吧"就直接落盘**:必须走退出条件逐项核对
- ❌ **落盘后顺手开始写 design**:违反 easysdd 的阶段 checkpoint 硬约束
- ❌ **把 brainstorm note 放到 feature 目录之外**:brainstorm note 只住 `features/{feature}/brainstorm.md`,和同 feature 的 design/acceptance 聚合在一起——不允许放到架构中心目录,不允许散落到任意位置
- ❌ **没和用户确认 slug 就默默创建 feature 目录**:feature slug 是跨阶段公用的标识,必须用户确认过再落盘
