# cs-close

关闭 issue，把这次工作里毕业的东西沉淀到长期实体。

## 背景

CodeStable 的复利不在 issue 本身，而在关闭时的回写。issue 记录一次变更的历史；`requirements/` 记录当前仍然有效的需求、约束和取舍；`notes/` 记录下次能复用的坑点、操作经验和调查结论。

如果做完就停，下一次 AI 还是会忘。`cs-close` 负责最后这一口：确认 issue 已经实现和验证，把仍然有未来价值的内容从 issue 中提炼出来，写到对应实体，再给 issue 留下清楚的关闭结论。

## 原则

### 先确认可以关闭

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、slug、关键词找候选并说明依据。

开始前确认五类东西：

- `cs` 技能：没读过就先读一次，理解 CodeStable 的实体、技能边界和 `cs-close` 在 `cs-do` 之后的位置；读过就复用。
- `.cs/facts.md`：没读过就读；读过且没有修改迹象就复用。
- 目标 issue：读目标、范围、实现设计、测试设计、验证、执行记录和关闭回写。
- `.cs/requirements/`：读取与本 issue 相关的当前需求、约束、领域词汇和设计取舍。
- `.cs/notes/` / `.cs/tools/`：只在关闭回写指向它们，或 issue 中出现可复用经验/工具时读取相关文件。

关闭前要确认 issue 的目标已经达成、范围没有暗中扩大、验证结果能支撑关闭。缺少执行记录或验证结果时，不要关闭；回到 `cs-do` 补齐。

### 只沉淀仍然有效的东西

不要把 issue 全文搬进长期实体。长期实体只接收以后还会影响判断或执行的内容：

- **requirements/**：当前仍然有效的需求、约束、领域词汇、设计取舍、被否决方案的理由。
- **notes/**：坑点、调研结论、操作经验、调试路径、下次可复用判断。
- **facts.md**：极少数启动时必须知道、短期内非常影响判断的事实。
- **tools/**：这次形成或修正了可复用脚本/工具时，更新用法、输入输出和危险边界。

历史叙事、实现流水账、一次性中间判断，留在 issue，不要污染 requirements。

### 保持实体当前真相

写回 requirements 时，要把它当“当前真相”维护，而不是追加会议纪要：

- 能合并到已有条目的，就更新已有条目。
- 过期或被推翻的说法，要改掉或标明不再成立。
- 新术语要和已有领域词汇对齐，不制造同义词。
- 不引用一堆代码位置来替代需求表达；代码位置可以留在 issue 历史里。

写 notes 时可以更轻，但也要让下次能搜到：标题和关键词要明确，内容围绕可复用经验。

### 写回 issue 的关闭结论

完成沉淀后，在目标 issue 中更新 `## 关闭结论`。没有这一节就新增；已有就更新。至少说明：

- 关闭判断：为什么可以关。
- 验证摘要：哪些验证支撑关闭。
- 回写位置：写到了哪些 requirements / notes / facts / tools。
- 遗留事项：如果有，应该成为新 issue，而不是藏在关闭结论里。

只有用户明确要求时才创建新的 issue；否则只记录建议。

## 应用场景

- **实现和验证已经完成，用户要关闭 issue。** 做沉淀并写关闭结论。
- **用户说“沉淀一下”。** 从 issue 的执行记录、验证和关闭回写里提炼长期内容。
- **一个 bug 修完了。** 把稳定的预期行为、回归约束和调试经验分别写到 requirements / notes。
- **一个 feature 完成了。** 把最终需求、边界、取舍和用户故事写回 requirements。

不适用：

- **不写业务代码。** 代码没完成回到 `cs-do`。
- **不补实现设计或测试设计。** 设计缺口回到 `cs-design` / `cs-test`。
- **不把 issue 全文归档到 requirements。** 只沉淀未来仍然有用的标准和经验。
- **不主动清理全仓库。** 只围绕目标 issue 收尾。
