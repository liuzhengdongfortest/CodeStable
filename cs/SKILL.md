# cs

CodeStable 工作流家族的统一入口。用户开口多半不指名某个 `cs-xxx`——可能只说"加个权限校验"、"这里有 bug"，甚至只发个 `cs`。本技能接住开放输入、弄清意图、路由到对的子技能。

**只做两件事**：① 带具体诉求 → 匹配路由表，告诉走哪个 `cs-*` + 一句理由；② 想了解体系 / 说不清 → 给两轴速读 + 让用户挑。**本技能不做事**：不写产物、不替子技能跑流程。

## 收到调用先扫一次

1. Glob `.codestable/` 看接没接入；没有 → 先 `cs-onboard`
2. 有 → 读 `.codestable/attention.md`（含变更轴载体）；扫 `context/` 和变更轴看进行中的工作（拿目录名就够）
3. 看用户原话：开放式给体系介绍，带诉求匹配路由表

## 体系两轴速读（用户要介绍时讲这个）

CodeStable 把开发活动建模成**两根正交的轴**，产物聚在 `.codestable/`：

- **变更轴**——要做、做完会关闭的事（`issues/ epics/` 或 GitHub）：`cs-issue`（bug / 重构 / 小功能）、`cs-epic`（大需求拆解）、`cs-audit`（主动扫描发现）
- **现状轴**——现在是什么、为什么这样（`context/`）：`cs-context`（领域词汇表 + 取舍说明）

写代码纪律靠 `cs-code`（正交），沉淀靠 `cs-keep` → compound。**变更轴关闭时把毕业的取舍回写现状轴。**

核心理念：编排软件本身的生命周期（变更 + 现状），不是编排 Agent。人在环——程序员对整体把控负责，AI 是高效执行体。

## 路由表

| 用户说 / 想做 | 路由 |
|---|---|
| 没有 `.codestable/` | **先 `cs-onboard`** |
| 想法模糊 / "先聊聊" / "没想清楚" | `cs-brainstorm` |
| 新功能 / "加个 X"（已清楚） | 直接写（`cs-code`）；大到塞不下 → `cs-epic` |
| BUG / 报错 / "这里不对" / 文档错 | `cs-issue`（tag bug） |
| 重构 / 优化（行为不变） | `cs-issue`（tag refactor） |
| 大需求 / "我想要一个 X 系统" / 拆解 | `cs-epic` |
| 扫一遍 / 审计 / "有什么问题 / 哪能优化" | `cs-audit` |
| 加术语 / 记取舍 / 拍板决策 / 刷新现状说明 | `cs-context` |
| 值得记下来 / 坑点 / 选型 / 调研 / 库用法 | `cs-keep` |
| 一两行启动注意 / 命令陷阱 → attention | `cs-note` |
| 改体系共享口径 | `cs-convention` |
| 开发者 / 用户指南 | `cs-doc-tutorial` |
| 库 API 参考 | `cs-doc-api` |

判不出 / 太抽象："听起来像 {猜测}，但你描述里缺 {X}。是 {A} 还是 {B}？" 让用户选，不硬猜。

## 几种要留心的

- **大需求被误当 feature**："我想要一个权限系统 / 通知中心"一眼做不完一个功能 → `cs-epic`（或 `cs-brainstorm` 先分诊）
- **"改一下 X"但 X 是已有功能**：先问 bug（现在表现错了）还是需求变更（表现没错策略变了）——bug → `cs-issue`；需求变更 → 直接写 + 关闭时回写 `cs-context`
- **进行中的工作**：扫到相关 issue/epic 已存在 → 问"接着做这个吗？"
- **沉淀两入口**：独立 markdown（坑点 / 选型，grep 检索）→ `cs-keep`；一两行启动必读 → `cs-note`

## 退出

告诉用户下一步触发哪个 `cs-*`（或确认只是了解）：

> 你这诉求建议走 **`cs-xxx`**——{一句理由}。触发后它会 {简述}。现在切过去吗？
