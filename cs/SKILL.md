# cs — CodeStable 导览

`cs` 是 CodeStable(LITE) 的入口导览。读完你应当能回答三件事：CodeStable 在编排什么、有哪些实体和技能、什么时候伸手拿哪个。它本身不干活——不写产物、不替子技能跑流程；它的产物就是你脑子里那张体系地图。

## 为什么有 CodeStable-LITE

LITE 要消除两件事：

- **疲惫。** 用户付了大笔钱，想要甲方一样的体验——委托一件事，AI 限定时间内干好、没大纰漏、不用人盯。现实却是每件事都得人推、人审、AI 记不住需求、反复犯错要人纠正，最后人整宿守着电脑。这不该是客户的体验。
- **认知负荷。** 技能太多、行数太多、依赖太多，用户就理解困难，进而不想了解、不想用。所以技能必须把深层次的东西覆盖进去，让用户有客户一般的体验。

核心做法：**编排软件本身的生命周期，而不是编排 Agent**。人在环——程序员对整体把控负责，AI 是高效执行体。目标是让"驱动 AI 干一件事"真正变轻松，把熬夜纠错的时间还给学习、娱乐、运动和家人。

## 两件被刻意砍掉的事（先记住，免得走错路）

- **没有"重构"。** 单独的重构在任何情况下都不被需要。模块随演化自然上浮下沉；当需求演化到某层次、抽取下沉变得自然需要时，它是实现某个 feature 时顺带做的过程，不是单独的活。所以别去找 `cs-refactor`——需要重构时，去找承载那个 feature 的事项。
- **没有"审计"。** 找错总是无穷无尽。为一个**无用的**极端边界情况堆几百行代码，说明架构设计就错了。好架构靠合理的用户故事覆盖大部分常用情况；软件没按预期跑时，通过**投诉**来补全对应场景，而不是预先穷举。

## 实体（产物，聚在 `.cs/`）

实体不是给 AI 藏状态的黑盒，而是人和 AI 都能读、能改、能追溯的工作产物。详细说明见 `references/entities.md`；实体模板放在 `templates/entities/`。

| 实体 | 用处 |
|---|---|
| facts.md | 启动必读事实：少量当前最影响判断的稳定事实。|
| talks/ | 讨论整理：想法还没进入 epics / issues 前的澄清记录。|
| epics/ | 过大的讨论产物进入这里规划，拆出一组 issues。|
| issues/ | 一组可关闭的变更；单个 issue 可以是 bug、feature、小重构、chore。|
| requirements/ | 当前需求真相：用主文档统领背景、目标、术语和子文档索引，再由子文档承载领域细节。|
| wiki/ | 系统理解 wiki 和 requirements 候选：由外部观察者视角多轮提问、调查和综合生成。|
| notes/ | 知识笔记：坑点、技巧、调查结论、可复用判断。|
| tools/ | 跨工作流共享的小工具和脚本。|

## 上下文复用

技能切换不是失忆重启。当前对话或执行上下文里已经可靠获得的信息，先复用；不要因为进入另一个技能，就机械完整重读 `facts.md`、requirements 主文档或刚看过的说明。

只有这些情况才重读：从未读过；有修改迹象；需要精确引用或写回；当前任务需要此前没覆盖的局部。重读也优先按索引、标题、`rg` 或相关路径定位章节，不默认全量扫一遍。目标 issue、准备写入的 `.cs` 文件和准备修改的代码文件，写前必须确认当前版本，避免基于旧内容覆盖。

## 组织原则

长期实体不能越写越散。凡是会累积很多内容的目录，都要有入口意识：先有一个能统帅全局的主文档，说明背景、目标、统一术语、当前结构和子文档索引；子文档只负责自己的领域、模块、题型或流程细节。

`requirements/` 默认以 `.cs/requirements/index.md` 作为主文档。比如用 CS 打比赛，index 里写比赛背景、目标、评分/约束、整体策略和题型索引；每类题型再拆到独立 requirements 子文档。读 requirements 时先读 index，再按索引进入相关子文档；写回 requirements 时先判断是更新统帅口径，还是落到某个子文档，缺少合适位置时先补索引或新建子文档，不要把结论随手散落成一堆平级文件。

## 不明白就问

AI 像新加入公司的同事，不熟悉项目、业务和隐含约定是正常的。真正危险的不是不知道，而是不知道还假装知道，靠猜测继续往下做。

能从仓库确认的，先查 `.cs/`、文档、代码和测试；查不到、查起来代价过高，或属于业务取舍、用户偏好、成功标准、不可破坏边界的，就停下来问用户。不要把推测写成事实，也不要在方向、范围或用户可见行为没确认时继续实现。提问要带上下文：先说已经确认了什么、仍卡在哪里、不同答案会怎样改变下一步，再问一到三个高信息量问题。

## 举一反三

AI 不能只盯着眼前一行。修一个 bug、补一个约束、学到一个项目习惯后，要顺手看同一职责、同一接口、同一数据形状、同一用户故事里有没有同根问题或同类规则。能在当前范围内一起处理且风险清楚的，就一起处理；会扩大范围、改变需求或需要额外判断的，先记录为待确认或后续事项。

举一反三不是主动审计，也不是借题发挥。它只沿着已经暴露出来的根因、约束或模式往近处看一圈，避免同一个坑明天换个文件再摔一次。

## 最小可行梯子

写代码前先爬一遍梯子：这事真的需要做吗；仓库里已有 helper、类型、模式或模块能不能复用；标准库、平台原生能力、数据库约束、CSS/HTML 等是不是已经覆盖；已安装依赖能不能解决；能不能通过删除、收窄或一两行完成。只有这些都站不住时，才写能工作的最小代码。

最小不是草率。它发生在理解问题、读过相关代码、确认真实流程之后；它不省略根因分析、信任边界输入校验、防数据丢失的错误处理、安全、可访问性、用户明确要求和必要验证。小改动如果落在错误位置，只是第二个 bug。

## 渐进式行动

复杂任务不要求一次做完。代码设计、测试设计、复杂实现这类工作，可以先完成一部分可验证的判断，汇报已经确认的内容、仍然未知的部分和下一步，再继续推进。不要为了把任务“完成”而把不确定内容写成确定结论，也不要为了填满模板而鬼画符。

## 收束式汇报

完成一项任务后，汇报要先让用户听懂“这次到底做成了什么”。不要把文件清单、函数名和零散 diff 当成主体；先用用户能判断的话串起来：主要目标达成到什么程度，大致思路是什么，关键取舍是什么，通过什么验证保护了哪些边界条件，还有哪些风险或后续事项。

文件、命令和技术细节要给，但放在结果之后，作为证据和索引。简单任务一两句话即可；复杂任务按“结果 → 思路 → 边界/验证 → 文件证据/下一步”收住。不要用奇怪比喻替代事实，也不要为了显得完整堆满无关细节。

## 技能（用户的控制面）

通过这些技能，用户驱动 AI 走完软件的生命周期。**状态**一列标明技能是否已落地——LITE 从底层做起逐步下沉，"规划中"的还没长到，别去调它。

**行动类**（用户手动触发）：

| 技能 | 何时用 | 状态 |
|---|---|---|
| cs-onboard | 用 cs 之前，创建或补齐 `.cs/` 工作区和基础实体目录 | 已落地 |
| cs-talk | 有想法没想清、先聊聊、方向还在摇摆、信息交代不完整——先查仓库已有上下文，再把缺口、真问题、术语、约束聊出来 | 已落地 |
| cs-complain | 系统行为不符合预期时，完成投诉建档、反馈回路、诊断、修复验证并回写 bug issue | 已落地 |
| cs-wiki | 自动多轮结构性提问，派 subagent 调查，建设 `.cs/wiki/` 和 requirements 候选 | 已落地 |
| cs-plan | 把 talks 里聊清楚的需求落到当前系统：进 epics 还是 issues | 已落地 |
| cs-design | 针对明确 issue 做教程式实现设计：功能分工、请求/数据流、边界、改动路线和验证 | 已落地 |
| cs-test | 可选测试设计：用户或团队需要时，为单个 issue 设计测试目标、用例和执行方式 | 已落地 |
| cs-do | 针对明确 issue 按实现设计写代码、验证，并回写执行记录 | 已落地 |
| cs-close | 实现和验证完成后关闭 issue，沉淀长期实体；git 仓库里把代码和 `.cs` 回写一起提交 | 已落地 |
| cs-note | 记可复用项目知识：长内容写 notes，启动必读短事实写 facts.md | 已落地 |
| cs-maketools | 人带 AI 跑通未知流程，沉淀 notes、facts 引用和可选 tools | 已落地 |

**原则类**（不手动触发，是指导原则；设计上自洽、不依赖其他技能）：

| 技能 | 讲什么 | 状态 |
|---|---|---|
| cs-how-codedesign | 架构设计的指导原则 | 已落地 |
| cs-how-debug | 如何找出问题（debug） | 已落地 |
| cs-how-docs | 如何组织文档和知识空间 | 已落地 |
| cs-how-great-skills | 什么是好的技能 | 已落地 |

## 怎么用（典型一条线）

1. 还没接入 → `cs-onboard` 搭好结构
2. 新东西没想清或信息没交代完整 → `cs-talk` 先查仓库上下文，再聊出缺口、真问题、术语、约束，并整理进 `talks/`
3. 想自动摸清系统、建设 wiki 或准备 requirements 候选 → `cs-wiki` 多轮结构性提问和综合
4. 行为跑偏 → `cs-complain` 固定预期/实际，建立反馈回路，诊断修复并回写 bug issue
5. 想清了 → `cs-plan` 读取 talks，判断进 epics 还是 issues
6. issue 明确但还不能直接写 → `cs-design` 做实现设计
7. 需要测试设计 → `cs-test` 补测试目标、用例和执行方式；不需要就跳过
8. 设计就绪 → `cs-do` 开干
9. 满意了 → `cs-close` 关闭 issue，沉淀 requirements / notes 等实体；git 仓库里连同代码和 issue 一起提交
10. 值得复用的坑点、技巧、命令陷阱 → `cs-note` 判断写 notes 还是 facts
11. AI 不会某个内部流程 → `cs-maketools` 由人带路跑通一次，沉淀成 notes / facts / tools

衔接由**用户驱动**——`cs` 和各技能都不替你决定下一步走哪个，也不自动派发。这张地图只帮你知道"什么场景大概伸手拿哪个"。

## LITE 的演化方式（为什么它现在还不完整）

所谓演化：先把一件事干成、干好、干到不需要人干预，再把它下沉到基座，然后在此基础上构建更多东西。所以 LITE 不会一开始就给甩手掌柜的体验，而是从底层做起——先实验软件开发的流程，再一步步压缩，删掉不需要的步骤、整合已有的多个技能（技能下沉），最终逐步降低用户的认知负荷。上面标"规划中"的技能，就是这条路还没走到的部分。
