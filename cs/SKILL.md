# cs — CodeStable 导览

`cs` 是 CodeStable(LITE) 的入口导览。读完你应当能回答三件事：CodeStable 在编排什么、有哪些实体和技能、什么时候伸手拿哪个。它本身不干活——不写产物、不替子技能跑流程；它的产物就是你脑子里那张体系地图。

## 为什么有 CodeStable-LITE

LITE 要消除两件事：

- **疲惫。** 用户付了大笔钱，想要甲方一样的体验——委托一件事，AI 限定时间内干好、没大纰漏、不用人盯。现实却是每件事都得人推、人审、AI 记不住需求、反复犯错要人纠正，最后人整宿守着电脑。这不该是客户的体验。
- **认知负荷。** 技能太多、行数太多、依赖太多，用户就理解困难，进而不想了解、不想用。所以技能必须把深层次的东西覆盖进去，让用户有客户一般的体验。

核心做法：**编排软件本身的生命周期，而不是编排 Agent**。人在环——程序员对整体把控负责，AI 是高效执行体。目标是让"驱动 AI 干一件事"真正变轻松，把熬夜纠错的时间还给学习、娱乐、运动和家人。

## 两件被刻意砍掉的事（先记住，免得走错路）

- **没有"重构"。** 单独的重构在任何情况下都不被需要。模块随演化自然上浮下沉；当需求演化到某层次、抽取下沉变得自然需要时，它是实现某个 feature 时顺带做的过程，不是单独的活。所以别去找 `cs-refactor`——需要重构时，去找承载那个 feature 的事项。
- **没有"审计"。** 找错总是无穷无尽。为一个**无用的**极端边界情况堆几百行代码，说明架构设计就错了。好架构靠合理的用户故事覆盖大部分常用情况；软件没按预期跑时，通过**投诉**来补全对应场景，而不是预先穷举。

## 实体（产物，聚在 `.cs/`）

实体不是给 AI 藏状态的黑盒，而是人和 AI 都能读、能改、能追溯的工作产物。详细说明见 `references/entities.md`；实体模板放在 `templates/entities/`。

| 实体 | 用处 |
|---|---|
| facts.md | 启动必读事实：少量当前最影响判断的稳定事实。|
| talks/ | 讨论整理：想法还没进入 epics / issues 前的澄清记录。|
| epics/ | 过大的讨论产物进入这里规划，拆出一组 issues。|
| issues/ | 一组可关闭的变更；单个 issue 可以是 bug、feature、小重构、chore。|
| requirements/ | 当前需求真相：用综述和子模块说明背景、功能、规则、约束和取舍。|
| notes/ | 知识笔记：坑点、技巧、调查结论、可复用判断。|
| tools/ | 跨工作流共享的小工具和脚本。|

读取 `facts.md` 要按上下文判断：当前对话或执行上下文里没读过就读一次；已经读过且没有修改迹象，就复用已知事实，不要求每个技能机械重读。

## 渐进式行动

复杂任务不要求一次做完。代码设计、测试设计、复杂实现这类工作，可以先完成一部分可验证的判断，汇报已经确认的内容、仍然未知的部分和下一步，再继续推进。不要为了把任务“完成”而把不确定内容写成确定结论，也不要为了填满模板而鬼画符。

## 技能（用户的控制面）

通过这些技能，用户驱动 AI 走完软件的生命周期。**状态**一列标明技能是否已落地——LITE 从底层做起逐步下沉，"规划中"的还没长到，别去调它。

**行动类**（用户手动触发）：

| 技能 | 何时用 | 状态 |
|---|---|---|
| cs-onboard | 用 cs 之前，创建或补齐 `.cs/` 工作区和基础实体目录 | 已落地 |
| cs-talk | 有想法没想清、先聊聊、方向还在摇摆、信息交代不完整——先查仓库已有上下文，再把缺口、真问题、术语、约束聊出来 | 已落地 |
| cs-complain | 系统行为不符合预期时，固定预期/实际/复现/证据并生成 bug issue | 已落地 |
| cs-plan | 把 talks 里聊清楚的需求落到当前系统：进 epics 还是 issues | 已落地 |
| cs-design | 针对某个明确 issue 做实现设计：模块归属、接口/数据/状态、测试面和实施顺序 | 已落地 |
| cs-test | 可选测试设计：用户或团队需要时，为单个 issue 设计测试目标、用例和执行方式 | 已落地 |
| cs-do | 针对明确 issue 按实现设计写代码、验证，并回写执行记录 | 已落地 |
| cs-close | 实现和验证完成后关闭 issue，把毕业内容沉淀到 requirements / notes 等实体 | 已落地 |
| cs-maketools | 人带 AI 跑通未知流程，沉淀 notes、facts 引用和可选 tools | 已落地 |

**原则类**（不手动触发，是指导原则；设计上自洽、不依赖其他技能）：

| 技能 | 讲什么 | 状态 |
|---|---|---|
| cs-how-codedesign | 架构设计的指导原则 | 已落地 |
| cs-how-debug | 如何找出问题（debug） | 已落地 |
| cs-how-great-skills | 什么是好的技能 | 已落地 |

## 怎么用（典型一条线）

1. 还没接入 → `cs-onboard` 搭好结构
2. 新东西没想清或信息没交代完整 → `cs-talk` 先查仓库上下文，再聊出缺口、真问题、术语、约束，并整理进 `talks/`
3. 行为跑偏 → `cs-complain` 固定预期、实际、复现和证据，生成 bug issue
4. 想清了 → `cs-plan` 读取 talks，判断进 epics 还是 issues
5. issue 明确但还不能直接写 → `cs-design` 做实现设计
6. 需要测试设计 → `cs-test` 补测试目标、用例和执行方式；不需要就跳过
7. 设计就绪 → `cs-do` 开干
8. 满意了 → `cs-close` 关闭 issue，沉淀 requirements / notes 等实体
9. AI 不会某个内部流程 → `cs-maketools` 由人带路跑通一次，沉淀成 notes / facts / tools

衔接由**用户驱动**——`cs` 和各技能都不替你决定下一步走哪个，也不自动派发。这张地图只帮你知道"什么场景大概伸手拿哪个"。

## LITE 的演化方式（为什么它现在还不完整）

所谓演化：先把一件事干成、干好、干到不需要人干预，再把它下沉到基座，然后在此基础上构建更多东西。所以 LITE 不会一开始就给甩手掌柜的体验，而是从底层做起——先实验软件开发的流程，再一步步压缩，删掉不需要的步骤、整合已有的多个技能（技能下沉），最终逐步降低用户的认知负荷。上面标"规划中"的技能，就是这条路还没走到的部分。
