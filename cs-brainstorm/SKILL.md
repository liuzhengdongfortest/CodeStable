# cs-brainstorm

遵循 `.codestable/convention.md`。

brainstorm 是想法模糊时的讨论层。三条姿态：

- **创意空间不是审计关卡**——探索 / 质疑 / 改主意 / 发现真正想做的是另一件事都正常
- **任何话题都能聊**——库 / Schema / 接口用户想聊就聊，不设黑名单
- **思考伙伴不是记录员**——用户来是想被挑战被启发；只把话整理一遍写下来这步就白做

## 开聊前扫一眼

读 `attention.md`；扫 `context/`（拿术语、看现状）和进行中的变更轴，看有没有相关 / 接续的工作。Grep 用户关键词防术语冲突。

## 分诊：聊一两轮判规模

| 判断 | 出口 |
|---|---|
| 已经够清楚（一句话说清做什么 / 成功标准 / 不做什么） | 不落盘，直接写（`cs-code`）；需要追踪就开 issue |
| 小需求，解法 / 边界还摇摆 | 用下面工具箱聊到收敛 → 直接写 / 开 issue |
| 大需求，心里已有模块划分 ready 拆 | 移交 `cs-epic`（附聊到的要点） |
| 大需求，说不清边界想先 grill 探索 | 存 `.codestable/brainstorms/{slug}/brainstorm.md` → 之后 cs-epic 读到 |

判错不是灾难，允许当场升降级。

## 怎么聊（工具箱）

**1. 区分"说的"和"要的"**——开口第一句往往是想到的方案不是真问题。听到"我想做 X"先问"X 为了解决什么场景下的什么问题"。常发现真问题有更轻的解法。在用户自己意识到之前完成这件事，是 brainstorm 最大价值。

**2. 带方案来时先评估再接受**——复述成"你要解决的问题是不是 P"；看到明显问题（解错 / 过度工程 / 有更轻路径）直接说 + 提 1-2 个不同方向，不为配合闭嘴。评估完确实合理 → 直接建议动手，别为凑流程硬发散。

**节奏**（三动作可回退）：

1. **挖问题**——问到能一句话复述、用户说"对就是这个"。价值最高不要急着跳
2. **发散**——2-3 个候选方向（用户的算一个），各 1-2 句价值 / 代价，**至少一个反直觉**；全列完再给推荐（先锚定会污染判断）
3. **收敛**——轻勾勒核心行为 / 明显不做 / 最大未知，给动手热身不替决定

**grill 档**（默认不开）：用户请求、或连续两次复述被驳回、或概念反复互指时启动。最多 3-5 轮，一轮一个问题 + 2-4 候选；"得写起来才知道"的标 open question 跳过；用户敷衍立即退到收敛。

**最小 demo / spike**（默认不做）：同时满足"是事实问题不是偏好 + 结果会改变方向 + 5-30 分钟能搭"才提议验一下，比再聊三轮省时。结论回写讨论。

## 横切：拍板了结构性决策？

聊中拍板了符合 3 判据（难回退 + 不显然 + 真权衡）的决策（选库 / 模块拆分 / 通信模式 / 稳定的"不做 X"）→ 提示"这条该记进 context，要不要走 `cs-context`？"不在 brainstorm 里直接写。

## 硬边界

- 不跳过分诊
- 不替用户决定规模（模糊就问"一个 issue 装得下，还是要先 grill 存着"）
- 不处理已明确的 bug（→ `cs-issue`）
- 别自己顺手开写代码 / 拆 epic——阶段间 checkpoint 是硬约束
