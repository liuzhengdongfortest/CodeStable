# cs-complain

`cs-complain` 是 CodeStable 的投诉入口：当系统行为和用户预期不一致时，把情绪化、零散、带现场感的反馈整理成一个可关闭的 bug issue。

它不负责修代码，也不做主动审计。它的产物是一个清楚的问题陈述：预期是什么、实际是什么、最小观察路径是什么、证据在哪里、这是否真的是既有预期被违反。

## 先确认上下文

1. 如当前上下文没读过 `cs` 技能，先读它，确认 CodeStable 的实体和流程定位。
2. 如当前上下文没读过 `.cs/facts.md`，且项目中存在该文件，先读一次；已经读过且没有修改迹象就复用。
3. 搜索相关 `requirements/`、已关闭 issue、notes 或代码位置，确认“预期行为”是否已经被记录。
4. 如果用户已给出截图、日志、报错、测试失败、操作路径或代码位置，把它们作为证据保留，不要只保留结论。

## 判断是不是投诉

把用户输入拆成四件事：

- **预期行为**：用户认为系统应该怎样。
- **实际行为**：系统实际怎样。
- **最小场景**：在哪里、用什么输入或操作，二者开始分叉。
- **证据**：截图、日志、堆栈、请求响应、失败测试、用户反馈、代码位置或最近变更。

只有“已有预期被系统行为打破”才进入 bug issue。

如果预期本身还没说清，先继续问到能写出预期/实际；如果这是新需求、需求变更或产品取舍不明，转 `cs-talk` 或 `cs-plan`，不要强行建 bug。

## 建立 bug issue

当投诉成立时：

1. 在 `.cs/issues/YYYY/MM/{slug}.md` 创建 issue；年份和月份使用创建当天日期。
2. 类型写 `bug`。
3. `目标` 写修复后系统应满足的可观察行为。
4. `范围` 写本次投诉覆盖和不覆盖的边界，避免把“顺手修一堆”混进来。
5. `当前证据` 写预期、实际、最小场景和原始证据来源。
6. `方案判断` 只写投诉判断：为什么这是 bug、关联哪个 requirement 或既有行为；不要提前写实现方案。
7. `验证` 写修复完成后必须通过的复现步骤、测试或手动检查。

使用项目里的 issue 模板结构；不要新增一套投诉模板，避免实体发散。

## 证据纪律

- 不要从修复开始。先把预期、实际和分叉场景钉住。
- 不要把“可能原因”写成事实。没有证据链时只写“待定位”。
- 能轻量复现就复现；复现会明显扩大工作量时，只记录复现建议，让 `cs-design` / `cs-do` 接手。
- 如果根因明显指向需求错位、责任边界错位或架构问题，也只把判断写进 issue；设计由 `cs-design` 做。
- 如果投诉里混着多个独立行为，拆成多个 issue，或先建一个 epic 让 `cs-plan` 规划。

## 结束输出

完成后告诉用户：

- 创建了哪个 issue 文件。
- 这次投诉被判定为什么类型的问题。
- 哪些证据已经记录，哪些仍需后续定位。
- 下一步建议走 `cs-design`，必要时再走 `cs-test` 和 `cs-do`。
