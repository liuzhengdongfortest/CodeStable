# cs-code

把需求变成代码这一步，难的不是写得出来——是 **AI 默认会写得太多**：多一层"以后可能要"的抽象、多顺手改一个邻居、多一个补丁分支、多碰一个文件。这个技能是一组**停止信号**，在漂移发生那一刻把它截住。

**承重判断一句话：只写当前明确要的那点代码。** 下面全是它的落点。

## 三条姿态

### 1. 默认写最少

只写当前明确要的。不顺手加"以后可能要"的可配置项 / 抽象层 / 开关 / 防御性兜底。判据：写完一段觉得"是不是还得加点 X"——X 是不是当前用户能感知到的？不是就别加。整段写完 200 行其实 50 行能讲清 → 重写。**多出来的代码不是中性的，是后人维护的负担。**

### 2. 只动该动的，不顺手"改善"邻居

改某个函数就只改它。同文件别的函数风格丑、命名怪——除非和本次改动直接冲突，否则别碰。新代码风格匹配当前文件已有写法。"顺手改"会把一个 PR 稀释成一坨综合改动，review 成本翻几倍。

死代码处理：你这次改动让某 import / 函数变成死代码 → 删；**不是**你造成的死代码 → 留着，记成顺手发现。

### 3. 没说清的别自己拍板

写到一半撞上需求没覆盖的角落（边界、错误路径、范围外文件）——默认**停下来问**，不自己挑一个填上去。硬猜着写下去，到头来对不上当初要的。

## 反射检查：漂移发生那一刻停下来

不是阈值是触发器——硬数字会诱发"为拆而拆"把自然聚合的代码切碎。每条是"遇到 X 就停下来问自己"：

| 触发场景 | 停下来问自己 |
|---|---|
| 往一个已经很长的文件追加代码 | 文件承担几件事？新加的是已有职责延伸还是第 N+1 件事？是 N+1 就新建文件 |
| 给已经很多方法的类加方法 | 是核心职责的自然扩展，还是把类推向"什么都能干"？ |
| 函数已超过一屏 | 在做几件事？几件就拆 |
| 要加 `if (特殊情况) { 特殊处理 }` | 抽象维度选错了？正确做法常是把特殊路径和通用路径分成不同函数 / 策略 |
| 要 copy-paste 一段 | 能抽共用还是只字面相似？能抽就抽 |
| 要给函数加第 4+ 个参数 | 函数做的事太多了？长参数列表是 API 恶化的早期信号 |
| 要新写"万能 util / helper" | 真没归属，还是想不起放哪就先堆 util？ |

**补丁分支那条最值钱**：`if (特殊情况)` 基本只有一个原因——需求没覆盖到这种情况。继续写得到的是"为了能跑硬加的特殊逻辑"，下个人不知道它为什么在。停下来：补进方案 / 砍掉 / 明确标成遗留。

**停下来之后**：反射只把问题提出来，结论用户定。停下来要做的动作（拆 / 新建 / 重命名 / 抽共用）会超出当前范围 → 跟用户对齐再决定（纳入本次 / 记顺手发现留后续）。不许偷偷拆完继续写，也不许忽略信号硬冲。默认动作是停、问、再继续。

## 落位与命名

要新建类型 / 函数 / 模块前，先想它在现有结构里属于哪儿、用现有词汇命名——别往最顺手的文件堆、别造同义词。设计的形状（interface / seam / depth）靠 `codebase-design`。

## 顺手发现

本次范围外但值得改的，别顺手做，记一条：

> 顺手发现：{文件:行号} {问题简述}。不在本次范围，留后续。

混进来的顺手改让 review 和 git blame 都分不清这次到底改了什么。

## 写完

说清动了哪些文件、改了哪些函数（file:line）、有没有碰到范围外、有没有引入新概念。含糊汇报等于把验证责任推回用户。

## 沉淀

撞到项目通用的硬约束 / 命令陷阱 / 环境设置（"原来这项目要先 X 才能 Y"），或可复用的写法约定 → 提示用户走 `cs-keep` 沉淀到 compound，下个人不必再踩同一个坑。
