# cs-design

为一个具体 issue 做实现设计，把“要做什么”讲成“实现该怎么安排”。

## 背景

`cs-talk` 先和用户澄清目标并做初步规划，把讨论结果或 epic spec 的本轮范围落成 issue；但 issue 还不一定能直接开写。很多失败发生在这里：目标看起来清楚，真正动手时却不知道能力归谁、接口叫什么、数据怎么流、测试打在哪，于是实现一路漂移。

`cs-design` 补的就是这一步。它面向一个明确 issue，先理解需求和当前代码，再把实现前的关键判断写回 issue。它不是大而空的架构文档，也不是单独重构；它只服务这个 issue，让后续 `cs-do` 可以沿着设计写代码、验证、回写。

## 原则

**设计必须贴着代码。** 实现设计不能只从文字推演。要查看真实代码来确认现有能力、功能分工、命名、数据流、状态、错误、配置、外部依赖和测试面。

**写设计，不写实现。** 设计回答功能如何分工、请求或数据怎么走、每部分负责什么、行为边界、实施顺序、验证面、风险和未知；不要写伪代码长篇铺陈，不要提前实现，不要为了“设计感”造抽象。

**像教新同事一样解释。** 实现设计不是字段表，也不是给 AI 自己看的备忘录。用接近教程的自然语言讲清：功能上分成哪几部分，A 为什么负责接收请求，B 为什么负责处理请求，请求/数据从哪里来、经过哪里、在哪里结束，哪些边界不碰。工程名词、模块名和函数名只作为证据和落点，不要当正文骨架。

**优先扩展已有归属。** 能用现有模块做深，就扩展现有模块；只有真实变化维度出现时才引入接缝。

**按风险选择设计侧重点。** 设计不是只面向功能。每个 issue 要选出真正重要的 2-4 个质量目标，例如可靠性、可测试性、安全性、可维护性、可观测性、易用性，并写成具体设计判断。不要机械全列，也不要写“需要保证可靠性”这种空话。

**测试设计并入实现设计。** 普通 issue 在“怎么确认做对”里写清验证面即可；当测试策略本身复杂、用户要求 TDD、或团队要求测试设计时，先和用户讨论测试层级、行为用例、数据环境、执行方式和 test-first 顺序，再写进实现设计。不要为了流程完整单独制造测试文档。

**渐进式设计。** 复杂设计可以先完成一段可验证判断，说明已确认范围、暂不覆盖范围、下一段要解决的问题。不要把还没读懂的模块、数据流或风险写成确定结论。

**遵循 `cs-how-docs` 的文档组织口径。** 设计文档要服务两个读者：用户判断方向是否对，执行者知道从哪里下手。不要把模块名、函数名和待办项平铺成一串。

## 行动指南

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、短语、关键词找最相关候选，并说明判断依据。找不到 issue 时不要凭空设计，先让用户指定或回到 `cs-talk` 做初步规划并创建 issue。

开始前先复用当前上下文。下面是必须掌握的信息，不是每次全量重读清单；只有没读过、有修改迹象、需要精确引用/写回，或当前设计需要新增局部时才补读。

- `cs` 技能：当前上下文没读过才读。
- `.cs/facts.md`：没读过就读；读过且没有修改迹象就复用。
- 目标 issue：必须确认当前版本，掌握目标、范围、证据/背景/当前问题、方案判断、验证和关闭回写。
- 相关 spec：独立 issue 读取 `.cs/spec/` 的相关入口；epic issue 先读对应 epic 的 `spec.md`，必要时再读其引用的 project spec。
- 相关代码：确认当前实现组织和 issue 是否匹配。

### 形成实现判断

先看现有能力在哪里、需求在功能上应该分成几段、每段由谁负责、应该沿用什么命名。再判断请求、数据、状态、错误、配置和外部依赖从哪进、从哪出，以及新增行为应该在哪个用户可观察行为或接口上验证。

然后选择本 issue 的设计侧重点：失败代价高就看可靠性，依赖或状态复杂就看可测试性，权限、输入、敏感数据或危险操作存在就看安全性，规则容易演化就看可维护性，失败难排查就看可观测性，调用方或用户容易误用就看易用性。每个侧重点必须落到“因此设计上怎么安排”，而不是只贴标签。

如果验证策略需要单独想清楚，就在设计阶段处理：测可观察行为，不测内部形状；优先通过公开接口、用户路径、服务边界或集成路径验证；mock 只用于不可控外部依赖，不要 mock 掉本该一起验证的本地协作。接口、数据或运行命令还不清楚时，标成待确认并向用户讨论，不要假造具体测试。

如果一次设计会横跨多个模块、多个用户路径或多个关键取舍，先产出第一段可验证设计，用户确认后再继续。

## 产物契约

把结果写回目标 issue 的 `## 实现设计`。没有这一节就新增；已有设计就更新，不要在文件末尾堆第二份。

实现设计用教程式自然语言，而不是只填字段。推荐写法：

- **这次要怎么做**：先用一小段话讲整体方案，让非实现者也能判断方向。
- **功能怎么分工**：用自然语言说明分成哪几部分，各自负责什么；模块名只放在括号或句尾作为落点。
- **请求/数据怎么走**：讲输入从哪里来，经过哪些处理，在哪里改变含义或形状，最后谁消费。
- **哪些边界不碰**：说明支持什么、不支持什么、兼容性和危险边界。
- **设计侧重点**：只列本 issue 真正重要的 2-4 个质量目标，写清对应设计判断。例：可靠性如何处理重试/幂等/恢复；可测试性测试打在哪个接口；安全性在哪里做权限和输入校验；可维护性如何保持能力归属。
- **一步步怎么改**：给 `cs-do` 可执行的步骤，但不要展开成伪代码。
- **怎么确认做对**：说明验证打在哪些行为上，还有哪些未知。需要详细测试设计时，写清测试目标、测试层级、行为用例、测试数据和环境、执行方式，以及是否 test-first。

写回后检查 issue 是否仍然可关闭：目标、范围、实现设计、验证要能互相咬上。如果设计发现 issue 太大或边界错了，停下来说明原因，建议回到 `cs-talk` 重新澄清和拆分。

## 收尾汇报

汇报要让用户先听懂“这件事准备怎么实现”。先用教程式语言讲整体方案：功能分成哪几部分，各部分负责什么，请求或数据怎么流；再把关键模块、接口和文件作为落点补充。

然后说清执行者真正需要注意的点：实施顺序是什么、设计侧重点守住了哪些质量目标、哪些风险或未知还没盖住、验证应该打在哪些行为上。文件和章节位置只作为索引放在后面，不要用模块名和函数名堆成一份只有实现者才看得懂的清单。

## 应用场景

- **用户指定某个 issue 要先设计。** 读取 issue 和代码，把实现设计写回 issue。
- **issue 已经有目标和范围，但还不能直接开写。** 补齐功能分工、请求/数据流、边界、验证和实施顺序。
- **实现前发现设计风险。** 在写代码前把风险和未知暴露出来，必要时建议拆分或回到计划。
- **用户或团队要求测试设计。** 在实现设计里补齐详细验证策略，而不是另走独立技能。

不适用：

- **不处理模糊想法。** 需求还没说清，回到 `cs-talk`。
- **不创建事项。** 没有 issue，先用 `cs-talk`。
- **不写代码。** 实现交给 `cs-do`。
- **不做全仓库架构审计。** 只围绕目标 issue 读必要代码。
