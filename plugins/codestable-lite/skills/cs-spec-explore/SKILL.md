# cs-spec-explore

围绕 project spec 的空白做调查，但把探索结果留在 issue 里，等人确认关闭时再合并进 project spec。

## 背景

很多项目不是缺少文档，而是缺少“这块到底该怎么理解”的中间探索。直接写进 project spec 太冒险；另建独立知识库又容易变成旁路材料，最后没人知道哪些已确认、哪些只是猜测。

`cs-spec-explore` 把这种工作纳入 issue 生命周期：探索也可以是一个 issue。issue 记录问题、证据、推理、候选说明和仍未知的点；当用户确认它可信并关闭时，`cs-close` 再把毕业结论合并进 `.cs/spec/`。

## 原则

**探索属于 issue。** 不创建独立文档目录，不维护独立候选池。一次探索的目标、材料、文章和争议都放在对应 issue 里，便于讨论、修改、关闭和回溯。

**问题要打开结构。** 好探索不是“补点文档”，而是回答会影响开发者理解的问题：数据怎么流、控制怎么走、责任归谁、术语是什么意思、边界和异常在哪里。

**先查仓库，再问用户。** 先读 project spec、facts、README、代码、历史 issue/epic、notes 和相关配置。仓库里已经有的信息不要问用户；查不到、冲突或需要业务判断时，再带证据提问。

**文档先在 issue 里被讨论。** 探索文章要写成未来可迁移到 project spec 的自然语言说明，但在关闭前它只是候选理解。错误、争议和修正直接在 issue 中迭代。

**核心理解不引用代码。** `## 探索文档` 先讲能力、流程、概念、边界和取舍；代码路径、函数名、命令结果放在 `## 当前证据` 或证据索引里。代码只能支撑理解，不能替代理解。

**关闭才进入主线。** 本技能不直接改 `.cs/spec/`。用户确认 close 时，由 `cs-close` 把仍然成立的需求、架构考量、统一语言和边界合并到 project spec 的合适层级。

## 行动指南

### 建立探索 issue

开始前复用当前上下文；必要时补读 `.cs/facts.md`、`.cs/spec/index.md`、相关 spec 子层、README、目录结构、关键代码和历史 issue/epic。没有 `.cs/` 时先建议 `cs-onboard`。

如果用户已经指定 issue，确认它的 `type: explore` 或适合改成探索型 issue。如果没有指定，就创建 `.cs/issues/YYYY/MM/DD/open-{短语}.md`，frontmatter 使用 `type: explore`、`status: open`、`epic: ""`，除非它明确隶属某个 epic。

探索 issue 的目标要说清：

- project spec 缺了哪块理解。
- 这块缺口为什么会影响开发者。
- 本轮准备调查到什么程度。
- 哪些问题需要用户确认，哪些可以靠仓库证据回答。

### 调查与成文

围绕一个高信息增益问题调查，不追求一次摸完整个系统。常见问题类型：

- 数据流：输入从哪里来，在哪里改变形状或语义，最后被谁消费。
- 控制流：一个动作触发哪些入口，成功、失败、重试和提前结束在哪里决定。
- 责任归属：规则归哪个模块，谁只是搬运、展示或适配。
- 统一语言：项目词汇是什么意思，相似词有什么区别。
- 边界异常：不支持什么，依赖失败、权限不足或输入异常时系统怎样。

把结果写入 issue 的 `## 探索文档`。不要粘贴问答流水，要组织成未来开发者能读懂的连续说明：先讲结论和地图，再讲理由、边界、仍未知的点。代码证据集中放在 `## 当前证据`，每条证据说明它支撑哪个理解。遵循 `cs-how-docs` 的费曼式表达、金字塔结构和必要图示。

### 处理不确定性

如果证据冲突、术语不明、需要业务判断或调查会触及危险操作，停止扩写结论，在 issue 的 `## 待确认问题` 里列出带证据的问题，再向用户提问。

用户修正后，更新 issue 内探索文档和证据，不把旧错误迁移到 project spec。需要保留争议时，写成“已排除理解”或“仍未知”，不要伪装成事实。

### 准备关闭

探索足够稳定时，在 issue 的 `## 关闭回写` 写明建议合并到 `.cs/spec/` 的哪一层、为什么放那里、哪些内容应该进入 project spec、哪些只留在 issue 作为证据。

交给 `cs-close` 关闭。只有用户确认 close，稳定结论才进入 project spec。

## 产物契约

探索型 issue 使用普通 issue 路径：`.cs/issues/YYYY/MM/DD/open-{短语}.md`。

frontmatter 的 `type` 必须是 `explore`。issue 正文至少包含或补齐：

- `## 目标`：要补上的 project spec 理解缺口。
- `## 当前证据`：仓库证据、代码路径、文档引用、命令结果；每条证据说明支撑的结论。
- `## 探索文档`：面向未来开发者阅读的候选说明；核心理解不引用代码路径。
- `## 待确认问题`：需要用户判断或补线索的点。
- `## 关闭回写`：建议合并位置和合并理由。

不创建独立文档目录、问题池、候选池或主题文件。

## 收尾汇报

先讲这次探索补上了哪块 project spec 理解、当前结论是什么、还卡在哪些确认点，以及是否已经准备好交给 `cs-close`。最后给 issue 路径作为证据。

## 应用场景

用户希望摸清系统某一块、补 project spec 前先调查、让 AI 作为新同事提出并回答结构性问题、或 plan/spec 时发现关键未知但还不能直接写入主线。

不适用：想法本身还在脑暴时用 `cs-talk`；规格已经确认只需要写入 project/epic spec 时用 `cs-spec`；行为不符合预期走 `cs-complain`；探索已经确认完成后用 `cs-close` 合并。
