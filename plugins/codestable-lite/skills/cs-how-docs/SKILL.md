# cs-how-docs

AI 会写单篇文档，但常常不会组织文档。它会把 `index.md` 写成目录，把一轮调查的问题平铺成同级小节，把所有页面套同一个模板，看起来完成了任务，实际上没有帮后来者理解系统。

本技能教的是文档组织，不是文风。它回答：这些知识应该怎样分层、入口在哪里、读者按什么路径读、当前真相和历史证据放哪，以及怎样把复杂内容讲到读者真的懂。

## 背景

文档不是内容容器，而是后来者理解系统的路径设计。一个好的文档系统不追求把材料都塞进去，而追求让读者知道先看什么、带着什么问题往下走、在哪里找细节、在哪里看当前真相、在哪里看历史和证据。

CodeStable 里的 project spec、epic spec、探索型 issue、`notes/`、README 和技能文档都会越写越多。如果没有组织判断，它们会变成文件垃圾场：每篇都能看，合起来不能用。

## 核心概念

**读者任务。** 先判断谁会读：新人理解系统、实现者改代码、维护者排障、用户确认需求，还是未来 AI 复用经验。读者不同，入口和层级就不同。

**使用叙事。** project spec 的主叙事应该围绕读者如何使用当前项目完成工作：从哪里进入、能做什么、怎么组合能力、遇到限制怎么办。代码层级可以帮助定位，但不能替代使用路径。

**主文档。** 主文档不是目录。它要统帅这个知识域：解释它是什么、为什么存在、核心概念、全局地图、重要入口和下一步阅读路径。索引只是主文档的一部分。

**层级。** 文档之间要有抽象关系：总览、领域、机制、流程、证据、历史、候选。不能把所有 topic 平铺成同级页面。

**递归入口。** 文档是树，不是固定二级目录。每一层入口文档都只负责让读者在这一层建立方向感，并决定下一步往哪读；简单内容一篇写完，复杂内容继续拆下一层。

**阅读路径。** 链接不是为了显得有关联，而是告诉读者下一步去哪：关心 X 读哪篇；要修改 Y 先看什么；只想确认需求看哪里。

**真相分层。** 当前真相、候选理解、活规格边界、历史过程和操作经验不能混在一起。project spec 放主线真相，epic spec 放大需求活规格边界的当前理解，探索型 issue 工作区放未关闭的候选文章、用户修正和证据，普通 `issues/` 放执行历史，`notes/` 放可复用经验。

**第一性原理。** 先回到这个文档要解决的最小本质问题：读者为什么需要它，系统必须守住什么约束，哪些概念不可再拆。不要从已有文件、模板或术语出发堆结构。

**金字塔表达。** 先给结论和地图，再展开理由和细节。每一层只回答一个上层问题，避免让读者先读一堆材料再自己归纳。

**费曼式表达。** 用能教给新同事的自然语言解释：这是什么，为什么需要它，分成哪几块，东西怎么流，哪里容易误解。工程名词、文件名、函数名是落点和证据，不是正文骨架。

**核心理解不引用代码。** spec 和探索文档的主叙述先讲能力、流程、概念、边界和取舍；代码路径、函数名和命令结果只放在证据索引、落点或附录里。读者应该先理解系统，再按索引去验证。

**视觉辅助。** 当关系比文字重要时，用图。流程、层级、数据流、状态变化、依赖关系、读者路径适合图示；文字只补解释和边界。图要服务理解，不做装饰。

## 判断框架

写或整理文档前，先回答七个问题：

1. 这批内容服务哪个读者任务？
2. 哪一篇是主文档，它应该统帅什么？
3. 哪些内容是总览、领域、机制、流程、证据、历史或候选？
4. 读者最自然的三条阅读路径是什么？
5. 哪些内容应该合并、拆分、降级为证据，或迁移到别的实体？
6. 如果用一句话讲给新同事，最小本质是什么？
7. 哪些关系用图比用文字更清楚？

组织文档时，按问题和结构分，不按产出批次分。一次调查问了三个问题，不等于应该生成三个 topic；要看它们分别属于数据流、控制流、模块责任、概念词汇、用户路径、外部依赖还是边界行为。

主文档要用说明文建立地图，再给入口。project spec 优先按使用场景、能力和决策路径建立地图，再补代码落点。避免只有表格。表格可以作为索引，但不能替代对系统结构、读者路径和重要性排序的解释。

每篇子文档只承担一个清楚职责。它可以按自己的材料长成不同形状：数据流文档讲流向和变形，概念文档讲术语和不变量，流程文档讲步骤和失败点，需求文档讲当前规则和取舍。不要为了整齐强迫所有文档使用同一组标题。

每篇文档先写“读者该带走什么”，再写为什么，最后写证据和细节。复杂机制优先用自然语言讲一遍，再给必要的工程名词、路径、命令或表格。不要用代码路径当标题或段落主语；需要追溯时集中到“证据索引”。

需要视觉辅助时，优先使用简短 ASCII 图、Mermaid、列表层级或小表格。图里只放关键节点和关系；如果图看不出方向、责任或变化，就不要画。

链接要带意图。不要只堆文件名；要说明为什么读它、读完能解决什么问题、什么时候不需要读。

## 反例

- `index.md` 只有一张 topic 表，没有讲这个知识域的目标、地图和阅读路径。
- 探索型 issue 的候选文章以轮次、问题编号或调查批次平铺，而不是按系统结构拆成可阅读文档。
- 每篇 topic 都套同一个模板，导致概念、流程、数据流和证据文档长得一样。
- 一上来就堆模块、函数、字段和路径，读者看完仍不知道功能怎么分工。
- project spec 按目录、包、类库分层，读者仍不知道要用这个项目完成一件事时应该从哪里开始。
- spec 的核心段落写成代码导览，真正的需求、边界和架构考量反而要读者自己推断。
- 明明是流程、状态或依赖关系，却用长段落硬讲，不给图或结构化路径。
- 把尚未确认的探索结论写进 project spec，或把关闭 issue 的历史过程塞进当前真相。
- 链接只是互相引用，没有告诉读者“为什么去那里”。

## 应用场景

写或重组 project spec、epic spec、探索型 issue、notes、README、技能文档、项目手册和多文件知识库时使用。尤其适合：文档越来越多但不好找、index 变成目录、内容平铺、AI 生成的文档看似完整但读者仍不知道该从哪开始。

不适用：润色单篇文章语气、生成对外营销文案、制作 API 逐项参考。那些可以有自己的写作或文档技能。
