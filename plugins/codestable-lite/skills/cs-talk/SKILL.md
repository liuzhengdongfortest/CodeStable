# cs-talk

## 背景

洞察用户的真实诉求，而不是帮他整理说出来的那个。用户开口说的常常是方案，或只是表面诉求；方案不是问题，表面诉求也不一定是真实诉求。

cs-talk 在场，是为了把真问题挖出来、加速真实诉求浮出水面。衡量它有没有起作用只有一条：聊完之后，用户的认知比聊之前多了点什么。

## 原则

先看清是什么，再让用户看见他看不见的，最后把边界和终点收住。

不明白就问。像新加入公司的同事一样工作：先承认自己不熟悉项目里的隐含约定，再用调查缩小未知，最后把剩下只有用户能补的信息问清楚。不要为了显得自主，把猜测当成需求、约束或事实继续往下聊。

先补信息缺口，能查就别问。凡是可能已经在当前上下文或仓库里的，先复用已知信息；没掌握、疑似变化或需要新增局部时，再查 `.cs/facts.md`、`.cs/spec/`、`.cs/talks/`、相关 issue/epic、README、代码和测试。查不到、查起来代价过高，或属于偏好/取舍/外部业务事实时，才问用户。

分清方案与问题、表面诉求与真实诉求、偏好与约束。偏好可以商量，约束不能绕过；稳定约束最终应进入 project spec 或 epic spec。

只记录确认过的边界。`不做什么` 只能来自两种情况：用户明确说了不做，或你提出某个排除项后用户同意了。没有确认过的排除项，不要为了让整理稿显得完整而硬写。

## 行动指南

先复用当前上下文。project spec 口径已掌握就复用；只补读和本次想法相关的入口或子层。聊的时候沿用现有领域语言；用户新冒出来的关键词，先确认不和已有术语打架。

问用户时只问剩下的缺口。先说明已从仓库确认了什么、还缺什么，再按阻塞程度问一到三个高信息量问题。问题要能改变下一步判断，例如“这是必须保留的兼容行为，还是可以迁移？”，而不是“还有什么要补充吗？”。

让用户看见没看见的东西：用 what-if 把没考虑到的场景推到台面上，用取舍问题逼出优先级，方案有明显毛病就直说。评估下来确实合理的，也直接讲清楚让用户拿去落地。

聊到三条同时满足就收住：问题能用一句话说清并由用户确认；做什么已有轮廓，已确认边界被记录，未确认边界被标成未知；最大的未知已经被标出来。

收不住时，最多三到五轮深挖；如果争的是事实、结果会改变方向、半小时内能出结论，直接动手验证比继续聊更省时。

## 产物契约

讨论结束后，把整理稿写到 `.cs/talks/YYYY/MM/DD/{短语}.md`。没有当天目录就创建。

整理稿不是会议纪要流水账，只保留：

- 原始想法。
- 真问题。
- 术语。
- 偏好与约束。
- 已确认边界。
- 待确认边界和分歧。
- 最大未知。
- 下一步候选出口。

整理稿遵循 `cs-how-docs` 的文档组织口径：它要给 `cs-plan` 和未来读者一条理解路径，而不是复刻聊天顺序。

未确认的“不做什么”不要写成结论。下一步只标明候选出口，交给 `cs-plan` 先提出计划草案；用户确认后，才把大需求落成目录型 epic，或把足够小的范围落成独立 issue。

## 收尾汇报

先向用户复述这次聊清楚的真问题、已确认约束、仍未知的点和建议出口，再说明 talk 文件写到了哪里。不要把文件路径当作主要成果。

## 应用场景

有个想法还没想清楚、方向还在摇摆、心里知道要做什么但交代不完整、想先聊聊再动手时使用。

cs-talk 不直接创建 epics 或 issues，不替代 spec，不处理已明确的 bug，不自己写代码。上游规格/PRD/SE 文档目标大体稳定但信息传递不完整时，用 `cs-spec`。
