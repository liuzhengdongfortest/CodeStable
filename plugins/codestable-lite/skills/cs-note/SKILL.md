# cs-note

## 背景

把“这事以后还会用到”落进 `.cs/`，但不要把 `.cs/` 写成垃圾桶。`cs-note` 同时接管原来的 keep/note：短而稳定的启动事实进 `.cs/facts.md`，需要背景、证据或步骤的知识进 `.cs/notes/`。

## 原则

先做“六个月测试”：半年后仍会影响判断或操作，而且很难只从代码、git diff、README 重新看出来，才值得写进 `.cs/`。代码已经明说的实现细节、一次性流水账、临时情绪和容易从历史记录重建的东西，不要沉淀成知识。

facts 是启动索引，不是知识库；每条最多两行，只放稳定、几乎每次会话都会影响判断或操作的事实。复杂内容写 notes，再在 facts 加一行指向它。

拿不准时默认写 notes，不要污染 facts。当前项目主线真相进 project spec；一次变更历史留在 issue 或 epic；未知流程由 `cs-maketools` 带路跑通。

遵循 `cs-how-docs` 的文档组织口径。notes 要服务未来读者的复用任务：标题、结论、触发场景和证据要帮助搜索与判断，不要把一次聊天或一次执行流水账平铺成知识。

## 行动指南

先复用当前上下文，确认 `cs`、`.cs/facts.md`、相关 notes、project/epic spec 和 issue/epic 是否已经掌握；没掌握、疑似变化、需要查重或准备写回时再读。`facts.md` 不存在但 `.cs/` 存在时，可以按模板创建最小文件。搜索 `.cs/notes/`、project spec、相关 epic spec 和 issue/epic，避免重复记录；发现已有相近 notes 时，确认当前版本后更新原文件并在末尾加一节日期更新。

如果信息来自代码、日志或命令结果，先保留可复核证据，不要只写口头结论。

写 facts 时，放到合适小节；没有小节就追加到“启动必读事实”。适合写项目固定测试命令、构建前置、环境陷阱、关键目录禁区、长期命名约定，或“做 X 前先读某篇 note”的索引。

写 notes 时，在 `.cs/notes/YYYY/MM/DD/{短语}.md` 写纯 markdown。标题和正文保留用户会搜索的关键词、模块名、命令名、错误名。

## 产物契约

`.cs/facts.md` 只能追加或更新短事实，写入前先查重；已有相近事实时更新原条目，不追加第二条相似说法。

`.cs/notes/YYYY/MM/DD/{短语}.md` 推荐结构：

```markdown
# {知识点}

## 结论

{以后最应该复用的判断或做法}

## 触发场景

{这条知识在哪类任务、错误、流程或模块里适用}

## 证据和细节

{代码位置、命令、日志、截图描述、操作步骤、失败分支、危险边界}

## 相关位置

- {issue / epic / spec / 文件路径}
```

一次处理一个主题；多个无关主题拆成多条 notes 或多次 facts 更新。

## 收尾汇报

告诉用户写到了 `.cs/notes/` 还是 `.cs/facts.md`、为什么选择这个去处、是新增还是更新已有文件。不要把文件清单当主要汇报，先讲这条知识未来会怎么帮上忙。

## 应用场景

用户说“记下来”“沉淀一下”“留个 note”“加到 facts”“以后每次都要知道”，或一次工作暴露出可复用坑点、操作经验、调研结论、长期约束、命令/路径/环境陷阱时使用。

不替代 `cs-close`、`cs-maketools` 或 spec。
