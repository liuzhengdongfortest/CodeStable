# cs-how-codedesign

好的结构，是让调用方用更少的认知，换到更多的能力。设计模块时，核心方向只有一个：往深里做。

使用前先确认：如果当前对话/执行上下文已经读过 `cs` 技能，就复用；没读过就先读一次，理解 CodeStable 的实体和技能边界。

## 背景

代码通常不是一下子坏掉的，而是在演化里慢慢变浅、变散。

一种坏法是模块太**浅**：接口几乎和实现一样复杂，调用方要理解的东西和自己重写一遍差不多，这层封装没有省下认知。另一种坏法是概念太**散**：同一个能力散在十几个小模块里，读懂一件事要在文件之间来回跳。浅和散都会增加认知负荷，也会让代码难测、难改，AI 也更难导航。

LITE 里没有单独的"重构"。模块的抽取、合并、下沉，应该长在实现 feature 的过程中。但顺手改也需要一把尺子，否则很容易把代码改得更碎。这把尺子就是**深度**：用尽量小、稳定、好理解的接口，承载尽量多的行为。

## 原则

### 先把词定准

这套判断依赖几个稳定的词。不要随手换成"组件"、"服务"、"API"、"边界"这类更含糊的说法。

- **模块（module）**：任何"有接口、有实现"的东西。规模不限，一个函数、一个类、一个包、一段跨层切片，都可以是模块。
- **接口（interface）**：调用方为了正确使用模块必须知道的全部信息。它不只是类型签名，还包括不变量、调用顺序、错误模式、必要配置和性能特征。
- **深度（depth）**：接口的杠杆。调用方每学一点接口，能换回多少行为。小接口后面藏了大量行为，就是深；接口和实现差不多复杂，就是浅。
- **接缝（seam）**：可以改变行为、但不必在当前位置改代码的地方。接缝通常就是接口所在的位置；接缝放哪，和接缝后面放什么，是两个不同的设计决定。

### 先定归属和名字

设计新能力时，先问它**属于哪里**、应该沿用现有词汇**叫什么**，再问接口怎么做深。

默认的失败方式，是图方便把能力丢进最近的文件，顺手起一个新同义词。这样会长出平行实现，把命名空间劈成两半，搜索也从此失效。先搜一遍现有概念和同义词：能扩展那个本该拥有它的模块，就扩展；确实没有合适归属，再新开一个模块。

### 先改善可修改性，再实施行为变更

当目标变更被现有结构阻碍时，不要把新行为硬塞进错误归属、重复概念或过浅接口里。先做服务于当前目标的最小结构调整：明确能力归属，统一命名，收拢重复规则，压缩接口表面积，建立必要的验证入口。然后再实施行为变更。

结构调整必须有当前目标约束，不能借机做无关重构。判断标准是：调整后目标修改是否更局部、更可解释、更容易验证；如果不能降低这次变更的风险或认知负荷，就不要做。

### 往深里做

- 设计接口时反复问三件事：**能不能少几个方法？能不能简化参数？能不能把更多复杂度藏进去？**
- 做一次**删除测试**：假想把这个模块删掉。如果复杂度也跟着消失，它只是穿透层，白占一层；如果复杂度会在多个调用方重新冒出来，它就在替系统省认知，值得存在。
- 记住：**深是接口的属性，不是实现的属性。** 深模块内部仍然可以由小部件组成，也可以有内部接缝。只要这些内部结构不泄漏到对外接口上，它们就不会增加调用方的认知负担。

### 接缝必须真实

不要为了"看起来可扩展"提前造接缝。一个 adapter 只是想象中的接缝，两个 adapter 才说明这里真的有变化维度。

通常，真实接缝至少要满足一种情况：生产和测试需要不同实现；多个运行环境需要不同实现；依赖本身不可控，必须隔离在可替换的位置。否则，只有单个实现的接缝往往只是多绕了一层。

### 接口就是测试面

- 调用方和测试应该穿过**同一道接口**。如果测试总想钻到接口背后，通常说明模块形状还不对。
- 让接口天然好测：依赖靠注入，不在内部自己 new；返回结果，少制造隐式副作用；表面积要小，方法和参数越少，测试搭建越轻。
- 接缝后面怎么测，取决于依赖性质。纯计算就通过新接口直接测；自己的远程服务可以定义 port，生产用真实 adapter，测试用内存 adapter；不可控第三方，比如支付、短信，就把它当注入的 port，测试时给 mock。
- **替换，别叠加。** 模块做深后，原先围着浅模块写的单测通常该删掉。在新接口上重写测试，断言可观察结果，而不是内部状态。这样的测试能扛住内部重构，只在行为真的变了时才需要改。

### 把边界和易用性收进设计

设计不是只让 happy path 能跑。一个深模块应该替调用方吃掉常见边界，让使用者不必在每个入口重复猜“这里要不要判空、禁用、确认、加载中”。

- 输入边界要有归属：必填、格式、范围、重复提交、幂等、权限、资源不存在、外部依赖失败，都要决定在哪一层校验和报错。不要让同一条规则散落在前端、API、数据库和测试里各写一遍。
- 默认路径要省心：常见参数给默认值，罕见配置藏到可选入口；错误信息要指向可修复动作，不要把内部异常直接甩给调用方或用户。
- 前端状态也是接口契约：加载中、空状态、错误状态、禁用状态、重复点击、请求取消、乐观更新失败，都应该在组件或交互模型里有明确设计，而不是等用户点出问题再补。
- 危险操作必须有护栏：删除、覆盖、重置、发布、扣费、权限变更等动作，要有二次确认、不可逆提示、执行中防重复提交，以及失败后的可恢复路径。
- 边界处理要提升深度，不要污染接口。优先把校验、状态机和确认流程封进合适模块；如果每个调用方都要自己记一套边界清单，说明模块还不够深。

### 按质量目标看设计

实现设计要根据当前 issue 的风险选择侧重点，而不是只问功能能不能跑。常见侧重点包括：

- **可靠性**：失败、重试、幂等、恢复、数据一致性和外部依赖失败由谁负责。
- **可测试性**：测试穿过哪个接口，哪些依赖需要注入，哪些场景用 fake、mock 或真实集成。
- **安全性**：权限、输入校验、敏感数据、危险操作、注入和越权风险在哪一层收住。
- **可维护性**：能力归属、命名、规则集中度和未来变化点是否清楚。
- **可观测性**：失败后如何定位，日志、错误信息、状态和指标应该在哪出现。
- **易用性**：调用方或用户是否少记规则，默认路径是否省心，错误是否指向可修复动作。

不要每次全列。选会改变设计的 2-4 个侧重点，写成具体取舍。

### 重要接口要设计两遍

第一个冒出来的接口，通常只是最顺手的版本，不一定是最好的版本。重要接口值得从几个不同方向各设计一版，例如：

- 把接口压到最小，让每个入口的杠杆最大。
- 为常见调用方优化，让默认路径最省心。
- 保留必要灵活性，但不把罕见分支暴露给所有调用方。

然后按深度、改动是否集中、接缝位置是否真实来比。选接口，不是选第一个能写出来的方案，而是选最能降低长期认知负荷的方案。

## 应用场景

- **实现 feature 时发现需要抽取、合并或下沉模块**：用深度判断新接口长什么样，接缝该放在哪。
- **代码读着累、改一处牵动很多地方、或者难测**：优先怀疑模块太浅，或概念散落在多个位置。
- **设计新模块接口**：先定归属和名字，再压接口、厚实现。

不适用：

- **不是主动审计工具。** LITE 没有"专门扫一遍代码找问题"这一步；架构改进发生在实现 feature 的过程中，不单独立项。
- **不规定技术栈，不写具体代码。** 这里只给判断模块深浅和接缝位置的尺子。
- **不手动触发。** 它是设计代码时随手对照的原则，不是一个要运行的流程。
