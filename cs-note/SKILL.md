# cs-note

遵循 `.codestable/convention.md`。本技能专管 `attention.md`——CodeStable 技能启动时的**强制必读**上下文。

`cs-keep` 产出独立 markdown 到 `compound/` 靠 grep 被读到；`attention.md` 是每次都读。两类归宿不同，本技能管后者：把"短、稳、每次都要知道"的碎片追加进 attention。缺 `.codestable/` → 提示先 `cs-onboard`；只缺 attention.md → 先建固定分节骨架再写。不读写 `AGENTS.md` / `CLAUDE.md` 等外部入口。

## 什么进 cs-note

判据：**长度 + 频次 + 稳定度**，三条都过才进。

| 项 | 进 cs-note | 走别处 |
|---|---|---|
| 长度 | 一两行讲清 | 超半屏 / 需展开 → cs-keep |
| 频次 | 几乎每次会话都用 | 只在某类任务相关 → cs-keep |
| 稳定度 | 项目长期硬约束 | 临时 workaround → 写进 issue |
| 拍板 | 已既成事实 | 选型理由 / 拒方案 → cs-keep；架构决策理由 → cs-context |

✅ 典型该进："编译要先 `pnpm run gen`"、"起服务前 `docker compose up redis`"、"别用 npm install，锁文件是 yarn berry"、"src/legacy/ 改前先问"、"OPENAI_KEY 走 1Password"

❌ 不该进：某 bug 修法 / 某库用法（→ cs-keep）、架构说明（→ cs-context）、"本周在做 X"（短期状态会过期）、3 行以上才讲清的

判不准反问："这条以后每次会话都要让 AI 知道吗？"答"不一定"→ 不是 cs-note。

## 固定分节（写死，不在列表的不开新节）

```markdown
## 项目碎片知识
<!-- cs-note managed -->
### 编译与构建
### 运行与本地起服务
### 测试
### 命令与脚本陷阱
### 路径与目录约定
### 环境变量与凭证
### 其他
```

- 新条目去对应分节末尾追加，每条一行（最多两行）
- 没合适分节 → "其他"；"其他" ≥ 5 条停下来和用户讨论是否新增固定分节（不默默加）
- 分节为空整段保留不删；`<!-- cs-note managed -->` 是识别锚，找不到就在文件末尾插整块
- 整段 ~150 行软上限，超过提示挑几条沉 cs-keep

## 流程

1. **判该不该进**——对照判据表，任一不过引导到别处
2. **找位置**——读 attention.md 定位锚，分节归类；段内 grep 查重，有相似条目问"更新已有还是另起一条"，不闷头新建
3. **写一条**——格式 `- {一句话事实 + 必要时一句话原因}`。写完用户 review 确认。**不主动连写多条**，一次一条，免得把没拍板的也塞进去
4. **软上限检查**——≥ 150 行提示沉 cs-keep；"其他" ≥ 5 条提示讨论新增分节。只提示不替决定

## 主动推荐时机

只两个信号推一句，不每次问：

1. 用户说出明显是碎片知识的事实（"这项目要先 X 才能 Y"）→ "这条要不要 cs-note？以后每次都看得到"
2. AI 自己刚踩一个一句话能讲清的项目特殊设置 → 修复后推"这坑是项目通用的吗？是的话 cs-note 记一笔"

用户说"不用"立刻跳过。

## 坑

- 详细背景 / 多步骤塞进 attention（超两行 → cs-keep）
- 写到 AGENTS.md / CLAUDE.md——不再兼容外部入口
- 默默新增分节
- 看到一条连带写其他几条
- 写短期状态（会过期没人删变误导）
- 不查重就追加（同一事实记 3 次反而不知哪条准）
