# cs-test

为一个 issue 补测试设计。需要时调用，不需要时跳过。

## 背景

有些团队要求每个重要变更在实现前有测试设计：测什么、不测什么、在哪一层测、用什么数据、怎么执行。它不是实现本身，也不应该让每个 CodeStable 流程都变重。

`cs-test` 是可插拔关卡。它位于 `cs-design` 之后、`cs-do` 之前：当用户或团队要求测试设计时，把测试判断写回 issue；不要求时，`cs-do` 仍然可以按 issue 的普通验证方式执行。

## 原则

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、slug、关键词找候选并说明依据。

开始前确认五类东西：

- `cs` 技能：如果当前对话/执行上下文已经读过 `cs` 技能，就复用；没读过就先读一次，理解 CodeStable 的实体、技能边界和 `cs-test` 的可选位置。
- `.cs/facts.md`：没读过就读；读过且没有修改迹象就复用。
- 目标 issue：读目标、范围、当前证据、实现设计、验证和关闭回写。
- `.cs/requirements/`：只读和测试判断相关的需求、约束、领域词汇和取舍。
- 测试现状：查看项目现有测试目录、命名、运行命令和常见测试层级。

没有实现设计时也可以做高层测试设计，但如果接口、数据、状态都不清楚，只能写测试目标和待定项；不要假造具体测试。

### 测行为，不测形状

测试设计要围绕可观察行为，而不是内部实现：

- 优先通过公开接口、用户路径、服务边界或集成路径验证。
- 少测私有方法、内部结构、临时字段形状。
- mock 只用于不可控外部依赖；不要 mock 掉本来应该一起验证的本地协作。
- 每个测试用例要能说明它守住哪条目标、约束或风险。

测试不是越多越好。只覆盖这个 issue 的关键行为、边界和回归风险；不把全系统质量愿望塞进测试设计。

### 可选 test-first

如果用户要求 TDD，或 issue 是明确行为变更/bug 修复，测试设计里标明 test-first 顺序：

- 一次只设计一个红绿循环的起点，不要先写一整排 imagined tests。
- bug 优先设计复现/回归测试，再修根因。
- feature 优先设计最小可观察行为，再按垂直切片增加用例。
- 探索性强、接口未定、纯视觉判断的 issue，不强行 TDD；先把验证方式讲清。

### 写回 issue

把结果写回目标 issue 的 `## 测试设计`。没有这一节就新增；已有测试设计就更新，不要追加第二份。

测试设计应包含：

- **测试目标**：这次测试要证明什么。
- **测试层级**：单元 / 集成 / 端到端 / 手动验证，为什么。
- **行为用例**：关键 happy path、边界、回归风险。
- **测试数据和环境**：需要什么数据、依赖、配置。
- **执行方式**：测试命令或手动步骤。
- **test-first 顺序**：需要 TDD 时列红绿循环；不需要就写“否”和理由。

## 应用场景

- **公司或用户要求测试设计。** 针对目标 issue 写 `## 测试设计`。
- **实现前想先确定怎么测。** 明确测试层级、用例和执行方式。
- **bug 修复需要回归测试。** 设计复现和回归用例，再交给 `cs-do` 实现。
- **用户明确要 TDD。** 给出最小红绿循环顺序。

不适用：

- **不写业务实现。** 实现交给 `cs-do`。
- **不替代实现设计。** 模块、接口、数据还不清楚时先用 `cs-design`。
- **不强制所有 issue 测试。** 只有用户或团队需要时才调用。
- **不做全仓库测试审计。** 只围绕目标 issue 设计测试。
