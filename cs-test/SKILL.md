# cs-test

为一个 issue 补测试设计。需要时调用，不需要时跳过。

## 背景

有些团队要求每个重要变更在实现前有测试设计：测什么、不测什么、在哪一层测、用什么数据、怎么执行。它不是实现本身，也不应该让每个 CodeStable 流程都变重。

`cs-test` 是可插拔关卡。它位于 `cs-design` 之后、`cs-do` 之前：当用户或团队要求测试设计时，把测试判断写回 issue；不要求时，`cs-do` 仍然可以按 issue 的普通验证方式执行。

## 原则

**测行为，不测形状。** 测试设计围绕可观察行为，而不是内部实现。优先通过公开接口、用户路径、服务边界或集成路径验证。

**测试不是越多越好。** 只覆盖这个 issue 的关键行为、边界和回归风险；不把全系统质量愿望塞进测试设计。

**mock 要有边界。** mock 只用于不可控外部依赖；不要 mock 掉本来应该一起验证的本地协作。

**test-first 是可选策略。** 用户要求 TDD，或 issue 是明确行为变更/bug 修复时，优先设计最小红绿循环。探索性强、接口未定、纯视觉判断的 issue，不强行 TDD。

**渐进式测试设计。** 接口、数据或运行命令还不清楚时，把它标成待确认并汇报下一步，不要假装已经知道项目怎么测。

## 行动指南

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、slug、关键词找候选并说明依据。

开始前先复用当前上下文，确认 `cs` 技能、`.cs/facts.md`、目标 issue、requirements 主文档口径、相关 requirements 子文档，以及项目现有测试目录、命名、运行命令和常见测试层级已经掌握。没掌握、疑似变化或需要精确写回时再读；目标 issue 写回前必须确认当前版本。

没有实现设计时也可以做高层测试设计，但如果代码形状、接口、数据、状态或行为边界都不清楚，只能写测试目标和待定项；不要假造具体测试。

### 设计测试面

每个测试用例都要能说明它守住哪条目标、约束或风险。bug 优先设计复现和回归用例；feature 优先设计最小可观察行为；refactor/chore 重点证明行为不变。

## 产物契约

把结果写回目标 issue 的 `## 测试设计`。没有这一节就新增；已有测试设计就更新，不要追加第二份。

测试设计应包含：测试目标、测试层级、行为用例、测试数据和环境、执行方式、test-first 顺序。无需 TDD 时写“否”和理由。

## 收尾汇报

汇报时先说明测试策略：主要守住什么行为、选择哪个测试层级、为什么要或不要 test-first；再列写回的 issue 位置和仍待确认的测试条件。

## 应用场景

- **公司或用户要求测试设计。** 针对目标 issue 写 `## 测试设计`。
- **实现前想先确定怎么测。** 明确测试层级、用例和执行方式。
- **bug 修复需要回归测试。** 设计复现和回归用例，再交给 `cs-do` 实现。
- **用户明确要 TDD。** 给出最小红绿循环顺序。

不适用：

- **不写业务实现。** 实现交给 `cs-do`。
- **不替代实现设计。** 模块、接口、数据还不清楚时先用 `cs-design`。
- **不强制所有 issue 测试。** 只有用户或团队需要时才调用。
- **不做全仓库测试审计。** 只围绕目标 issue 设计测试。
