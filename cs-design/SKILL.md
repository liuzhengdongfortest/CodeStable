# cs-design

为一个具体 issue 做实现设计，把“要做什么”翻译成“代码应该怎么长”。

## 背景

`cs-plan` 把讨论结果落成 issue 或 epic，但 issue 还不一定能直接开写。很多失败发生在这里：目标看起来清楚，真正动手时却不知道能力归谁、接口叫什么、数据怎么流、测试打在哪，于是实现一路漂移。

`cs-design` 补的就是这一步。它面向一个明确 issue，先理解需求和当前代码，再把实现前的关键判断写回 issue。它不是大而空的架构文档，也不是单独重构；它只服务这个 issue，让后续 `cs-do` 可以沿着设计写代码、验证、回写。

## 原则

**设计必须贴着代码。** 实现设计不能只从文字推演。要查看真实代码来确认现有能力、模块归属、命名、数据流、状态、错误、配置、外部依赖和测试面。

**写设计，不写实现。** 设计回答模块归属、接口和数据、行为边界、实施顺序、验证面、风险和未知；不要写伪代码长篇铺陈，不要提前实现，不要为了“设计感”造抽象。

**优先扩展已有归属。** 能用现有模块做深，就扩展现有模块；只有真实变化维度出现时才引入接缝。

**渐进式设计。** 复杂设计可以先完成一段可验证判断，说明已确认范围、暂不覆盖范围、下一段要解决的问题。不要把还没读懂的模块、数据流或风险写成确定结论。

## 行动指南

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、slug、关键词找最相关候选，并说明判断依据。找不到 issue 时不要凭空设计，先让用户指定或回到 `cs-plan` 创建 issue。

开始前确认：

- `cs` 技能：当前上下文没读过就先读一次。
- `.cs/facts.md`：没读过就读；读过且没有修改迹象就复用。
- 目标 issue：读目标、范围、当前证据、方案判断、验证和关闭回写。
- `.cs/requirements/`：先读 `.cs/requirements/index.md` 掌握主文档口径和子文档索引，再只读和该 issue 相关的当前需求、约束、领域词汇和设计取舍。
- 相关代码：确认当前代码形状和 issue 是否匹配。

### 形成实现判断

先看现有能力在哪里、需求属于哪个模块、应该沿用什么命名。再判断数据、状态、错误、配置和外部依赖从哪进、从哪出，以及新增行为应该在哪个接口上验证。

如果一次设计会横跨多个模块、多个用户路径或多个关键取舍，先产出第一段可验证设计，用户确认后再继续。

## 产物契约

把结果写回目标 issue 的 `## 实现设计`。没有这一节就新增；已有设计就更新，不要在文件末尾堆第二份。

实现设计至少包含：模块归属、接口 / 数据 / 状态、行为边界、实施顺序、验证面、风险和未知。

写回后检查 issue 是否仍然可关闭：目标、范围、实现设计、验证要能互相咬上。如果设计发现 issue 太大或边界错了，停下来说明原因，建议回到 `cs-plan` 重新拆分。

## 收尾汇报

汇报要让用户先听懂“代码准备怎么长”。先用一句话讲设计结论：这次能力归到哪里，沿用还是新增哪个接口/模块；再解释为什么这样放，主要数据、状态或边界会怎么流动。

然后说清执行者真正需要注意的点：实施顺序是什么、哪些风险或未知还没盖住、验证应该打在哪些行为上。文件和章节位置只作为索引放在后面，不要用模块名和函数名堆成一份只有实现者才看得懂的清单。

## 应用场景

- **用户指定某个 issue 要先设计。** 读取 issue 和代码，把实现设计写回 issue。
- **issue 已经有目标和范围，但还不能直接开写。** 补齐模块归属、接口、数据、测试和实施顺序。
- **实现前发现设计风险。** 在写代码前把风险和未知暴露出来，必要时建议拆分或回到计划。

不适用：

- **不处理模糊想法。** 需求还没说清，回到 `cs-talk`。
- **不创建事项。** 没有 issue，先用 `cs-plan`。
- **不写代码。** 实现交给 `cs-do`。
- **不做全仓库架构审计。** 只围绕目标 issue 读必要代码。
