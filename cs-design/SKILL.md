# cs-design

为一个具体 issue 做实现设计，把“要做什么”翻译成“代码应该怎么长”。

## 背景

`cs-plan` 把讨论结果落成 issue 或 epic，但 issue 还不一定能直接开写。很多失败发生在这里：目标看起来清楚，真正动手时却不知道能力归谁、接口叫什么、数据怎么流、测试打在哪，于是实现一路漂移。

`cs-design` 补的就是这一步。它面向一个明确 issue，先理解需求和当前代码，再把实现前的关键判断写回 issue。它不是大而空的架构文档，也不是单独重构；它只服务这个 issue，让后续 `cs-do` 可以沿着设计写代码、验证、回写。

## 原则

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` / grep 递归搜索 `.cs/issues/`，按标题、slug、关键词找最相关的候选，并说明判断依据。找不到 issue 时不要凭空设计，先让用户指定或回到 `cs-plan` 创建 issue。

开始前确认四类东西：

- `cs` 技能：如果当前对话/执行上下文已经读过 `cs` 技能，就复用；没读过就先读一次，理解 CodeStable 的实体、技能边界和 `cs-design` 在 `cs-plan` 与 `cs-do` 之间的位置。
- `.cs/facts.md`：没读过就读；读过且没有修改迹象就复用。
- 目标 issue：读目标、范围、当前证据、方案判断、验证和关闭回写。
- `.cs/requirements/`：只读和该 issue 相关的当前需求、约束、领域词汇和设计取舍。

### 设计必须贴着代码

实现设计不能只从文字推演。要查看真实代码来确认：

- 现有能力在哪里，能不能扩展，而不是平行新建。
- 这个需求属于哪个模块，应该沿用什么命名。
- 数据、状态、错误、配置和外部依赖从哪进、从哪出。
- 当前测试面在哪里，新增行为应该在哪个接口上验证。

看不清代码现状时，先继续读代码或做小范围调查。不要把“不知道”写成设计结论。

### 写设计，不写实现

设计要回答能指导实现的判断：

- **模块归属**：改哪些模块，为什么能力属于那里。
- **接口和数据**：新增或调整什么入口、字段、状态、错误语义。
- **行为边界**：做什么，不做什么，哪些路径保持不变。
- **实施顺序**：先改什么再改什么，必要时拆成小提交或子步骤。
- **验证面**：测试打在哪个接口，手动验证看什么。
- **风险和未知**：哪些判断还不确定，动手时一旦不成立应该停下来。

不要写伪代码长篇铺陈，不要提前实现，不要为了“设计感”造抽象。能用现有模块做深，就扩展现有模块；只有真实变化维度出现时才引入接缝。

### 写回 issue

设计完成后，把结果写回目标 issue 的 `## 实现设计`。没有这一节就新增；已有设计就更新，不要在文件末尾堆第二份。

设计写回后，检查 issue 是否仍然可关闭：目标、范围、实现设计、验证要能互相咬上。如果设计发现 issue 太大或边界错了，停下来说明原因，建议回到 `cs-plan` 重新拆分，而不是强行写一个巨型设计。

## 应用场景

- **用户指定某个 issue 要先设计。** 读取 issue 和代码，把实现设计写回 issue。
- **issue 已经有目标和范围，但还不能直接开写。** 补齐模块归属、接口、数据、测试和实施顺序。
- **实现前发现设计风险。** 在写代码前把风险和未知暴露出来，必要时建议拆分或回到计划。

不适用：

- **不处理模糊想法。** 需求还没说清，回到 `cs-talk`。
- **不创建事项。** 没有 issue，先用 `cs-plan`。
- **不写代码。** 实现交给 `cs-do`。
- **不做全仓库架构审计。** 只围绕目标 issue 读必要代码。
