# cs-context

遵循 `.codestable/convention.md`（两轴模型 / 启动必读 / 单目标 / 路径命名 / 退出骨架）。

`.codestable/context/` 是**现状轴**——只回答「现在是什么、为什么这样」。

**北极星：读 context 即知代码的需求与取舍。** 但不追求"记下全部需求"（永远没法确认完整）。目标是：

> context 只承载**代码本身展示不出来的东西**——为什么这样、为什么不那样、被否决的方案、哪根轴要变。

代码讲"做了什么"，context 补"为什么"。**代码 + context = 完整图景，谁都不重复谁。**

贯穿全篇的硬判据：**一段内容若能靠读代码重新生成出来，就删掉它。**

## 不引用代码位置，靠领域语言指路

context 不写文件名 / 行号 / 函数名——它们会烂。读者从"为什么"跳到"在哪"靠**领域语言**：context 用领域概念说事，代码用同名概念命名，概念本身即指针，回代码搜词就到。这反过来要求代码必须说领域语言（属 `cs-code` 的纪律），否则两边接不上。

## 两类文档

### ① 词汇表（glossary）

领域语言、ubiquitous language。最稳，是 context↔代码的连接键。**只下定义不描述行为**——写"X 是什么"不写"X 怎么做"。同义词挑最好的，其余列 `_Avoid_`。只收项目特有术语（timeout、error type 这类通用概念不进）。

```md
# {Context 名}
{一两句：这个 context 是什么、为什么存在。}

## Language
**Order**: {一两句定义。}
_Avoid_: Purchase, transaction
```

### ② 取舍说明（rationale）

按**模块**或按**矛盾**一篇，固定形状：

```md
# {直接平铺说这块是什么，不玩比喻}
## Happy path
主干业务怎么走（主要矛盾）。
## 边界条件
什么情况把它搞复杂了（次要矛盾）。
## 为什么需要灵活性
哪根轴会变 → 因此有哪层抽象。也放"为什么选这个、否决了什么"。
```

结构（分几层、什么模式）能从代码读回来，**不写**；赌注（为什么在这里开缝）读不回来，**才写**。每层抽象都该有对应的"为什么这轴要变"；讲不出，那层该被 audit 标成过度设计。

**什么取舍配单独成段（3 判据须同时满足）**：① 难回退 ② 不带上下文就难理解 ③ 真实权衡的结果（有备选，因具体原因挑了一个）。三缺一就不写。

## status 与 RFC 式关系

context 可演化但**不讲历史叙事**（"曾经…后来…"属于关闭的 issue）。演化靠 RFC 式指针，不靠时间态。三值：`current`（现行）/ `superseded`（被取代，带 `superseded_by` 指继任或具体节）/ `retired`（退休，带一行"为什么死 + 别复活"）。

| 场景 | 处理 |
|---|---|
| 部分被覆盖（某节迁走，其余现行） | 老文档留，迁走那节换指针，其余不动 |
| 整篇有继任 | 老文档缩成一行重定向 stub（或删，git 有） |
| 退休（死了无继任） | 留精简墓碑 `retired` + 一行"因 X 移除，别加回来" |

**退休墓碑不违反"无历史叙事"**——它是朝前看的约束（别复活），不是回头看的叙事，正是"被否决的方案也是 context"。全文老内容进 git / 关闭 issue。

## 没有 draft

"想要但没做"的能力**不进 context**——它是变更轴上一个 open 的 issue。context 只描述已为真的；能力做完、issue/epic 关闭时再把毕业的取舍写进来。

## 单 / 多 context 拓扑

只有 `context/CONTEXT.md` → 单 context；存在 `context/CONTEXT-MAP.md` → 多 context（各子目录自带 CONTEXT.md 和取舍说明）。**升级显式触发**（用户说"要分子系统了"才升，不因变长自动建议），按归属拆词汇与取舍，跨 context 的留顶层。

## 写之前不该出现的东西

代码位置 / 实现细节 / 算法 / 库名；产品口吻（用户故事 / 痛点 / 宣传词）；历史叙事；能从代码读回来的废段落。

## 接口

| 对方 | 关系 |
|---|---|
| `cs-audit` | 拿 context 当**对账基准**：代码偏离取舍、抽象缺"为什么要变" → 开 issue |
| `cs-epic` | epic 打开期间接口契约是 provisional；**关闭时稳定的取舍毕业进 context** |
| `cs-issue` | issue 关闭若改变了取舍，回写对应取舍说明（DoD 一条） |
| `cs-code` | code 必须用领域语言命名，否则 context 接不上代码 |
| `cs-keep` | 操作性沉淀（坑点 / 命令）走 compound，不进 context |

## 退出条件

通用骨架见 convention，另加：

- [ ] 词汇表：只下定义，同义词列 `_Avoid_`，拓扑位置正确
- [ ] 取舍说明：三节齐；每层抽象有对应"为什么这轴要变"
- [ ] 无代码位置 / 实现细节 / 产品口吻 / 历史叙事 / 能从代码读回来的废段
- [ ] status 正确；superseded / retired 带指针或墓碑理由
