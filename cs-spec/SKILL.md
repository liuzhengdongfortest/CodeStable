# cs-spec

把上游规格、代码现状和实现目标持续对齐，把已经足够明确的部分交给 `cs-plan`，把仍不明确的部分留在 spec 工作区继续澄清。

## 背景

企业里的需求常从产品经理或 SE 的规格文档开始。目标大体已经下达，但规格常常不够细：术语没定义、系统名没人认识、流程缺失败分支、实现中才发现冲突，甚至实现后规格又改。

`cs-spec` 处理这种“目标稳定但信息传递不完整”的场景。它不像 `cs-talk` 那样重新发现真实诉求，也不直接拆 issue；它把规格读成一个可演进的实现地图：整体流程顺不顺、哪些部分已经能计划、哪些卡点还要问人、哪些材料要调查并留档。

## 原则

**先顺整体，再拆细节。** 不要一上来钻字段、接口或某个功能点。先把用户目标、端到端流程、主要角色、状态变化、验收结果和破坏性影响顺一遍；整体讲不通时，细节问得再多也没用。

**问题必须面向实现阻碍。** 不问“还有什么补充吗”。只问会改变实现路线、plan 粒度、风险边界或规格正确性的问题。提问要带调查证据和假设。

**先调查，再问人。** 规格里的未知系统、术语、角色、状态和流程，先查 `.cs/`、README、代码、历史 issue、wiki、requirements 和 notes。查不到或有多个可能含义时，再请用户给线索或确认。

**规格不是圣旨，但目标先稳定。** 默认上游目标稳定；如果规格逻辑无法满足用户需求、与现有系统冲突、或会破坏已确认边界，就明确标成“规格需要调整”，不要把矛盾藏进实现。

**plan 可以分批，不必等全局完美。** 真实规格会反复补充、修改甚至推翻。只要某一块目标、范围、依赖和验证已经足够清楚，就可以在 `plan-brief.md` 标成可计划范围，交给 `cs-plan` 先落 issue；不清楚或可能推翻的部分继续留在 spec 里。

**资料要留在规格工作区。** 每个 AI 原先不知道、但会影响实现的系统、概念、流程或外部依赖，都要在 `findings/` 留一篇调查资料，并从 `index.md`、`glossary.md` 或 `questions.md` 链回去。

**遵循 `cs-how-docs` 的文档组织口径。** spec 工作区可能很长，必须主文档统帅、子文档分治。`index.md` 讲整体和读者路径；细节进 `findings/`、`flow.md`、`glossary.md`、`changes.md`。

## 行动指南

先锁定规格材料：用户给文档、截图、会议纪要、issue、链接或口头说明都可以。没有 `.cs/` 时先建议 `cs-onboard`。

创建或更新 `.cs/specs/YYYY/MM/DD/{slug}/`。如果用户指定已有 spec 工作区，就在原目录迭代，不新建一份。

第一轮只做整体通读：

- 上游目标是什么，用户需求是否能一句话复述。
- 端到端流程能不能从触发到结果讲通。
- 规格中的未知词、系统、角色、状态、外部依赖有哪些。
- 哪些点会阻塞 plan，哪些可以留到 design/do。
- 是否存在规格自相矛盾、破坏已有实现或需要推翻旧方案的地方。

随后围绕最高阻塞问题循环：调查仓库和 `.cs/`，写入 findings 或 glossary，向用户提出带证据的问题，接收线索后继续调查。每轮都更新 `questions.md`、`changes.md`、`index.md` 和 `plan-brief.md`：哪些部分已经可以 plan，哪些部分 blocked，哪些已实现内容可能受新规格影响。

问题示例：

- “规格提到 `xxx 系统`，我在仓库和 `.cs/` 没找到可靠对应物。它是现有外部系统、新建能力，还是内部别名？”
- “规格里的 `客户` 可能对应代码里的 `account` 或 `tenant`，两者语义不同。这里指哪一个？”
- “这里要求同步状态，但没有失败路径。失败时要重试、回滚、提示用户，还是允许部分成功？”

## 产物契约

默认结构：

- `index.md`：规格目标、用户需求复述、整体流程、关键阻碍、本轮可计划判断和文档入口。
- `source.md`：上游规格来源、版本、会议材料和摘要。
- `questions.md`：待确认问题、已确认答案、阻塞级别、下一步。
- `glossary.md`：规格里的系统名、业务词、角色、状态和确认程度。
- `flow.md`：端到端流程、状态变化、异常路径和破坏性变更。
- `findings/{topic}.md`：未知系统/概念/流程/依赖的一事一篇调查资料。
- `changes.md`：规格增删改查、会议结论、推翻点、影响范围和受影响的已计划/已实现事项。
- `plan-brief.md`：交给 `cs-plan` 的入口材料，可以是全量 ready，也可以是某一批 ready。

`plan-brief.md` 必须写清：上游目标、用户需求、本轮可计划范围、暂不计划范围、关键约束、已确认问题、剩余阻碍、已有关联 issue/epic、建议进入 epic 还是 issue。

当主文档超过承载能力时拆子文档；每个拆出的资料必须从 `index.md`、`glossary.md` 或 `questions.md` 链回去。

## 收尾汇报

先讲规格澄清到什么程度：整体流程是否顺了、本轮哪些范围可以先 plan、最关键阻碍是什么、哪些问题已确认、哪些已计划/已实现内容可能受影响。再给 spec 工作区路径和 `plan-brief.md`，说明下一步是否交给 `cs-plan`，还是继续澄清。

## 应用场景

用户给上游 PRD/SE 规格/会议材料，要求先过规格、澄清落地、找实现阻碍、整理不明确术语、补充/修改已经计划或实现过的规格、或说“先别 plan，先把规格和实现卡点对齐”时使用。

不适用：目标本身还在脑暴时用 `cs-talk`；某一批范围已经足够明确、只需要生成事项时用 `cs-plan`；单个 issue 的实现方案用 `cs-design`；长期系统理解用 `cs-wiki`。
