# 原则：构建优秀技能

一个技能好不好，不看它写了多少，看它能不能让一个聪明、但不了解上下文的执行者，在该用的时候用对，在没覆盖到的情况下也能自己推断。

审视技能时优先让技能自洽。知识与行动规则可以通过同一技能内的渐进式资源组织，但入口必须明确说明何时读取，不能让核心契约变成隐藏知识。

## 背景

技能会越堆越多，也会烂。烂的方式通常是两头：行动类技能写成一堆抽象口号，读完不知道怎么推进；原则类技能写成操作手册，把判断框架压扁成步骤。还有一种常见坏味道，是把产物格式、路径、状态变化散落在原则里，最后原则不纯，行动也不清。

所以需要一把统一的尺子，但不是一个统一模板。先判断技能类型，再决定结构。行动类技能通常需要行动指南和产物契约；原则类技能更需要清楚的概念、判断框架和应用边界。

## 原则

**先判型，再定结构。**

- **背景——解决什么问题。** 不是介绍技能"做什么"，是讲清它为什么存在、在解决什么问题。读者（人或 AI）只有知道了问题，才能在技能没明说的情况下，靠原则推断该怎么办。背景是技能能举一反三的根。没有背景，技能就只能照搬，不能迁移。
- **原则——判断力，不是步骤。** 原则回答"为什么这样判断"和"遇到变化时守住什么"。原则要稳定、可迁移、能跨场景复用；不要塞进读哪些文件、写哪些字段、先后顺序这些执行细节。
- **行动类技能** 要让执行者知道怎么推进：开始前必须掌握哪些上下文、哪些可复用、哪些必须重读或确认当前版本、怎么找目标、什么时候停下来问、什么时候交给别的技能、怎么验证和回写。它通常需要 `行动指南`。
- **会写文件或状态的行动类技能** 要说明产物契约：路径、frontmatter、字段、状态变化、文件重命名、模板必须保留的部分。产物契约要具体，因为它保护互操作；不要混在原则里。
- **原则类技能** 不需要伪装成流程。它可以由背景、核心概念、判断框架、反例、应用场景组成。重点是让读者拿它做判断，而不是按步骤执行。
- **应用场景——什么时候用、什么时候不用。** 触发条件要具体到读者一眼能认出"这就是该用它的时候"。同样重要的是负空间：说清它**不**处理什么。说不清不做什么，就等于边界没画，迟早范围蔓延。

**原则保持纯净。** 如果一句话只在当前技能的一步操作里成立，它不是原则，放进行动指南。如果它描述文件名、字段、状态或模板，它不是原则，放进产物契约。原则应该能帮执行者在未覆盖场景里做判断，而不是替代操作清单。

**收尾汇报单独处理。** 会产出结果的行动类技能，要说明完成后怎么向用户汇报。好的汇报先讲结果、思路、边界和验证，再给文件和命令作为证据；不要把文件列表当成主体。原则类技能通常不需要这一层。

**自洽，不依赖其他技能。** 一个技能要能独立成立。它采用的结构要从自己内部讲清楚，不靠"去看另一个技能"来补。依赖越多，隐患越多——被依赖的一变，自己就跟着坏。

**核心单一，适配层要薄。** 同一技能分发到多个 Agent、命令系统或插件宿主时，把行为和知识保留在一个权威来源；宿主适配器只负责发现、加载、参数转换和生命周期接入，不发展第二套规则。宿主必须复制文本或生成包时，用确定性生成或一致性检查保护同步，不能只靠 README 提醒维护者“记得一起改”。

**适当精简，不设统一行数上限。** `SKILL.md` 首先要完整讲清会影响理解和判断的背景、核心结构、原则、契约与边界；这些内容不能为了追求短而被压缩成口号，也不能被拆散到读者无法建立整体认识。与此同时，不影响主线判断的大段材料会占用上下文并增加认知负荷，应在它们只服务特定场景或已经妨碍阅读时按渐进式披露抽出去。讲清核心以后就停。

**渐进式披露。** 不需要把所有东西都堆进 `SKILL.md`。主文件只放会影响判断和执行的核心，以及必要的资源路由。大段参考材料、长例子、命令细节、模板和可复用脚本，应该迁到同一技能目录里的 `references/`、`templates/`、`scripts/` 或 `assets/`，让执行者在需要时再读。拆出去的内容必须被 `SKILL.md` 明确指到，不能变成隐藏知识；也不要为了拆而拆，一屏能讲清的东西就留在主文件里。

**原则多了就归类，别平铺。** 原则一多，平铺成一长串就散了——条目类型不一，读者没有落点。按"它们各自在解决同一类事"把同类归到一起，给几个组标题，顶层锚点就从十几个收成三四个。归类要按主题分，不能按"第一步第二步"，否则原则又退回成了步骤。

## 应用场景

- **写一个新技能**——先判断它是行动类、原则类、知识入口、工具型还是混合型，再选择结构。不要因为某个模板顺手，就强套所有技能。
- **审视一个已有技能够不够好**——先看类型是否清楚，再看结构是否服务这个类型：行动类有没有行动指南，原则类有没有判断框架，写文件的技能有没有产物契约。
- **一个技能让人不想用、或者 AI 老用错**——多半是层次错位：该做判断的地方写成步骤，该执行的地方只剩口号，该固定格式的地方没有契约。

不适用：写具体的代码、文档或业务流程——那是各个技能自己的事。这里只管"技能本身该长什么样"。
