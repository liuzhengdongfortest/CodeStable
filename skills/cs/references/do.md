# Do：实现明确改变

按用户已明确授权的目标写代码并验证；有常规 issue 则回写执行记录。用户要快且需求小 → 走 [fast](fast.md)（**快改** + `ff` issue），不要用本文件的完整受管理路径硬套。

**完成 ≠ 关闭**（常规 issue）。实现与验证做完即止；不自动关闭常规 issue、不 commit、不 push / 发布。快改须落 `x-ff` 痕迹，见 fast。关闭与毕业见 [close](close.md) 与 `SKILL.md`。

## 背景

危险有两端：写着写着目标漂了；或为几分钟能做完的明确改变强制建档。Do 按明确范围实现，用与风险相称的证据证明结果；发现目标或设计不成立就停下说明。

## 原则

**只做已授权的改变。** 受管理守 issue 范围；直接改守用户明确目标与必要影响面。额外功能、相邻 bug、无关重构不塞进本次；值得追踪时建议新 issue，由用户决定。

**先选管理强度。** 决策表见 `SKILL.md`。用户明确要求直接改 → 直接改；只表达目标且是否留痕会实质改变后续 → 带推荐询问。

**沿现状说明实施。** 开工前能解释相关触发如何产生结果，以及落点与需验证的相邻行为。不能解释 → 先现状说明或 Design；跨多未知边界 → Explore issue。见 [explore](explore.md)。

**小偏差回写，大偏差停下。** 与设计/代码小冲突可更新 issue 设计或执行记录后继续；大偏差停止，建议回 Design 或 Talk。

**最小充分、非草率。** 梯子与有界简化见 [economy](economy.md)；不能省略根因、校验、信息安全性与安全性相关护栏、数据保护、可访问性、已选质量目标与必要验证。

**渐进式实现。** 复杂时可先完成可验证垂直切片，回写进展再继续。

**验证要能说明问题。** Bug 先复现或定位分叉；feature 优先可观察行为/接口/集成路径；refactor 验证行为不变。非平凡逻辑至少一个会在行为损坏时失败的最小可运行检查。

**质量目标逐项兑现。** 受管理兑现 issue 已选目标；直接改守 spec/用户要求与必要风险边界，不生成九项清单。见 [quality](quality.md)。

**UI 服从视觉契约。** 见 [ui-spec](ui-spec.md)。图与文字冲突先消歧，不静默改稳定约束。

## 行动指南

### 锁定目标与管理方式

- 用户给出 issue 或要求实现某 issue → 确认当前版本，**受管理**推进（路径形如 `NNN-o-名称`）
- 已在明确 issue 上下文中 → 延续，不重复选择
- 用户要快 / 快速 / 直接开干且需求小 → **[fast](fast.md)**，不走本文件长路径
- 明确要求修改且简单、未要求快改痕迹 → **直接改**（也可建议升为快改以留 `ff`）
- 只在讨论、是否改动不清 → Talk，或一句带推荐确认快改/直接改 vs 受管理

不要仅因仓库存在 `.cs/issues/` 就绑旧 issue；也不要在明显需要受管理时悄悄跳过记录。

写前掌握：目标与范围；**已完成与本次主题相关的 `.cs/` 检索**（见 `SKILL.md`；本会话设计阶段已扫过且无新写入可复用）；相关 **project spec**（重中之重）与 epic `spec.md`（若适用）；相关代码是否仍贴合设计。未检索不得只靠代码开写。已掌握且无变化迹象的不机械重读。

确认现状说明足够；有变化时分清必须修改 / 需要验证 / 仍待调查。直接改须简单到能在当前上下文讲清方案与验证；否则建议建 issue 并先 Design。

### 实现与验证

按已确认目标与判断改代码。有 issue 逐项兑现质量目标；直接改继承 spec 并处理必要风险。阈值/显著成本/兼容策略/目标冲突 → 停下说明。复杂度涨到不再适合直接改 → 建议转 issue，未经同意不自建。

UI：只实现已确认“目标”图的本范围；验证覆盖稳定约束主路径与相关关键状态。

有界简化：确认未破坏目标与质量承诺；受管理补回 issue；直接改若上限会持续影响维护者，写入相关 spec/note 或向用户说明，不自动建 issue。

跑最相关测试/检查；不能跑时说明原因与手动步骤。需重新设计、补验证策略或危险操作授权时，汇报已完成部分与阻塞点。

## 产物契约

**受管理**：更新目标 issue 的

- `## 执行记录`：改了什么、为何、与设计偏差；有界简化与 UI 偏离原因
- `## 验证`：命令/步骤、结果、分别支撑哪些质量目标
- `## 关闭回写`：关闭时应进入 project/epic spec 或 notes 的候选（**此时不毕业**，除非用户明确要求收尾）

现状或影响不准时更新原章节，不在执行记录另存冲突解释。

**直接改**：不创建常规 issue。若希望可追溯，改用 [fast](fast.md) 落 `ff`。仅当已记录真相失效时同步 project spec；不自动改 Vision；**默认不 commit**。

## 收尾汇报

先说解决了什么、思路、兑现的质量目标、守住的边界、跑了什么验证；关键文件作证据放后。说明：实现已完成；若有常规 issue，关闭须用户再授权。

## 应用场景

- 明确要求直接改且简单 → 直接改；要快且留痕 → [fast](fast.md)
- 指定 issue 实现 → 按设计改并验证
- 设计已完成可开写
- 非 bug 类 issue 推进
- 已诊断 bug 的剩余实现（缺根因或默认试修失败时先 [complain](complain.md)）

不适用：模糊需求 → Talk；快改通道 → [fast](fast.md)；风险未打通需先垂直走通 → [pierce](pierce.md)；擅自建/拆事项；高风险无设计 → Design；bug 诊断 → Complain；用户要审代码 → [review](review.md)；关闭与沉淀 → Close。
