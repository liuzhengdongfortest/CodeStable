# Do：实现明确 Issue

按一个明确 issue 写代码、验证、回写。

## 背景

CodeStable 里真正危险的不是“AI 不会写代码”，而是写着写着目标漂了：顺手加功能、顺手重构、发现设计不对还硬往前冲，最后 issue 关不上，spec 也没有沉淀。

Do 模式的任务是把一个 issue 闭环推进：按目标和范围实现，按实现设计落代码，按验证方式证明结果。如果执行中发现 issue 或设计不成立，就停下来回写，不把漂移藏进代码里。

## 原则

**只做这个 issue。** 实现时只改目标 issue 范围内的代码。遇到顺手重构、额外功能、相邻 bug，先记到 issue 的执行记录或建议新 issue，不塞进本次变更。

**沿现状模型实施。** 开工前要能解释相关触发怎样产生结果，并知道变化直接落在哪里、哪些相邻行为需要验证。不能解释时先回到轻量 Explore / Design；现状跨越多个未知边界时升级为完整 Explore，不边写边猜。

**小偏差回写，大偏差停下。** 如果设计和代码现状冲突，小偏差可以更新 issue 的实现设计或执行记录后继续；大偏差要停止实现，说明 issue 边界或设计为什么不成立，建议切换到 Design 或 Talk 模式。

**先走最小实现梯子。** 开始写代码前，先判断这个改动是否仍然需要存在，仓库里是否已有 helper、类型、模块或同类模式，标准库、平台原生能力、数据库约束、CSS/HTML 或已安装依赖是否能覆盖，能否删除、收窄或用一两行完成。能复用就复用，能不写就不写，最后才写新的最小代码。

**最小不是草率。** 不要为了少写而跳过根因、输入校验、安全、可访问性、防数据丢失的错误处理、用户明确要求或必要验证。小而错不叫轻量，只是把复杂度推迟爆炸。

**渐进式实现。** 复杂实现可以先完成一段可验证的垂直切片或一个明确子步骤，跑对应验证，回写进展和未知，再继续下一段。停在清楚的中间态，比交出一堆无法解释的改动更好。

**验证要能说明问题。** 验证不是最后跑一下命令。bug 先复现或定位预期/实际分叉；feature 优先通过用户可观察行为、接口结果或集成路径验证；refactor/chore 验证行为不变。

**质量目标逐项兑现。** Issue 已选中的质量目标是实现与关闭约束，不是设计备注。实现必须遵循对应设计判断，验证必须留下对应证据；未选中的特性不在执行末尾补做形式化检查。

## 行动指南

### 先锁定可执行 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` 递归搜索 `.cs/issues/`，按标题、短语、关键词找候选并说明依据。找不到 issue 时不要凭空开工，先让用户指定或切换到 Talk 模式做初步规划并创建 issue。

开始前先复用当前上下文。下面是必须掌握的信息，不是每次全量重读清单；只有没读过、有修改迹象、需要精确引用/写回，或本次实现需要新增局部时才补读。

- 目标 issue：必须确认当前版本，掌握目标、范围、证据/背景/当前问题、质量目标、方案判断、实现设计、验证和关闭回写。
- 相关 spec：独立 issue 读取 `.cs/spec/` 的相关入口；epic issue 先读对应 epic 的 `spec.md`，必要时再读其引用的 project spec。
- 相关代码：确认实现设计仍然贴合代码现状。

先确认 issue 的 `## 现状如何工作` 或同职责内容足以解释本次相关路径；存在具体变化时，确认 `## 影响范围` 或同职责内容已区分必须修改、需要验证和仍待调查。简单改动可以很短；缺失且无法从当前上下文可靠补齐时，先切换到 Explore。

没有 `## 实现设计` 不一定不能做，但如果 issue 涉及多个功能段、请求/数据流、状态变化、或测试面不清楚，先切换到 Design 模式补设计；不要边猜边写。实现设计应当建立在现状模型和影响范围上，再讲整体方案、功能分工、请求/数据怎么走、哪些边界不碰、一步步怎么改、怎么确认做对，而不是只有字段清单。用户或团队要求详细测试设计时，也切换到 Design 一并补进实现设计。

### 实现和验证

按 issue 的目标、范围、质量目标和实现设计修改代码。对每个已选目标，确认实现设计已经说明“质量目标如何落实”，再按该判断实施。设计响应缺失或不成立时回到 Design；执行发现新的关键质量风险时也先补目标、来源、预期证据和设计响应，涉及阈值、显著成本、兼容策略或目标冲突时停下来让用户决定。不要等到 Close 才补。

如果实现设计里包含详细验证策略或 test-first 顺序，按它执行；否则为每个已选质量目标选择与失败代价匹配的证据，再完成 issue 类型要求的必要验证，不临时发明企业级测试计划。

实现后运行最相关的测试/检查。能跑自动验证就不要只手测；不能跑时说明原因和手动验证步骤。

如果下一步需要重新设计、补详细验证策略或用户确认危险操作，停下来汇报当前完成部分和阻塞点。

## 产物契约

完成后更新目标 issue：

- `## 执行记录`：改了什么、为什么、和设计是否有偏差。
- `## 验证`：实际跑过的命令或手动步骤、结果，以及它们分别支撑哪些已选质量目标。
- `## 关闭回写`：哪些结论应该在关闭时进入 project spec / epic spec 或 `notes/`。

实现发现现状模型或影响范围不准确时，更新原章节并记录证据；不要在执行记录里另存一份互相冲突的系统解释。

不要直接把长期结论写进 spec，除非用户明确要求收尾；通常等切换到 Close 模式关闭事项时再毕业沉淀。

## 收尾汇报

对用户汇报时用收束式汇报：先说这次实现主要解决了什么、采用了什么思路、兑现了哪些质量目标、守住了哪些边界、跑了什么验证；再列关键文件作为证据。不要按文件逐条散报，用户需要先理解结果，再看索引。

## 应用场景

- **用户指定一个 issue 要实现。** 读取 issue，按设计修改代码并验证。
- **设计已经完成，可以开写。** 执行 `## 实现设计`，必要时做小幅回写。
- **非 bug issue 要实现。** 按目标和设计推进 feature / chore / 小重构类事项。
- **已诊断完成的 bug issue 要执行剩余实现。** 优先确认 Complain 模式已写清反馈回路、根因定位和验证；缺这些就切换到 Complain，不要在 Do 模式里重新猜根因。

不适用：

- **不处理模糊需求。** 还没说清楚，切换到 Talk。
- **不创建或拆分事项。** 没有 issue 或 issue 太大，切换到 Talk。
- **不替代实现设计。** 设计缺失且风险较高，先切换到 Design。
- **不替代高风险设计判断。** 用户或团队要求详细测试设计时，先切换到 Design 补进实现设计。
- **不替代 bug 诊断。** 行为跑偏、报错、性能回退等 bug 全生命周期优先切换到 Complain。
- **不负责最终收尾。** 关闭事项和沉淀 project/epic spec、notes 切换到 Close。
