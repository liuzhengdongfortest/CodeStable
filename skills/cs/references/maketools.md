# Make Tools：学习未知流程

## 背景

Make Tools 模式用来走通一次未知流程，把人的带路变成项目能力。

它不是普通实现任务，也不是为了先写自动化脚本。第一次的目标是让 AI 在用户指导下完成真实流程，记录成功路径、危险点和判断标准。

## 原则

人是教练，AI 是学徒。AI 不要假装自己懂内部系统、登录态、审批流、后台页面或企业约定。遇到未知页面、权限、按钮、参数、确认框、报错和环境差异时，停下来让用户指导；每次用户纠正，立刻把纠正背后的规则记下来。

跑通后必写 `notes/`；只有几乎每次相关会话启动都必须知道这条流程时，才在 `AGENTS.md` 或 `CLAUDE.md` 加一行引用；只有稳定、重复、机械化的部分才进入 `tools/`。不要为了“装备制造”强行写脚本，很多内部流程只需要一篇好 notes。

流程 notes 遵循文档组织原则：面向下次执行者组织入口、步骤、危险点、人工节点和证据，不按这次带路的聊天顺序平铺。

如果流程会修改线上状态、发布、付款、删除、覆盖、重置、改权限或触发外部通知，执行前必须再次向用户确认。

## 行动指南

开始前先复用框架已经注入的项目约束和当前上下文，确认这次要学会的流程名称、目标、成功信号和危险操作已经掌握。准备写 Agent 指令、notes 或 tools 前必须确认目标文件的当前版本。

按真实操作推进，不要只写想象中的步骤。每一步记录入口、操作、必要输入、等待条件、成功/失败信号。遇到不懂的地方先问用户，不要猜按钮含义、权限含义或业务后果。

用户手动完成了某一步，也要记录它为什么必须人工做，以及 AI 下次应如何识别这个节点。如果需要命令、脚本、浏览器操作或系统工具，优先使用项目已有工具；没有再临时探索。

只把确实跑通过的路径写成确定步骤。没验证的分支写成“未知/待确认”。

## 产物契约

notes 必写：在 `.cs/notes/{NNN}-{名称}.md` 写流程笔记（`NNN` 取 notes 目录最大序号 + 1），正文包含流程用途、前置条件、操作步骤、成功判断、失败处理、危险边界和可自动化机会。

Agent 指令按需加引用：只有该流程是相关工作开始前的稳定前置时，才在已有 `AGENTS.md` 或 `CLAUDE.md` 追加一行轻量索引，而不是复制流程全文。

```markdown
- 做 X 流程前，先读 .cs/notes/{NNN}-{名称}.md。
```

tools 按需沉淀：只有当某个步骤稳定、重复、机械化，且自动化不会绕过必要确认时，才写入 `.cs/tools/`。工具说明必须包含输入、输出、运行方式、依赖、危险边界和验证方式。

## 收尾汇报

告诉用户流程是否跑通、notes 写到了哪里、是否增加了 Agent 指令引用、是否创建或更新了 tools、还有哪些步骤必须人工指导或下次继续跑通。

## 应用场景

用户说我带你跑一遍、这个流程 AI 不知道、企业内部流程、登录/发布/后台操作/网页操作/批处理流程要教给 AI、以后让 AI 自己会做时使用。
