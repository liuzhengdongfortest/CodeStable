# 原则：Debug

debug 的核心不是“把报错消掉”，而是找到行为为什么偏离预期，并确认修完后偏离消失。

## 背景

压力会诱人乱改：看到症状 → 猜一个原因 → 改一处 → 不行再改。看起来快，系统会变成猜测残留物。

CodeStable 默认允许**快路径**：钉住问题 → 定位最可能根因 → 试修 → 用原来的坏法确认。  
只有快路径走不通时，再升级到更严的证据链与假设检验（见下「升级」）。

## 默认：定位并试修

### 先把问题钉住

先说清三件事（几句话即可）：

- **预期行为是什么。**
- **实际行为是什么。**
- **在什么场景里分叉。**

用户步骤、堆栈、日志、截图、挂掉的测试都是证据。能顺手再点一次就再点；不能稳定复现就先靠已有证据，不要空猜。

### 追因果链，不追症状

从症状往回追：坏值从哪来？谁调用了这里？状态在哪里第一次变坏？

根因是**有效状态第一次变成无效状态的地方**，不是“这里报错了”。说不清中间环节时，链条还没立住——但仍可对**最可疑的一环**做一次最小试修；试修失败再升级，而不是一上来禁止动手。

### 试修

确认或高度确信根因后，做**最小修复**：消除因果断点，不在症状处堆特殊分支，不顺手重构。

修完用**同一观察**确认：同一操作 / 同一命令 / 同一测试 / 同一日志。好了就停。

## 升级：快路径不够时再用

出现这些情况再升级，而不是默认就升级：

- 试修后问题还在，或“好像好了”但解释对不上
- 根因无法收窄到一两处
- 偶现、环境相关、需要固定失败信号
- 连续两三次假设被推翻

升级时：

1. **复现与最小化**（能稳现则稳现）。
2. **可证伪假设**：“根因是 X，因为 …；若真，应看到 Y。”
3. **一次只动一个变量**；不要叠一堆改动“看会不会好”。
4. 需要时再补最小失败检查（失败测试、脚本或固定手测步骤）——这是升级手段，不是入场门票。
5. 修根因，回跑同一失败信号，并确认相关检查没被打坏。

### 修不动时，怀疑结构

连续两三次假设被推翻，不要硬试：

- 证据互相矛盾 → 心智模型可能错了，重读路径
- 本地好、CI/线上坏 → 环境、配置、依赖、时序、数据
- 每个修复都加新分支 → 可能是责任/接口设计问题
- 越修越散 → 回到归属与接缝（[code-design](code-design.md)）

## 应用场景

- 测试失败、构建失败、异常、性能、集成问题
- 用户给报错、堆栈、截图、“为什么不行”
- **已经试修没成功** → 停止乱补丁，走升级路径
- 问题越修越散 → 查架构/责任边界

不适用：无异常的全仓审计；正确行为都说不清时先澄清需求；用临时绕过假装根因已消。
