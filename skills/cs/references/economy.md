# 原则：实现经济性

实现经济性不是把代码写短，也不是少读代码。它是在理解真实路径、目标和质量承诺以后，用系统能够长期承受的最少新增复杂度完成变化。

## 背景

软件变化很容易把“以后也许会需要”伪装成今天的工程需要：为一个实现提前造接口，为一个固定值增加配置，为平台已有能力再包一层，或在症状旁边增加只服务当前调用路径的分支。这些代码即使正确，也会扩大理解面、测试面、升级面和故障面。

另一种失败是只追求行数少：跳过真实调用链，在离症状最近的位置补丁；用难懂的一行替代清楚的几行；删掉输入校验、数据保护或验证。这样的改动只是把复杂度和风险藏起来。

CodeStable 追求的是**最小充分变化**：在正确责任边界上解决当前问题，不提前购买没有证据支持的灵活性，并给非平凡行为留下足以发现回归的最小反馈回路。

## 原则

### 先理解，再最小化

最小实现梯子只能在现状模型成立以后使用。先顺清触发、责任、数据、状态、调用方和可观察结果，再判断哪一层能力足以完成变化。没有理解路径时，“最小 diff”往往只是改在最近的位置，而不是正确的位置。

不要把梯子变成无边界调研。当前路径和约束已经足以判断时，选择第一个完整覆盖目标的高位方案并继续。

### 从不新增开始

按下面顺序判断，命中后停止：

1. 目标是否已经由现有行为满足，或能否通过删除、收窄需求、调整配置或修正规格完成，而不新增行为。
2. 正确归属的现有模块、helper、类型、规则或项目模式是否已经提供能力；先搜索和复用，不建立平行实现。
3. 标准库、语言机制、浏览器 / 操作系统 / 数据库等平台原生能力是否完整覆盖。
4. 已安装依赖是否已经提供经过验证的能力；不要为几行稳定代码新增依赖，也不要在现成依赖旁边手写第二套。
5. 只有前面都不成立时，才编写完成当前目标所需的最小新代码。

“一行”不是独立目标。几种方案同样小，选择边界行为更可靠、命名更清楚、调用方更省心的一种。

### 在正确责任边界修一次

最小变化看的是**系统总复杂度**，不是当前文件的 diff。Bug 报告通常只给出一条症状路径；修改共享责任前要搜索它的调用方和相邻入口，确认无效状态最早在哪里产生。能在共同归属处消除根因，就不要在每个调用方各加一个 guard。

Feature 也遵循同一判断：能力属于现有模块就把该模块做深，不要因为入口离某个文件近就把规则放在那里。必要的结构调整可以先做，但必须直接降低当前变化的风险或认知负荷。

### 不为想象中的未来付费

没有真实变化维度时，不增加只有一个实现的接口、只有一个产品的工厂、没人会修改的配置、只转发调用的 wrapper，或“以后可能用到”的脚手架。新抽象、新接缝和新依赖都要能指出当前调用方、当前差异或当前风险。

这不是禁止扩展性。Project / Epic Spec 已经确认的兼容策略、部署差异、测试替身或替换需求，就是今天的证据，应按对应质量目标设计。

### 把有意简化做成有界决定

有时当前最合适的方案确实有已知上限，例如线性扫描只适合当前数据量、全局锁只适合当前吞吐、简单启发式只覆盖已确认输入。采用这类方案时，紧凑记录：

- **已知上限**：什么规模、环境或输入下会开始失效或明显退化。
- **升级触发**：出现什么可观察证据时需要重新设计。
- **升级方向**：届时应替换哪一层能力，而不是今天提前实现完整方案。

实现设计和执行记录保存本次决定；上限是稳定产品边界时在 Close 回写 spec，属于复用坑点时写 notes。只有未来维护者必须在代码现场看到才能避免误用时才留短注释，不推广项目专属标记，也不为每个普通取舍造债务台账。

没有上限或触发条件的“先这样”不是有界简化。升级触发已经发生、却仍会破坏 issue 目标或已选质量承诺时，也不能把它包装成可关闭的简化。

### 保护不能被省略的部分

最小化不能删除：

- 用户明确要求和相关 spec 的稳定约束；
- issue 已选中的质量目标；
- 信任边界的输入校验、权限与信息安全控制；
- 防止数据丢失或不可恢复副作用的错误处理；
- 可访问性基础、危险操作护栏和真实硬件需要的校准边界；
- 与失败代价相称的验证证据。

这些内容不是“额外保险”，而是完整解的一部分。

### 留下最小有用反馈

非平凡逻辑变化至少留下一个可运行、会在该行为再次损坏时失败的检查。优先复用现有测试入口和用户可观察接口；一个窄测试、最小脚本或已有集成检查足够时，不新建测试框架、fixture 体系或每函数一套测试。

简单文案、声明式配置或显然等价的一行变化不强制新增测试，但仍要运行最相关的现有检查。涉及解析、分支、循环、金额、权限、安全、迁移或数据丢失风险时，不能以“改动很小”为理由省略反馈回路。

同一行为被复制到生成文件、多宿主适配器或文档副本时，人工提醒不是反馈回路；优先建立单一来源，并用确定性生成或一致性检查防止漂移。

### 对收益保持诚实

可以汇报删除了什么、避免新增了什么，以及何时才需要升级；没有真实对照基线时，不声称“节省了多少行、多少成本或多少时间”。最小实现的价值由更小的责任面和可验证结果支撑，不靠虚构数字。

## 应用场景

- **Design**：比较实现方案、决定是否新增模块 / 接缝 / 依赖，并记录有界简化。
- **Do**：在正确归属处完成最小充分 diff，复用现有能力，留下最小有用反馈。
- **Complain**：从症状追到共享根因，一次修复所有同责任调用路径。
- **Close**：检查有界简化是否仍成立，把稳定上限和升级触发沉淀到正确层级。
- **技能或多宿主分发设计**：让核心行为只有一个来源，适配层保持薄，并用检查保护副本一致性。

不适用：没有具体目标时主动扫描全仓库寻找可删代码；为了减少行数牺牲可读性或质量承诺；把所有潜在优化提前建成 issue。
