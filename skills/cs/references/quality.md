# 原则：质量特征与承诺

软件质量不能只在设计时顺手提一句，也不能靠关闭前补一张检查表。CodeStable 使用 ISO/IEC 25010:2023 的产品质量模型作为统一语言，把值得承担的质量要求从 spec 传到 issue，再落实为设计判断、实现证据和关闭条件。

这不是合规或认证框架。标准负责提供共同词汇，CodeStable 负责把与当前变化真正相关的部分变成可执行、可验证、可关闭的承诺。

## 九项产品质量特征

下表使用英文标准术语作为锚点，中文是 CodeStable 的解释性名称。子特性用于追问和收窄目标，不是每次都要展开的检查项。

| 特性 | 当前变化需要关注什么 | 子特性提示 |
|---|---|---|
| 功能适宜性（Functional suitability） | 功能是否覆盖目标、结果正确、步骤适合任务 | 功能完整性、功能正确性、功能适当性 |
| 性能效率（Performance efficiency） | 响应、吞吐、资源和容量是否满足明确条件 | 时间特性、资源利用、容量 |
| 兼容性（Compatibility） | 是否能与其他产品共存并正确交换、使用信息 | 共存性、互操作性 |
| 交互能力（Interaction capability） | 指定用户是否能识别、学习、操作并获得必要帮助 | 适当性可识别、易学、易操作、用户差错防御、用户参与、包容性、用户辅助、自描述性 |
| 可靠性（Reliability） | 正常和故障条件下能否持续工作、容错并恢复 | 无故障性、可用性、容错性、可恢复性 |
| 信息安全性（Security） | 信息和操作是否抵御未授权访问、篡改与攻击 | 保密性、完整性、抗抵赖性、可核查性、真实性、抗攻击性 |
| 可维护性（Maintainability） | 预期维护者能否有效理解、修改、复用和测试 | 模块化、可复用性、可分析性、可修改性、可测试性 |
| 灵活性（Flexibility） | 产品能否适应需求、负载、使用情境和运行环境变化 | 适应性、可伸缩性、易安装性、可替换性 |
| 安全性（Safety） | 是否能避免危及生命、健康、财产或环境的状态 | 运行约束、风险识别、失效安全、危险告警、安全集成 |

Security 和 Safety 不能混用。Security 面向恶意行为、越权和信息保护；Safety 面向人身、财产与环境危害。交互能力描述产品支持交互的属性，不等同于完整的使用质量评价。模型来源见 [ISO/IEC 25010:2023](https://www.iso.org/standard/78176.html)。

## 质量承诺链

```text
Project / Epic Spec 中的稳定质量约束
                    ↓ 继承
Talk / Explore / Complain 的风险发现
                    ↓ 选择
Issue 中的具体质量目标
                    ↓ 落实
Design 判断 → Do 证据 → Close 检查与回写
```

**先扫描，再选择。** 用九特征检查当前变化可能伤到什么，但不要逐项填写“满足 / 不适用”。只选择会改变方案、实现或验收判断的特性；通常是少数几项，不设固定数量。

**继承与发现并用。** Project / Epic Spec 中与当前变化相关的稳定质量约束自动进入 issue。Agent 再根据影响范围、失败代价和证据主动补充候选目标。普通工程判断不逐项要求用户确认；指标阈值、成本显著增加、兼容策略和质量目标冲突等产品取舍必须交给用户。

**选中即承诺。** Issue 中每个质量目标都要写清：

- 特性，必要时收窄到子特性。
- 在什么条件下，系统应当达到什么可观察或可评估的结果。
- 目标来自 project spec、epic spec、用户决定，还是本次风险扫描。
- 准备用什么证据判断它是否达到。

“保证可靠性”“注意性能”不是质量目标。可执行的表达类似：“相同扣款请求重复提交时只产生一笔交易；以幂等集成测试验证”。普通功能正确性已经由 issue 的目标和验证覆盖时，不必为了凑分类重复选择功能适宜性；只有完整性、正确性或任务适当性本身构成关键风险或取舍时才单列。

**设计逐项响应，执行逐项举证。** Design 说明每个已选目标具体改变了哪个责任、接口、状态、数据、错误处理或交互判断。Do 实施这些判断，并在 `验证` 中留下对应证据。测试、静态检查、评审、基准、故障演练和手动路径都可以成为证据，强度与失败代价匹配。

**只按承诺关闭。** Close 不重新要求九项全扫描，只检查 issue 已选目标是否都有足够证据。执行中若暴露关键遗漏，先回到 Design 补目标和方案，不能在关闭结论里临时补一句。一次变化的证据留在 issue；长期仍然有效的质量约束和取舍回写 project spec 或所属 epic spec。

## 工程手段与质量特征

可测试性是可维护性的子特性。可观测性不是九项顶层产品质量特征，但常用于支撑可分析性、可靠性、信息安全性和验证证据。可访问性实践通常支撑交互能力中的包容性和用户辅助。不要为了对齐分类而丢掉这些工程手段，也不要把手段与最终质量目标并列成另一套模型。

架构原则、测试策略或团队自有质量模型可以继续存在，但要能映射回本项目采用的质量目标，避免同一 issue 出现互不相认的两套词汇。

## 应用边界

- Talk 用它发现质量影响和需要用户决定的取舍，候选项尚不是承诺。
- Spec 只保存长期有效、会约束后续变化的质量要求与取舍，不保存九项清单。
- Explore 只有在服务具体变化时才标出候选质量风险；纯 How it works 探索不产生质量承诺。
- Complain 用它识别被破坏的质量预期；Design、Do、Close 沿已选目标继续推进。
- 它不提供 ISO 合规判断、认证结论或完整测量方法，也不替代领域法规、功能安全分析和组织级质量制度。
