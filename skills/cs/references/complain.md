# Complain：处理行为投诉与 Bug

## 背景

Complain 模式是 LITE 的 bug 入口：坏了就走这里。它把 report、analyze 和 fix 压进同一个可关闭 issue，把反馈回路纪律压进执行过程。

## 原则

bug 不是从“猜哪行错了”开始，而是从“系统在哪个可观察场景里偏离预期”开始。完整闭环是：

`投诉记录 -> 反馈回路 -> 复现与最小化 -> 假设与定位 -> 修复与回归 -> issue 回写`

定位前先建立能抓住这个 bug 的反馈回路；没有 red-capable 回路时，不进入修复。复杂 bug 可以停在任一清楚中间态：已建 issue、已建反馈回路、已最小化、已定位假设、已修复待验证。不要为了显得完成，把没验证的判断写成事实。

只消除根因，不补表象，不顺手重构，不引入无关抽象。定位到共享责任后，搜索它的调用方和相邻入口，优先在无效状态最早产生的位置修一次，不在报告命中的单一路径旁边堆 guard。范围外发现写到 issue 的“顺手发现”，后续另开 issue。

**识别被破坏的质量承诺。** Bug 除了预期/实际分叉，还可能破坏 project / epic spec 已有的可靠性、兼容性、信息安全性等质量约束。使用九特征识别真正相关的目标；没有额外质量风险时不为了分类重复写功能正确性。被选中的质量目标必须进入修复设计、回归证据和关闭判断。

**Bug 因果链不等于 Explore。** Complain 围绕“预期为何变成实际”建立反馈回路、假设和根因链，可以使用从触发到结果的轻量系统理解；只有更大的现状机制本身说不清、阻碍诊断且值得独立复用时，才升级为完整 Explore issue。不要为了调一个明确 bug 额外制造探索事项。

## 行动指南

先复用框架已经注入的项目约束和当前上下文，确认 project spec 或相关 epic spec、已关闭 issue、notes 和代码里是否已经记录“正确行为”。没掌握、疑似变化或需要精确引用时再读；读取 spec 优先按索引和关键词定位相关局部。搜索 `.cs/issues/` 是否已有同类 bug；有就确认当前版本并更新原 issue，不重复建档。

投诉建档时，把用户输入拆成预期行为、实际行为、最小场景和证据。只有“已有预期被系统行为打破”才是 bug；预期说不清就继续问，如果是新需求、需求变更或产品取舍不明，切换到 Talk 或 Spec 模式。再从相关 spec 继承质量约束，并按失败代价补充会改变修复或回归判断的质量目标；阈值、成本和目标冲突仍由用户确认。

建立反馈回路时，优先选择能触达真实 bug 模式的最小接口：失败测试、HTTP 脚本、CLI、浏览器脚本、捕获重放、临时 harness，或针对非确定性 bug 的循环/并发/固定 seed。写清入口、断言症状、最近结果，以及是否 red-capable、确定、足够快、agent 可运行。

复现与最小化时，一次只删一个因素，删完回跑；留下的每个因素都应该是 load-bearing。

定位时先列 3-5 个可证伪假设，并写出预测：如果 X 是根因，观察或改动 Y 会让症状消失、变重或出现 Z 证据。每次只改一个变量；优先用调试器或 REPL，其次用有唯一前缀的临时日志。性能问题先建立基线和 profiling / query plan。

沿反馈回路从触发顺到异常结果，说明正常主路径、实际分叉点和根因传播范围。若连正常主路径都无法紧凑解释，先做轻量 Explore；跨越多个未知责任边界时再升级完整 Explore。

修复后必须回跑原始反馈回路、相关回归测试或手动复现步骤、影响面的冒烟检查，并为每个已选质量目标留下对应证据。清理临时埋点和 throwaway harness，除非明确移入 notes/tools。

修复若因外部限制采用有界简化，按 `economy.md` 记录已知上限、升级触发和升级方向。没有触发条件的 workaround、已经跨过上限的方案，或会继续破坏质量承诺的方案不能作为关闭依据。

## 产物契约

投诉成立时，在 `.cs/issues/YYYY/MM/DD/open-{短语}.md` 创建或更新 issue。新建时使用 `templates/entities/bug-issue.md`，frontmatter 至少写 `status: open`，类型写 `bug`。

issue 中至少维护：

- `目标`：修复后可观察的正确行为。
- `范围`：本次 bug 覆盖和不覆盖的边界。
- `当前证据`：预期、实际、最小场景、原始证据来源。
- `## 质量目标`：只记录相关特性、具体承诺、来源和预期证据；没有额外目标则删除该节。
- `## 反馈回路`：命令或操作入口、断言症状、最近一次运行结果和可运行性。
- `## 复现与最小化`：最小复现、必要因素、排除因素。
- `## 根因定位`：假设、证据、根因链和影响面。
- `## 执行记录`：实际代码改动、偏离和顺手发现。
- `## 验证`：回归命令、手动检查和结果。
- `## 关闭回写`：哪些稳定预期、约束、坑点或流程要进入 project spec / epic spec / notes / Agent 指令 / tools。
- `## 关闭结论`：根因摘要、修复摘要、验证摘要和遗留事项。

如果用户要正式关闭，再切换到 Close 模式做长期实体沉淀；如果只是修到可验证，也可以停在 issue 已更新的状态。

## 收尾汇报

用用户能听懂的话收束：先说预期和实际在哪里分叉、根因链是什么、修复主要改变了什么、哪些质量承诺被恢复、用什么反馈回路和回归检查保护了边界；文件和命令作为证据放后面。

## 应用场景

用户说不符合预期、行为跑偏、这个不对、debug、诊断、修 bug，或给出截图、日志、测试失败、性能回退时使用。

不用在没有具体异常时主动审计全仓库；不用处理新需求；不要复活 report / analysis / fix 三段 issue 技能。
