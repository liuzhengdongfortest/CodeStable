# Complain：处理行为投诉与 Bug

Bug 入口：先搞清坏在哪、可能根因是什么，**直接尝试修复**；修不好或根因不清，再升级到完整复现与反馈回路。复杂或需追踪的用 **受管理** issue；简单明确的可 **直接改** 或 **快改**。

**完成 ≠ 关闭。** 直接改/快改修完并验证即汇报；受管理事项正式关闭走 [close](close.md)。质量与经济性见 [quality](quality.md)、[economy](economy.md)；定位细则见 [debug](debug.md)。

## 原则

**默认快路径（先这样）：**

```text
钉住预期/实际 → 扫 .cs/ 与代码定位 → 最可能的根因 → 尝试修复 → 用原来的坏法再看一眼
```

**升级慢路径（默认修不好、或根因糊、或易回归时再用）：**

```text
复现与最小化 → 可重复的失败信号 → 可证伪假设 → 一次改一个变量 → 修根因 → 回归
```

不要一上来就搭正式 harness / 可变红回路。用户给的步骤、日志、堆栈、挂掉的测试，本身就可以当“坏什么样”的信号。

只消除根因，不补表象，不顺手无关重构。定位到共享责任后，在无效状态最早产生处修一次，搜索调用方与相邻入口。范围外发现向用户说明；另开 issue 须用户同意。

**识别被破坏的质量承诺。** 用九特征识别真正相关目标；无额外风险不为凑分类重复写功能正确性。

**Bug 因果链 ≠ Explore issue。** 可用**现状说明**讲清主路径与分叉；仅当更大机制本身说不清、阻碍诊断且值得独立复用时，才建 Explore issue。

## 行动指南

### 1. 准备

复用已注入约束与上下文。**强制**：对 `.cs/` 做一轮主题检索（`SKILL.md` 制度记忆）——project/epic spec 定正确行为，notes 常有解法，issues（含 `-x-`、`ff`、`done/`）防重复踩坑。需受管理时，命中同类 bug 则更新原 issue。用户明确要求直接修 → 不强制建档。

把输入拆成：预期、实际、最小场景、证据。不是“已有预期被打破”则继续问，或转 Talk/Spec。管理强度见 `SKILL.md`。

### 2. 默认：定位 → 试修 → 看一眼

1. 沿报错、堆栈、用户路径或相关代码，找到**最可能**的根因（状态第一次变坏的地方）。
2. 做最小修复；不顺手重构。
3. **用原来坏的方式再验证一次**（同一操作、同一命令、同一测试、同一日志点）。好了就收；说明改了什么、为什么。

主路径完全说不清 → 先做短现状说明；跨多未知边界 → Explore issue。

### 3. 升级：默认路径不够时

出现任一情况，再上完整手段（见 [debug](debug.md)）：

- 试修后症状还在，或修好了但预测对不上（像碰巧遮住）
- 根因说不清，只能瞎猜
- 偶现 / 难复现，需要固定失败信号
- 核心路径、金额、权限等，值得留下可重复检查

升级时再做：复现最小化、明确失败信号（手测步骤也行，不必须新 harness）、3–5 个可证伪假设、一次改一个变量、修完回跑同一信号并做相关回归。

有界简化按 [economy](economy.md) 记录；无上限的 workaround 不能当关闭依据。

### 4. 清理与回写

修完清理临时埋点与 throwaway harness（除非移入 notes/tools）。受管理则回写 issue 各节；快改落 `ff`；真相变了同步 project spec。

## 产物契约

**受管理**：`.cs/issues/{NNN}-o-{名称}.md`，模板 `bug-issue.md`。至少维护：目标、范围、当前证据、质量目标（无则删节）、根因定位、执行记录、验证、关闭回写、关闭结论。若走过升级路径，再补反馈回路 / 复现与最小化。关闭时 `{NNN}-x-{名称}.md`。

**直接改 / 快改**：简单 bug 可直接改；要可追溯走 [fast](fast.md)。汇报：分叉、根因、修了什么、怎么确认好了。默认不 commit。正式关闭受管理事项 → Close。

## 收尾汇报

预期/实际差在哪、根因是什么、改了什么、怎么确认好了（默认路径还是升级路径）。文件与命令作证据。

## 应用场景

不符合预期、行为跑偏、debug、日志/截图/测试失败、性能回退。

不适用：无具体异常的全仓审计；新需求；复活 report/analysis/fix 三段技能。
