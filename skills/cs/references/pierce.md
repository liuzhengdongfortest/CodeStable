# 穿刺：按风险打通需求通路

为**某个明确需求**做风险驱动的垂直打通：先让关键风险点证明「走得通」，再铺开完整实现。  
不是限时可扔实验，也不是全仓探索；打通的结果通常就是正式实现的第一刀。

用户说「穿刺」「先打通」「先把风险点走通」「垂直切片先通」时进入本模式。也可从 Design / Do 在风险高时建议切换，**由用户确认**后进入。

## 背景

需求失败常见原因不是写得慢，而是**最难的那一环到最后才碰**：鉴权、落库、外部 API、迁移、主路径状态机。前面铺了很多周边，卡在中间。

穿刺把顺序反过来：**先戳穿会卡死交付的点**，用最小可运行路径证明可通，再加厚边界、UI 和次要能力。

与相邻模式：

| 模式 | 差别 |
|---|---|
| Explore | 解释**现状**怎么跑；穿刺证明**这次需求的未来通路**可通 |
| Design | 安排实现；穿刺**动手验证**风险，可补充或修正设计 |
| Do | 按已确认范围完整实现；穿刺优先打通，再交 Do 加厚，或同一会话连续做完 |
| 快改 | 小改直接交付；穿刺用于**有明显技术/集成风险**的需求 |
| Review | 审已有改动；穿刺在写主实现前/中打通 |

## 原则

**有明确需求才穿刺。** 目标糊 → Talk；只想懂现状 → Explore。

**先风险，后厚度。** 顺序由「不确定 + 失败代价」决定，不由「好写/好看」决定。

**打通 = 可观察地走通一次。** 每个风险点要有触发 → 关键处理 → 可观察结果；不是只写了类型或注释。

**最小但真实。** 用真实路径或尽量贴近真实的路径；mock 只挡不可控外部，不 mock 掉本次要证明的那一环。

**结果可留下。** 打通的代码默认是正式实现的骨架，不是必须扔掉的 spike；若方案作废，删掉并记录结论。

**不通就停。** 某个风险点走不通 → 换方案、收窄需求或问用户，不假装设计已成立。

**先查 `.cs/`。** 制度记忆协议见 `SKILL.md`；spec/notes/issue 里可能已有边界与坑。

## 行动指南

### 1. 锁定需求与风险表

一句话说清：做完后用户/调用方应观察到什么。

列出本次 **2–5 个风险点**（只写会卡死或大改方案的），例如：

- 主路径端到端能否通  
- 鉴权/权限  
- 读写与数据形状  
- 外部依赖  
- 迁移/兼容  
- 关键状态与失败恢复  

每个风险点写清：**怎样算打通**（一条可重复的验证）。

范围过大 → 建议收窄或进 epic/issue；用户决定。

### 2. 排序

按「不确定 × 失败代价」降序。最先做最可能推翻方案的点。

### 3. 逐点穿刺

对当前风险点：

1. 最小改动接到真实归属（见 [code-design](code-design.md)、[economy](economy.md)）。  
2. 跑通「怎样算打通」的验证。  
3. 记下：通了 / 不通 + 原因 + 对方案的影响。  

一点不通：停止加厚，先处理方案或问用户。  
主路径尚未端到端通之前，不优先做样式打磨、次要入口、大范围重构。

### 4. 主路径端到端

风险点分别通之后，串成一条：**触发 → 关键步骤 → 可观察结果**。  
能演示或脚本跑通一次，再进入加厚。

### 5. 交接

- **继续实现**：切 Do（或用户授权下本会话直接加厚）。  
- **方案变了**：回写 Design / issue / epic spec 中仍成立的部分。  
- **需要留痕**：受管理则写入 issue（风险表、已打通证据、剩余工作）；或极短 note；快改级小需求可用 `ff` 记结论。  
- **不要**把未稳定的试探写进 project spec。

## 产物契约

默认以**对话 + 可运行证据**为主。

受管理时建议在目标 issue 增加或更新：

- `## 风险与穿刺`（或并入实现设计）：风险表、顺序、每点验证与结果、主路径是否已端到端通、剩余加厚项。

无 issue 时对话交付即可；用户要求可写 `notes/{NNN}-pierce-….md`（结论与验证方式，不写流水账）。

## 收尾汇报

1. 需求一句话与风险表  
2. 哪些已打通（验证是什么）  
3. 哪些不通或仍未知  
4. 对方案/范围的影响  
5. 下一步：Do 加厚 / 改方案 / 等用户决定  

## 应用场景

- 新能力跨层、接外部系统、有迁移或权限不确定性  
- 用户说先穿刺、先打通、先保证风险点能走  
- Design 发现多处「不确定是否可行」  

不适用：

- 纯文案/明显一行的小改 → 快改或直接改  
- 需求未清 → Talk  
- 只理解现状 → Explore  
- 用户未要求且风险很低 → 不必强行进穿刺
