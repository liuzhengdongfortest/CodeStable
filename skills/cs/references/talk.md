# Talk：讨论与初步规划

洞察真实诉求（用户开口常是方案或表面诉求），收束成可判断的初步安排。聊完后用户认知应更清楚；出口是快改、直接改、独立 issue、Explore issue、epic、Vision，或继续澄清。

管理强度与写入口见 `SKILL.md` 决策表。质量候选规则见 [quality](quality.md)；UI 草图见 [ui-spec](ui-spec.md)。

## 原则

先看清是什么，再让用户看见他看不见的，最后收住边界、终点与出口。

**先讨论，再落盘。** 不边聊边维护整理稿。阶段性复述让用户纠正；仅当用户明确说“差不多了 / 整理一下 / 落盘 / 写个 talk”后写入 `.cs/talks/`。

**能查先查，不明白再问。** 不把猜测当需求或事实。偏好可商量，约束不能绕过；稳定约束最终进 project / epic spec。

**做第二层推演。** 主动想影响谁、改什么工作流、牵动哪些历史约束、长期代价。风险按场景选，不机械全列。

**具体变化做质量扫描。** 用九特征找会改变方案/验收/长期代价的候选；不逐项问“是否需要”。候选项在 Talk 中尚非承诺。

**UI 分歧优先画出来。** 讨论草图标“目标”或“示意”；确认前不是 spec。

**一次一个分叉。** 每个问题给推荐答案与理由。

**初步规划是收束的一部分。** 出口草案须含目标、范围、验证、依赖、暂不纳入与理由。管理是否值得见 `SKILL.md` 表「快改 / 直接改 / Issue / Epic」。

**只记录确认过的边界。** “不做什么”须用户说过或同意你提出的排除项。

## 行动指南

复用已掌握上下文；项目有 `.cs/` 时先做一轮主题检索（`SKILL.md` 制度记忆：路径 + grep，深读以 project/epic spec、notes、相关 issue 为重）。想法来自 Vision 或落在已有 epic 时深读对应分支。

提问只问缺口：已从仓库确认什么、还缺什么、当前最关键分叉。按“事实 → 推荐 → 请用户决策”。

至少一轮影响扫描（兼容、数据、命名、外部依赖、发布/回滚、维护成本等）；涉及质量时用 [quality](quality.md)。候选质量目标写成草案（条件、结果、来源、证明方式），仍标为候选。

用户未确认停止前：不写 talks，不更新 Vision，不建 issue/epic。

收住条件：真问题一句话且用户确认；做什么有轮廓、已确认/未确认边界清楚；最大未知已标出。最多三到五轮深挖；争的是事实且半小时内能验证时，动手查比继续聊更省。

收住后给出口草案；用户确认安排后再落盘并执行对应出口。

## 产物契约

确认前无文件。确认后：`.cs/talks/{NNN}-{名称}.md`（扁平，无日期目录；`NNN` = 该目录已有最大序号 + 1，至少三位），保留：原始想法、真问题、术语、已确认决策、偏好与约束、影响与取舍、候选质量目标（无则省略）、UI 草图（无则省略）、已确认/待确认边界、最大未知、初步出口草案。按阅读路径组织，不复刻聊天顺序。

用户只确认整理 talk → 只写 talk。同时确认安排时再执行出口：

- **快改** → [fast](fast.md)（完成后 `NNN-x-ff-名称.md`）
- **直接改** → Do（可不建 issue；要留痕则升快改）
- 目标全景 → Vision
- 独立 feature/chore/refactor → `.cs/issues/{NNN}-o-{名称}.md` + 对应模板
- 新 epic → `.cs/epics/{NNN}-o-{名称}/spec.md`
- Explore issue → [explore](explore.md)（`{NNN}-o-{名称}/`）
- bug → [complain](complain.md)
- 已有 epic 规格变化 → 谈记录决策后 [spec](spec.md)

## 收尾汇报

确认前：聊清了什么、仍不确定什么、质量风险候选、建议出口。落盘后：真问题、约束、候选质量、未知、最终安排，再列写入文件。路径不作主要成果。

## 应用场景

局部想法未清、方向摇摆、交代不完整、想先聊再动手。

不适用：维护 Vision/当前真相/epic 活规格；写实现设计；已明确 bug；写代码。全景构想 → Vision；上游规格大体稳定只差落到开发者可读规格 → Spec。
