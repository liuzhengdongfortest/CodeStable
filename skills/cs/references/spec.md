# Spec：维护当前真相与活规格

把当前理解写进正确层级的 spec：项目主线进入 `.cs/spec/`，大需求活规格边界进入 `.cs/epics/YYYY/MM/DD/{短语}/spec.md`。

## 背景

真实开发不是“规格全清楚再实现”。通常先讨论出一个初始需求，形成 epic，切出一批 issue；做着做着发现规格补充、修改甚至推翻，于是回到 spec 重新整理，再调整本轮推进范围、生成新 issue、修改旧 issue 或关闭旧 issue。

Spec 模式的职责不是记流水账，而是维护“当前为什么这样”的说明。project spec 是主线真相；epic 是一条有边界的活规格和知识承载单元。独立 issue 可以直接引用 project spec；epic issue 先服从 epic spec，等 epic 人工关闭后再把毕业结论合并回 project spec。

## 原则

**文档面向查看。** spec 写出来是给开发者看的。入口文档先让读者知道“这是什么”，再让读者选择“往哪里深入”。不要把 `index.md` 写成文件清单。

**主叙事围绕项目怎么用。** project spec 先讲开发者如何使用当前项目完成事情：有哪些入口、能力、场景、约束和扩展路径。代码分层、目录结构和模块名只是帮助落手的索引，不是组织 spec 的主线。

**spec 是树，不是二级目录。** 简单内容一篇写完；复杂子系统可以继续拆层。只要一个目录下有多篇文档，就要有该层 `index.md` 统领方向感、统一语言和阅读路径。

**统一语言就近放置。** 不另起 domain 体系。术语放在离它生效范围最近的 `index.md`；术语太多时可拆相邻 `terms.md`，但必须由该层入口指过去。

**写考量，不写流水。** 不创建 `changes.md`。规格变化要呈现为当前需求、架构考量、被排除方案、边界和影响，而不是“某时从 A 变成 B”。

**核心理解不引用代码。** spec 的正文按能力、流程、概念、边界和取舍组织；代码路径、函数名和命令结果只作为证据索引或落点。读者不应该靠读代码路径来推断需求和架构。

**主线只收毕业结论。** epic 过程中的变化先进入 epic spec；epic 下 issue 关闭也先回写 epic spec。只有 epic 人工关闭后，才把稳定结论合并回 project spec。

**质量约束属于规格，不属于检查表。** 只有长期有效、会约束后续设计或验收的质量目标与取舍才进入 project / epic spec。按九特征统一语言，但不列九项“满足 / 不适用”；一次变化的测试结果、基准数据和执行证据留在 issue。

**Epic 是活规格边界。** 只有跨模块、会多轮变化、会分批 issue、或规格会在一个可圈住范围内反复演进时，才创建 epic。它不是任务桶，也不是 project spec 的缩小版；它负责让变化留在一个人类能 review、AI 能持续对齐的范围里。

**Epic spec 同时承载当前推进。** 不必等整个 epic 全局完美。某一批目标、范围、依赖和验证足够清楚，就在同一个 `spec.md` 的“当前推进”中标出可推进范围、issue 列表、暂停或废弃项和剩余阻碍。推进视图不是独立真相，也不替 issue 写实现设计。

## 行动指南

先判断目标层级：

- 项目长期真相、独立小需求沉淀、全局统一语言或架构方向 → 更新 `.cs/spec/`。
- 大需求、跨模块变更、会分多批 issue、规格会反复变化 → 更新 `.cs/epics/YYYY/MM/DD/{短语}/spec.md`。
- 没有现成 epic 但明显需要一条变更线 → 先说明为什么需要 epic；用户同意后创建目录型 epic。

更新 project spec 时，先读 `.cs/spec/index.md`。按阅读路径找到最近的子层；缺少合适位置时，先补该层 `index.md` 或新建子层入口，不要把结论散落成平级文件。

更新 epic spec 时，先读该 epic 的 `spec.md`、它明确引用的相邻材料和相关 project spec。epic spec 要说明它相对 project spec 改变什么、为什么现在这样设计、哪些术语只在本 epic 生效、哪些范围本轮可推进、哪些还悬着。

写入 spec 时，先完成面向人的核心理解，再按需补证据索引。核心段落不要把文件、类、函数或命令当主语；它们只说明“这个理解可以从哪里验证”或“执行者到哪里落手”。如果一个章节只能按代码目录命名，先问它对应什么使用场景、能力或决策路径。

涉及质量要求时读取 `quality.md`：记录特性或子特性、适用条件、具体约束和关键取舍。指标阈值、兼容策略、成本显著增加或目标冲突由用户确认；普通工程约束可以根据已确认需求和证据直接维护。Epic 切 issue 时，只把与该 issue 相关的质量约束继承过去。

遇到未知系统、术语、角色、状态和流程，先查 `.cs/`、README、代码、历史 issue/epic、notes 和 project spec。查不到或存在多个可能含义时，向用户提带证据的问题；需要顺清“触发如何产生结果”且无法在当前上下文紧凑解释时，切换到完整 Explore。

## 产物契约

project spec 默认入口是 `.cs/spec/index.md`，使用 `templates/entities/project-spec-index.md`。子层入口用 `templates/entities/spec-section-index.md`。

epic 默认结构只有一个权威入口：`.cs/epics/YYYY/MM/DD/{短语}/spec.md`。使用 `templates/entities/epic-spec.md`；不要创建重复职责的 `index.md`、`plan.md` 或 `changes.md`。材料复杂时可增加按内容命名的相邻文档，但不能承载第二份状态或计划，并必须由 `spec.md` 说明为什么读、何时读。

`spec.md` 必须覆盖：当前方案、需求变化、架构考量、统一语言、本轮可推进范围、暂不推进范围、未确认问题、合并回 project spec 的候选；存在长期质量约束时还要覆盖适用条件、具体目标和取舍，不生成九项空清单。

`spec.md` 中的“当前推进”只记录当前可推进范围、issue 列表、暂停或废弃 issue 和剩余阻碍；不要在这里复制 issue 正文或实现设计。

## 收尾汇报

先讲 spec 更新后的当前判断：这次澄清了什么、为什么现在这样考虑、哪些质量约束会影响后续 issue、哪些范围本轮可以推进、哪些仍悬着、是否影响已有 issue。再给文件路径作为索引。

## 应用场景

更新 project spec、创建或调整 epic spec、规格补充/修改/推翻、PRD/SE 文档需要落到开发者能读的规格、实现中发现原规格不成立时使用。

不适用：目标本身还在脑暴或需要初步规划时切换到 Talk；单个 issue 的实现方案切换到 Design；当前系统怎样工作仍说不清时切换到 Explore；代码执行切换到 Do。
