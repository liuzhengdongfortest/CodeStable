# Design：为明确 Issue 做实现设计

为一个具体 issue 做实现设计，把“要做什么”讲成“实现该怎么安排”。

## 阅读入口

- 先读[原则](#原则)，确认先解释现状再设计未来。
- 实际设计时读[行动指南](#行动指南)，从轻量 Explore 进入实现判断。
- 写回 issue 时读[产物契约](#产物契约)，按现状、影响、质量目标、方案和验证的顺序表达。

## 背景

Talk 模式先和用户澄清目标并做初步规划，把讨论结果或 epic spec 的当前可推进范围落成 issue；但 issue 还不一定能直接开写。很多失败发生在这里：目标看起来清楚，却没有先顺清现有触发怎样产生结果、影响会沿哪里传播，就直接讨论能力归属、接口和改法，于是实现一路漂移。

Design 模式补的就是这一步。它面向一个明确 issue，先理解需求和当前代码，再把实现前的关键判断写回 issue。它不是大而空的架构文档，也不是单独重构；它只服务这个 issue，让后续 Do 模式可以沿着设计写代码、验证、回写。

## 原则

**设计必须贴着代码。** 实现设计不能只从文字推演。要查看真实代码来确认现有能力、功能分工、命名、数据流、状态、错误、配置、外部依赖和测试面。

**先解释现状，再设计未来。** 默认先做一次轻量 Explore：用紧凑叙事讲清当前触发如何产生结果，有具体变化时再区分必须修改、需要验证和仍待调查的影响。现状模型解释不清、横跨多个责任边界或值得长期复用时，先切换到完整 Explore。

**写设计，不写实现。** 设计回答功能如何分工、请求或数据怎么走、每部分负责什么、行为边界、实施顺序、验证面、风险和未知；不要写伪代码长篇铺陈，不要提前实现，不要为了“设计感”造抽象。

**像教新同事一样解释。** 实现设计不是字段表，也不是给 AI 自己看的备忘录。用接近教程的自然语言讲清：功能上分成哪几部分，A 为什么负责接收请求，B 为什么负责处理请求，请求/数据从哪里来、经过哪里、在哪里结束，哪些边界不碰。工程名词、模块名和函数名只作为证据和落点，不要当正文骨架。

**优先扩展已有归属。** 能用现有模块做深，就扩展现有模块；只有真实变化维度出现时才引入接缝。

**选择最小充分方案。** 在理解现状和质量目标以后使用 `economy.md` 的实现梯子：优先不新增、删除或收窄，复用正确归属的已有能力，再考虑标准库、平台原生能力和已安装依赖，最后才写新代码。最短方案不一定最小；改在错误责任边界的局部补丁会增加系统总复杂度。

**按风险选择质量目标。** 设计不是只面向功能。先从 project / epic spec 继承相关质量约束，再用九特征补充会改变设计或验收的目标；通常只有少数几项，不设固定数量。目标必须写成条件、结果、来源和预期证据，选中后就是本 issue 的设计与关闭约束。

**测试设计并入实现设计。** 普通 issue 在“怎么确认做对”里写清验证面即可；当测试策略本身复杂、用户要求 TDD、或团队要求测试设计时，先和用户讨论测试层级、行为用例、数据环境、执行方式和 test-first 顺序，再写进实现设计。不要为了流程完整单独制造测试文档。

**渐进式设计。** 复杂设计可以先完成一段可验证判断，说明已确认范围、暂不覆盖范围、下一段要解决的问题。不要把还没读懂的模块、数据流或风险写成确定结论。

**遵循文档组织原则。** 设计文档要服务两个读者：用户判断方向是否对，执行者知道从哪里下手。不要把模块名、函数名和待办项平铺成一串。

## 行动指南

### 先锁定目标 issue

必须有一个明确 issue。用户给路径就读该文件；没给路径就用 `rg` 递归搜索 `.cs/issues/`，按标题、短语、关键词找最相关候选，并说明判断依据。找不到 issue 时不要凭空设计，先让用户指定或切换到 Talk 模式做初步规划并创建 issue。

开始前先复用当前上下文。下面是必须掌握的信息，不是每次全量重读清单；只有没读过、有修改迹象、需要精确引用/写回，或当前设计需要新增局部时才补读。

- `cs` 技能：当前上下文没读过才读。
- 目标 issue：必须确认当前版本，掌握目标、范围、证据/背景/当前问题、质量目标、方案判断、验证和关闭回写。
- 相关 spec：独立 issue 读取 `.cs/spec/` 的相关入口；epic issue 先读对应 epic 的 `spec.md`，必要时再读其引用的 project spec。
- 相关代码：确认当前实现组织和 issue 是否匹配。

### 先建立现状模型

沿目标 issue 最相关的一次触发走到可观察结果，说明入口、关键责任、数据和状态变化、外部副作用，以及与本次目标有关的分支。已经有可靠的 `## 现状如何工作` 就复用并按代码现状校准；没有就补一段轻量说明。

存在具体变化时，再从变化点沿现状模型写 `## 影响范围`：

- 必须修改：变化直接穿过的位置。
- 需要验证：共享契约、调用方、持久化数据、缓存、测试、可观测性或副作用等可能受波及但理论上应保持的区域。
- 仍待调查：证据不足或边界不清的区域。

简单 issue 可以各用几句话。若现状无法紧凑解释、影响跨越多个能力或证据互相冲突，先切换到 Explore 建立独立探索 issue，不要在 Design 里假装已经理解。

### 选择质量目标

读取 `quality.md`，先把相关 spec 中长期有效的质量约束继承到 issue，再根据现状模型、影响范围和失败代价识别遗漏。普通工程判断由 Agent 直接给出依据；指标阈值、显著成本、兼容策略或目标冲突交给用户决定。未选中的特性不写“不适用”。

每个已选目标写清特性或子特性、适用条件下的具体结果、来源和预期证据。普通功能正确性已由 issue 目标与验证覆盖时，不为凑分类重复选择功能适宜性。测试性、可观测性、日志或 mock 等工程手段应映射到它们支撑的质量目标，不另造并列分类。

### 形成实现判断

基于现状模型和影响范围，再看需求在功能上应该分成几段、每段由谁负责、应该沿用什么命名。判断请求、数据、状态、错误、配置和外部依赖将怎样改变，以及新增行为应该在哪个用户可观察行为或接口上验证。

用实现梯子检查整体方案：当前目标是否已经被满足，能否删除或收窄，能否复用正确归属的现有能力，是否已有标准库、平台能力或依赖覆盖。不要为了展示完整性增加单实现接口、无人修改的配置、纯转发 wrapper 或未来脚手架。

如果选择了带已知上限的简单方案，在实现设计中写清上限、升级触发和届时应替换的能力层；没有这类取舍时不要增加空字段。升级触发已经发生，或简化会破坏已选质量目标时，改选方案而不是把风险留给 Do。

然后逐项响应本 issue 的质量目标：说明它具体改变了哪个责任、接口、状态、数据、错误处理、交互或部署判断，以及为什么这些安排足以产生预期证据。不要只贴“可靠性”“性能”标签，也不要在实现设计里重新定义一套质量目标。

如果验证策略需要单独想清楚，就在设计阶段处理：测可观察行为，不测内部形状；优先通过公开接口、用户路径、服务边界或集成路径验证；mock 只用于不可控外部依赖，不要 mock 掉本该一起验证的本地协作。接口、数据或运行命令还不清楚时，标成待确认并向用户讨论，不要假造具体测试。

如果一次设计会横跨多个模块、多个用户路径或多个关键取舍，先产出第一段可验证设计，用户确认后再继续。

## 产物契约

按认知顺序把结果写回目标 issue：先更新 `## 现状如何工作`，有具体变化时更新 `## 影响范围`，再更新 `## 质量目标` 和 `## 实现设计`。已有相同职责章节就更新，不要制造第二份现状、影响分析、质量目标或设计。

`## 现状如何工作` 从一句话模型开始，再按需展开触发、主路径、关键责任、数据、状态和相关分支。`## 影响范围` 使用“必须修改 / 需要验证 / 仍待调查”，不把文件清单当结论。

`## 质量目标` 只列本 issue 已选中的目标。每项包含特性 / 子特性、具体目标、来源和预期证据；没有需要单列的质量目标时删除该节，不生成九项空清单。

`## 实现设计` 用教程式自然语言，而不是只填字段。推荐写法：

- **这次要怎么做**：先用一小段话讲整体方案，让非实现者也能判断方向。
- **功能怎么分工**：用自然语言说明分成哪几部分，各自负责什么；模块名只放在括号或句尾作为落点。
- **请求/数据怎么走**：讲输入从哪里来，经过哪些处理，在哪里改变含义或形状，最后谁消费。
- **哪些边界不碰**：说明支持什么、不支持什么、兼容性和危险边界。
- **有界简化（如有）**：在相关方案或边界中写清已知上限、升级触发和升级方向；没有就不单列。
- **质量目标如何落实**：逐项说明已选质量目标怎样改变责任、接口、数据、状态、错误处理或交互；可测试性、可观测性等工程手段写在它们所支撑的目标下面。
- **一步步怎么改**：给 Do 模式可执行的步骤，但不要展开成伪代码。
- **怎么确认做对**：说明验证打在哪些行为上，还有哪些未知。非平凡逻辑至少设计一个会在行为损坏时失败的最小可运行检查，优先复用现有测试入口，不为简单改动搭建新框架。需要详细测试设计时，写清测试目标、测试层级、行为用例、测试数据和环境、执行方式，以及是否 test-first。

写回后检查 issue 是否仍然可关闭：目标、范围、质量目标、实现设计、验证要能互相咬上。如果设计发现 issue 太大或边界错了，停下来说明原因，建议切换到 Talk 模式重新澄清和拆分。

## 收尾汇报

汇报先让用户听懂现状怎样工作、变化会触及哪里，再讲这件事准备怎么实现。用教程式语言讲整体方案：功能分成哪几部分，各部分负责什么，请求或数据怎么流；再把关键模块、接口和文件作为落点补充。

然后说清执行者真正需要注意的点：实施顺序是什么、每个质量目标怎样被设计响应、哪些风险或未知还没盖住、验证应该打在哪些行为上。文件和章节位置只作为索引放在后面，不要用模块名和函数名堆成一份只有实现者才看得懂的清单。

## 应用场景

- **用户指定某个 issue 要先设计。** 读取 issue 和代码，把实现设计写回 issue。
- **issue 已经有目标和范围，但还不能直接开写。** 补齐功能分工、请求/数据流、边界、验证和实施顺序。
- **改动前需要顺清逻辑和影响。** 先做轻量 Explore；解释不清时升级为完整 Explore issue。
- **实现前发现设计风险。** 在写代码前把风险和未知暴露出来，必要时建议拆分或回到计划。
- **用户或团队要求测试设计。** 在实现设计里补齐详细验证策略，而不是另走独立技能。

不适用：

- **不处理模糊想法。** 需求还没说清，切换到 Talk。
- **不创建事项。** 没有 issue，先切换到 Talk。
- **不写代码。** 实现切换到 Do。
- **不做全仓库架构审计。** 只围绕目标 issue 读必要代码。
