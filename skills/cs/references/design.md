# Design：为明确改变做实现设计

把“要做什么”收成“实现该怎么安排”；有 issue 时写回，无则对话交付。不写代码、不建 issue。

全局规则（管理强度、质量、实现经济性、UI、完成≠关闭）见 `SKILL.md` 与 [quality](quality.md)、[economy](economy.md)、[ui-spec](ui-spec.md)。本文件只写 Design 增量。

## 阅读入口

- [原则](#原则) → 先现状说明再设计
- [行动指南](#行动指南) → 锁定目标、现状、质量、方案
- [产物契约](#产物契约) → 有/无 issue 的交付形状

## 背景

目标明确不等于实现判断成立。未先讲清触发如何产生结果就谈归属与接口，实现容易漂移。Design 先贴着代码建立现状与影响，再形成可执行的实现安排。

## 原则

**先查 `.cs/`，再贴代码。** project / epic spec、notes、历史 issue 里有代码读不出的取舍与解法；检索协议见 `SKILL.md`「`.cs/` 是代码之外的制度记忆」。**Design 在形成方案前必须完成该检索**（本会话同主题已扫过且无新写入除外）。

**设计必须贴着代码。** 在 `.cs/` 上下文之上确认现有能力、分工、命名、数据流、状态、错误、配置、依赖与测试面。

**先现状说明，再设计未来。** 默认做跨模式**现状说明**（见 [explore](explore.md)）。解释不清、跨多边界或值得复用时，升级为 Explore issue，不在此硬推方案。

**写设计，不写实现。** 回答分工、数据/请求路径、边界、顺序、验证、风险与未知；不写伪代码长篇，不提前实现，不为“设计感”造抽象。

**像教新同事。** 自然语言讲清分成几部分、各负责什么、东西怎么流、哪些边界不碰；模块名只作落点。

**优先扩展已有归属；方案走最小充分梯子。** 细节见 [economy](economy.md)、[code-design](code-design.md)。

**UI 规格不是组件树。** 有线框图/状态标注时先确认仍是当前契约，再映射到组件归属、状态与验证；视觉分歧回 Talk / Vision / Spec。见 [ui-spec](ui-spec.md)。

**按风险选质量目标。** 继承 spec 约束，用九特征补会改变设计/验收的目标。受管理写入 issue；直接改的设计在对话结果中保留必要边界与证据计划。见 [quality](quality.md)。

**测试设计并入实现设计。** 策略复杂、用户要求 TDD 或团队要求时，在设计阶段写清；不另造测试技能文档。

**渐进式设计。** 复杂时可先交一段可验证判断；未读懂的部分不写成确定结论。

## 行动指南

### 锁定目标与管理方式

必须有明确目标，不强制有 issue。目标模糊 → Talk。范围/风险已需受管理 → 说明理由建议建 issue，由用户决定（决策表见 `SKILL.md`）。

### 强制：检索 `.cs/` 制度记忆

项目存在 `.cs/` 时，**在写实现设计之前**必须完成一轮检索（协议见 `SKILL.md`）：

1. 对整个 `.cs/` 做主题相关的路径浏览 + 关键词 grep（模块名、能力名、错误名、用户术语）。
2. **深读权重**：相关 **project spec** 入口/子层为重中之重 → 相关 **epic spec**（若在 epic 内或命中）→ 命中的 **notes** → 命中的 **issues**（含 closed）→ 按需 talks/vision/tools。
3. 写前再确认：目标 issue（若有）当前版本；将引用或回写的 `.cs` 文件当前内容；相关代码局部。

已掌握且无变化迹象的段落不机械重读；**未做过本主题 `.cs/` 扫描则不得跳过。** 检索中发现的既有取舍、有界简化、坑点与失败尝试，必须进入后续设计判断（采纳、显式排除或标为待确认），不得假装不知道。

### 现状说明与影响

沿最相关触发走到可观察结果，并用 `.cs/` 命中项校准“为何如此”。有 issue 且已有可靠 `## 现状如何工作` 则复用并校准；直接改的设计在对话中给紧凑模型。

有具体变化时写影响三层：必须修改 / 需要验证 / 仍待调查。不够则升级 Explore issue。

### 质量目标与实现判断

读 [quality](quality.md)：从 **project / epic spec 继承**约束 → 按影响与失败代价补候选 → 受管理写入 issue，直接改在结果中说明必要目标。未选项不写“不适用”。

基于现状与 `.cs/` 既有取舍做功能分工、数据/状态/错误路径与验证面判断。涉及 UI 时确认图标记（当前/目标/示意）未混用。用 [economy](economy.md) 梯子检查；有界简化须写上限、触发、方向。逐项响应已选质量目标。

验证策略：测可观察行为；mock 仅不可控外部依赖。接口或命令未清时标待确认，不假造测试。

跨多模块/多路径/多关键取舍时，先交第一段可验证设计，用户确认后再继续。

## 产物契约

**有目标 issue（受管理）**：按认知顺序写回 `## 现状如何工作`、`## 影响范围`、`## 质量目标`、`## 实现设计`。同职责章节更新，不另开第二份。

**无 issue（直接改的设计）**：仅对话交付；不创建 `.cs` 文件。用户随后要快实现 → [fast](fast.md)；否则 Do 复用当前上下文。

`## 质量目标`：只列已选目标（特性/子特性、具体结果、来源、预期证据）；无则删除整节。

`## 实现设计` 推荐结构：这次要怎么做 → 功能怎么分工 → 请求/数据怎么走 → 哪些边界不碰 → 有界简化（如有）→ 质量目标如何落实 → 一步步怎么改 → 怎么确认做对。

写回后检查目标、范围、质量目标、设计、验证能否互相咬合。issue 过大或边界错 → 停，建议回 Talk 澄清。

## 收尾汇报

先讲现状与影响，再讲整体实现安排（分工与数据流），再讲顺序、质量响应、风险与验证。文件路径作索引放后。

## 应用场景

- 简单明确改变先设计 → 对话交付，不建 issue
- 指定 issue 设计 → 写回 issue
- 有目标但还不能开写 → 补分工、流、边界、验证与顺序
- 改前需顺逻辑 → 先现状说明；不够则 Explore issue

不适用：模糊想法 → Talk；写代码加厚 → Do；**先按风险打通可运行通路** → [pierce](pierce.md)；全仓架构审计。
