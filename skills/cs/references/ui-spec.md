# 原则：UI 可视化规格

UI 的很多需求不是一句“增加设置页”能说清的。区域怎么排列、信息如何分组、动作和状态是否放在一起、用户从哪里进入下一步，都带有空间关系。只用文字描述，会迫使每个读者在脑中重画一次界面；不同人往往画出不同结果。

UI 可视化规格用可版本化的简图把这些关系放进 spec。图不是装饰，也不是提前做高保真设计稿；它与文字标注一起构成当前界面或目标变化的需求契约。

## 什么时候必须画

UI 变化出现以下任一情况时，必须提供可视化规格：

- 区域位置、分栏、分组、层级或控件共置关系会影响需求理解。
- 页面、面板、弹窗、导航或多步骤流程之间存在关系。
- 同一界面有会改变结构或可用动作的关键状态，例如加载、空、错误、禁用、未登录、执行中或二次确认。
- 响应式尺寸会改变布局结构，而不只是等比例缩小。
- 当前界面与目标界面的差异仅靠段落难以快速判断。

纯后端能力、简单文案、颜色或设计 token 调整，以及不涉及空间和状态歧义的单一控件变化，不强制画图。不要为了模板完整生成空线框图。

## 选择合适的图

### ASCII 线框图表达界面结构

默认使用简短 ASCII / Unicode 线框图表达区域、相对位置、层级、主要控件和可见状态。它可以直接进入 Markdown、参与版本对比，也便于人和 AI 一起修改。

```text
┌─ 账号设置 ─────────────────────────────┐
│ 当前状态：未登录                  [登录] │
├──────────────┬─────────────────────────┤
│ 服务商列表    │ 服务商详情               │
│ ● OpenAI     │ API Key  [••••••••]     │
│ ○ Local      │ 工作目录 [选择文件夹]     │
│              │                [保存]    │
└──────────────┴─────────────────────────┘
```

线框图只保留会影响理解的节点。不要用大量装饰字符模拟像素，不要在 spec 中提前写组件名、CSS 属性或实现层级；精确尺寸只有构成真实约束时才出现。

### Mermaid 表达流程和状态

页面跳转、操作流程、状态迁移或多分支反馈更适合 Mermaid。不要用流程图假装表达二维布局，也不要把同一关系同时画成两种图。

### 截图和原型表达视觉证据

现有截图、设计稿和可运行原型可以帮助说明视觉风格、密度和细节，但不能成为唯一规格：它们不易搜索和差异比较，也常常无法区分稳定要求与偶然像素。把它们作为证据或相关材料链接，并用线框图与标注保留需要长期维护的结构契约。

## 一张 UI 规格需要说明什么

图和相邻文字共同回答：

- **角色与入口**：谁在什么场景进入这个界面，要完成什么任务。
- **图示状态**：这是当前界面、目标界面，还是只用于讨论的示意。
- **结构关系**：主要区域、信息层级、控件共置和导航方向。
- **交互与状态**：哪些动作改变什么状态，用户会看到什么反馈。
- **稳定约束**：哪些位置、层级、可见性或动作关系必须保留。
- **示意部分**：哪些文字、尺寸、顺序或视觉细节仍可由后续设计决定。

可以给复杂图加 `① ② ③` 等编号，在图下逐条标注。只展开与当前需求有关的状态；加载、空、错误、禁用、确认和响应式不是固定检查表。

涉及交互能力时，用 `quality.md` 判断易操作、用户差错防御、包容性、自描述性等是否需要成为质量目标。线框图能澄清目标，但不是达成目标的验证证据；最终仍需通过可运行界面、自动化检查、可访问性检查或用户路径验证。

## 真相放在哪一层

### Project Spec 保存当前稳定界面

Project Spec 可以保存当前信息架构、页面关系、长期布局模式和稳定交互原则。图必须描述已经成立的当前界面；尚未实现的目标不能提前混入主线真相。

### Epic Spec 保存目标变化

Epic 涉及大范围 UI 变化时，画当前与目标，或至少画目标并说明相对 Project Spec 改变什么。每张图明确标记“当前 / 目标”，只保留当前仍成立的方案；被推翻版本留在历史证据，不在 `spec.md` 并排堆叠。

### Issue 保存本次局部变化

独立 UI feature、UI bug 或 epic 下的具体 UI issue，只画本次变化涉及的局部界面和状态，并链接所属 Project / Epic Spec。Bug 图明确区分实际与预期；不要复制整张产品界面。关闭后，稳定结构按归属回写上层 spec。

### Design 解释怎样实现

Spec 和 issue 规定用户看到什么、怎样操作以及哪些关系必须成立；Design 再说明这些要求如何映射到组件归属、状态管理、数据流、错误处理和验证入口。Design 不能用实现方便为理由静默改写线框图，也不要把组件树反向塞进 UI spec。

## 保持图与文字一致

图负责空间和关系，标注负责行为、边界和取舍，两者没有静默覆盖优先级。出现冲突时，先判断哪一方已由用户最新确认并同步修正另一方；无法判断就把冲突标成未确认问题，不能让实现者自己猜。

同一张图不要混合“当前”和“目标”。纯示意图必须明确写出哪些部分不构成约束。图发生实质变化时，同时检查相邻交互说明、质量目标、issue 范围和验证方式，避免两套规格漂移。

## 应用场景

- **Talk**：用目标草图快速暴露布局、层级和状态分歧；确认前它只是对话中的候选。
- **Spec**：把已确认的当前或目标 UI 关系维护在正确真相层级。
- **Design**：读取视觉契约并映射到实现，不重新定义需求。
- **Close**：把已经验证、仍然稳定的 UI 结构与交互关系回写 Project / Epic Spec。

不适用：把 UI spec 做成像素级设计系统、用图片替代所有行为文字、为每个小样式改动画图，或在没有 UI 的项目中保留空章节。
