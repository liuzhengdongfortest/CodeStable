# Review：按需审代码

**动作，不是流程步骤。** 只有用户明确要求 review / 评审 / 看看这改动 / 审一下 PR 时才进入。实现、快改、关闭都**不**自动 Review。

尺子主要借 [code-design](code-design.md)（深度、归属、接缝、边界），辅以 [economy](economy.md) 与相关 [quality](quality.md) 约束。默认只出判断与建议，**不擅自改代码**；用户说“按 review 改”再切 Do / 快改。

## 原则

**用户要才做。** 不问不审；不在 Do/快改末尾默认塞一轮。

**有范围才审。** 用户指定的 diff、分支、PR、文件/目录或“刚改的那批”。没有范围就问一句要审哪一块，**不做无边界全仓审计**。

**先对照意图与 `.cs/`，再挑结构。** 这批改动想解决什么？project/epic spec、目标 issue 是否说得通？怪写法是否已有记录取舍？

**用 code-design 的词审，不另发明一套。** 重点看：

- **归属与命名**：能力是否在该在的模块？有没有平行同义词？
- **深度**：接口是否浅（调用方几乎没省认知）？有没有纯穿透层？
- **接缝**：是否为想象中的扩展提前造 adapter/工厂？
- **边界**：校验、错误、危险操作是否散落或缺失？
- **经济性**：能否不新增、能否复用、有无无界的简化？
- **可测性**：测试是否扎在可观察行为，还是绑死内部形状？

**按严重度说话，不堆清单。** 先给结论（能不能合 / 主要风险），再列少数高信号问题；每条说清：现象、为何要紧、建议怎么改（可选）。吹毛求疵的风格/命名小偏好往后放或省略。

**默认只审不改。** 除非用户授权“直接修 review 里的问题”。

## 行动指南

1. **锁定范围**：diff / 路径 / PR。读相关代码与测试；扫 `.cs/` 中与本范围相关的 spec/issue/notes（协议见 `SKILL.md`）。
2. **对照意图**：用户或 issue 说要什么；实现是否越界、是否漏了验证。
3. **过 code-design 尺子**：归属、深度、接缝、边界；需要时用 economy 梯子看是否过重。
4. **质量与风险**：只提会改变正确性或失败代价的点（信息安全性、数据丢失、危险操作等）；不机械过九项表。
5. **组织汇报**（见下）。用户要落地修改 → 列建议后问是否改，或按其授权切 Do/快改。

## 产物契约

默认**无强制落盘**。对话交付即可。

用户要求留下痕迹时：可写入目标 issue 一节，或极短 note（`notes/{NNN}-review-….md`）；不要为 review 新建流程实体。

## 收尾汇报

1. **一句话结论**：可合 / 有条件可合 / 建议先改再合。  
2. **必须处理**（若有）：阻塞正确性、安全、数据、明显错误归属。  
3. **建议处理**：深度/接缝/经济性等中信号问题。  
4. **可选 / 观察**：风格、非阻塞整洁度（可省略）。  
5. 每条尽量带位置（文件或符号）与改法方向。  

## 应用场景

- 用户说 review、评审、看看这 PR/这 diff、实现完帮我审一下  
- 合并前想要第二意见  

不适用：

- 未要求时的默认步骤  
- 无范围全仓挑刺（那是主动审计，code-design 也禁止）  
- 当 Design 用（Design 在写码前安排实现；Review 在已有改动上挑问题）  
- 当 Complain 用（行为已偏离预期 → Complain）
