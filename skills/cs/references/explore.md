# Explore：现状说明与系统理解

围绕有边界的问题，沿“触发如何穿过系统并产生可观察结果”重建因果模型。先回答**现状机制说明**，存在具体变化时再分析影响；不先设计未来，不以补文档为目的。

**现状说明**是跨模式动作（Design / Do / Complain 都会用）；**Explore issue** 是仅在需要完整、可复用探索时建立的独立事项。二者不是同一件事。

## 阅读入口

- 先读[原则](#原则)，分清现状说明与 Explore issue、以及与 Design / Complain 的边界。
- 调查时读[行动指南](#行动指南)。
- 写回或关闭时读[产物契约](#产物契约)。

## 背景

修改陌生代码前，通常会把相关逻辑顺一遍：入口、控制与数据流、状态变化、责任参与、可观察结果。没有这层理解，影响只能靠猜。Explore 把这种理解变成显式能力：多数时候做**现状说明**即可；链路复杂、证据冲突或理解值得复用时，再升级为 **Explore issue**。Project Spec 缺口可能是结果，不是起点。

## 原则

**从触发走到结果。** 最小叙事单元是一次动作/请求/事件如何穿过系统产生结果，不是模块目录。

**先现状，后改变。** 只解释当前如何工作与仍未知处。未来改法属 Design；预期为何变成实际属 Complain。

**渐进式披露。** 一句话模型 → 主路径 → 相关责任/数据/状态/分支 → 证据。读者停在任一层都应得到可用理解。

**代码是证据，不是叙事骨架。** 主叙述用自然语言；路径与命令放证据层。

**影响分析有变化才出现。** 分“必须修改 / 需要验证 / 仍待调查”。纯理解不虚构影响范围。

**质量风险是候选。** 仅服务具体变化时可用九特征标风险；是否承诺由后续澄清或 Design 决定。纯现状理解不扫九项。细则见 [quality](quality.md)。

**足够行动就停止。** 不追求穷尽系统。

**现状说明 vs Explore issue**

| | 现状说明 | Explore issue |
|---|---|---|
| 何时 | 相关路径能用紧凑叙事讲清 | 讲不清、跨多边界、证据冲突、需多篇路径文、或理解将被多处复用 |
| 写哪 | 对话 / 目标 issue 的 `## 现状如何工作`（及有变化时的 `## 影响范围`） | `.cs/issues/{NNN}-o-{名称}/` |
| 关闭 | 随目标 issue 或直接进入 Design / Do | 用户确认后 Close：稳定现状机制说明毕业到 Project Spec |

## 行动指南

### 选择形态

先明确探索问题与要支持的决策。

- 范围清楚、一篇紧凑说明够用 → **现状说明**（有目标 issue 则写回该 issue；直接改则留在对话/工作上下文）。
- 需要独立工作区 → **Explore issue**（须用户授权创建或落盘；仅解释/评审不自动建档）。

### 建立一句话模型与主路径

先答：什么触发、经什么关键机制、产生什么结果。一句话仍说不清就缩小边界，不用模块名掩盖缺口。

从真实触发顺到可观察结果：入口与输入 → 责任与决策点 → 数据读写与变换 → 状态与副作用 → 结果如何被确认。先主路径，再按当前问题展开关键分支。

### 用证据校准

**先检索 `.cs/`**（协议见 `SKILL.md`「制度记忆」）：对 `.cs/` 路径浏览 + 关键词 grep，再按权重深读——**project spec 最重**，其次相关 epic spec、notes、历史 issues（含 closed）。路径名本身常已点题，不必通读全树。

再读 README、代码、测试、配置；必要时跑只读命令或最小验证。spec/notes 解释“为何这样写”；notes/issue 常有现成解法与失败边界。冲突或业务判断不明时，带着已确认事实问一个会改变理解的问题。用户纠正后只保留最新主叙述；旧误解仅在 Explore issue 的修正记录中保留。

### 分析变化影响（有明确变化点时）

说明变化触及输入、行为、数据形状、状态、不变量还是外部契约，再分三层：

- **必须修改**
- **需要验证**
- **仍待调查**

写行为与责任，不把长文件清单当结论。会影响方案或验收时，用 [quality](quality.md) 标**候选**质量风险，不伪装成已承诺目标。

## 产物契约

### 现状说明

- 有目标 issue：更新 `## 现状如何工作`；有具体变化时更新 `## 影响范围`。已有同职责章节则更新，不另开第二份。
- 直接改 / 对话内设计：在当前上下文给出紧凑模型即可，不为此建 Explore issue 或形式清单。
- 候选质量风险留在影响描述中供 Design 选择。

### Explore issue

路径：`.cs/issues/{NNN}-o-{名称}/`，入口 `index.md`（`templates/entities/explore-index.md`），路径文章用 `explore-article.md`。关闭时目录改为 `{NNN}-x-{名称}/`。`NNN` 取 issues 下最大序号 + 1。

`index.md` 是认知地图：探索问题与停止条件、一句话模型、边界、阅读路径、当前结论、与具体变化的关系、用户修正、未知、候选毕业位置。路径文章按“某触发如何产生结果”组织，不按调查轮次或代码目录。

### 停止与关闭

可停止当：能讲清触发—过程—结果；相关责任/数据/状态/分支有证据；未知显式标出；有变化时影响已分层。

- 现状说明 → 继续 Design / Do（或 Complain 诊断）。
- Explore issue → 仅用户确认关闭后走 [close](close.md)：稳定现状机制说明按渐进式披露合并到 Project Spec；影响分析留目标 issue；证据与已排除理解留 Explore issue。

## 收尾汇报

先一句话讲系统怎样工作，再讲展开了哪条路径、哪些影响已确认/仍未知、是否足以支持下一步。路径与证据放后面。

## 应用场景

改前顺逻辑、理解一块系统、Design 发现现状不足、多处复用同一理解、或 Project Spec 需要稳定机制说明时使用。

不适用：需求脑暴 → Talk；方案已明只需实现设计 → Design；**为需求按风险打通未来通路** → [pierce](pierce.md)；行为偏离预期 → Complain；规格已确认只写入 → Spec；无边界全仓审计。
