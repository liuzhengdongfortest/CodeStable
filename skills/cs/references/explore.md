# Explore：理解系统如何工作

围绕一个有边界的问题，沿“触发如何穿过系统并产生可观察结果”的主线，重建现有系统的因果模型。Explore 先回答 **How it works**，再在存在具体变化时分析影响范围；它不先设计未来，也不以补文档为目的。

## 阅读入口

- 先读[原则](#原则)，理解 Explore 与 Design / Complain 的边界。
- 开始调查时读[行动指南](#行动指南)，按一句话模型、主路径、影响和证据逐层推进。
- 准备写回或关闭时读[产物契约](#产物契约)，选择轻量或完整形态。

## 背景

人准备修改陌生代码时，通常不会先列文件或直接画方案，而会把相关逻辑顺一遍：动作从哪里进入，控制和数据怎样流动，状态在哪里改变，哪些责任参与，最后产生什么结果。没有这层理解，影响范围只能靠猜，设计也容易落在错误位置。

Explore 把这种系统理解变成显式能力。它既可以作为目标 issue 内的一次轻量调查，也可以在链路复杂、证据冲突或理解值得复用时升级为独立探索 issue。Project Spec 缺口可能是结果，但不是 Explore 的起点。

## 原则

**从触发走到结果。** 默认最小叙事单元不是模块或目录，而是一次用户动作、请求、事件、定时任务或状态变化如何穿过系统，最终产生可观察结果。

**先现状，后改变。** Explore 只解释当前如何工作、哪里仍未知。未来怎样修改属于 Design；bug 为什么偏离预期属于 Complain。不要把猜测中的新方案写成现状。

**渐进式披露是认知顺序。** 先给一句话模型，再讲主路径，只展开与探索问题有关的责任、数据、状态、分支和边界，最后给证据。读者停在任何一层，都应得到完整且可用的理解。不要为了缩短文件机械拆分，也不要一次展开所有调用和异常。

**代码是证据，不是叙事骨架。** 主叙述用自然语言解释行为和因果；文件、函数、命令、日志和运行结果集中放在证据层，说明它们支撑哪条理解。

**影响分析有变化才出现。** 面向具体改动时，从变化点沿现状模型判断必须修改、需要验证和仍待调查的区域。纯粹理解系统时，只说明上下游、依赖边界和未知区域，不虚构影响范围。

**轻量优先，必要时升级。** 当前 issue 的相关路径能用紧凑叙事讲清，就直接写回该 issue；解释不清、横跨多个责任边界、证据冲突、需要多人校准，或理解会被多个 issue 复用时，升级为完整 Explore issue。

**足够行动就停止。** Explore 不追求穷尽系统。只要现状模型已经足以支持当前决策，相关分支有证据，未知被显式标出，存在变化时影响范围也已分层，就可以停止。

**完整探索才进入独立工作区。** 独立 Explore 使用 issue 目录承载认知地图、路径文章、用户修正和证据。关闭前都是候选理解；用户确认关闭后，只有稳定的 How it works 结论毕业到 Project Spec。

## 行动指南

### 选择轻量或完整形态

先明确探索问题和当前需要支持的决策，不要从“读完整个模块”开始。

- 已有目标 issue，相关路径范围清楚，可以用一篇紧凑说明讲明白：轻量 Explore。
- 现状无法可靠解释、链路跨越多个能力或外部系统、需要多篇路径文章、证据互相冲突，或结果值得长期复用：完整 Explore。

如果只是用户要求解释、评审或报告，不自动创建文件。只有用户已经授权更新目标 issue，或明确要求建立探索 issue / 落盘，才写入工作区。

### 建立一句话模型

先回答：什么触发了这块系统，它经过什么关键机制，最终产生什么结果。这个模型应该让没有读过代码的人也能判断方向是否正确。

一句话仍说不清时，先缩小探索边界；不要用更多模块名掩盖理解缺口。

### 顺主路径

从一个真实触发开始，沿主路径追到可观察结果：

1. 入口怎样接收触发和输入。
2. 哪些责任依次参与，在哪里改变含义或作出决定。
3. 数据在哪里读取、转换、持久化和消费。
4. 状态在哪里发生变化，产生哪些外部副作用。
5. 最终结果由谁观察，成功怎样被确认。

先把主路径讲通，再按当前问题展开关键分支：失败、重试、权限、缓存、并发、提前结束、兼容行为或外部依赖。与当前问题无关的分支只说明存在，不继续深挖。

### 用证据校准

优先读取相关 Project / Epic Spec、目标 issue、README、代码、测试、配置、历史 issue 和 notes。必要时运行只读命令、最小调用或测试来验证路径。证据冲突、术语不明或需要业务判断时，明确当前推断和卡点，再向用户提一个会改变理解的问题。

用户纠正后，先改写现状模型和受影响路径，让主叙述只保留最新理解；旧误解、纠正来源和排除原因只在完整 Explore issue 的“用户修正与已排除理解”中保留。

### 分析变化影响

只有存在明确变化点时才执行。先说明变化改变的是输入、行为、数据形状、状态、不变量还是外部契约，再沿现状模型分三层记录：

- **必须修改**：变化直接穿过的责任、数据或状态位置。
- **需要验证**：理论上不应改变，但共享契约、调用方、持久化数据、缓存、测试、可观测性或副作用可能受波及的区域。
- **仍待调查**：证据不足、边界不清或依赖外部判断的区域。

影响范围写行为和责任，不把长文件清单当结论；代码位置只作为落点。

### 渐进式表达

无论轻量还是完整形态，都按以下阅读顺序组织，但不强迫每篇文档使用完全相同的标题：

1. 读者先带走的一句话模型。
2. 触发、主路径和可观察结果。
3. 与当前问题有关的责任、数据、状态和关键分支。
4. 可选的影响范围与未知。
5. 代码、文档、命令和运行结果组成的证据索引。

关系比文字更容易看懂时才画简短流程图、状态图或数据流图。图先展示关键节点和方向，细节继续留在下层文字与证据中。

## 产物契约

### 轻量 Explore

把结果写回目标 issue：

- `## 现状如何工作`：一句话模型、触发到结果的主路径，以及本次真正相关的分支和边界。
- `## 影响范围`：仅在存在具体变化时写，区分“必须修改 / 需要验证 / 仍待调查”。

简单 issue 可以各用一小段，不为形式扩写。若已经有相同职责的章节，更新原章节，不制造第二份现状或影响分析。

### 完整 Explore

完整探索使用 `.cs/issues/YYYY/MM/DD/open-{短语}/`，入口是 `index.md`，使用 `templates/entities/explore-index.md`；路径文章使用 `templates/entities/explore-article.md`。

`index.md` 是认知地图，不承载全部细节。它维护探索问题与停止条件、一句话模型、探索边界、渐进式阅读路径、当前结论、与具体变化的关系、用户修正、未知问题和候选毕业位置。

路径文章按“某个触发如何产生结果”组织，而不是按调查轮次、问题编号或代码目录组织。每篇只承担一条可独立理解的路径；简单探索一篇即可，复杂时再增加路径文章。

### 停止与关闭

满足以下条件即可停止探索：

- 能自然讲清触发、关键过程和可观察结果。
- 与当前问题有关的责任、数据、状态和分支已有证据。
- 未确认部分被明确标出，没有伪装成结论。
- 面向具体变化时，影响已经分成必须修改、需要验证和仍待调查。

轻量 Explore 随目标 issue 继续进入 Design / Do。完整 Explore 只有在用户确认关闭后才切换到 Close：稳定的 How it works 理解按渐进式披露原则合并到 Project Spec；与某次改动绑定的影响分析留在目标 issue；代码证据、调查过程、已排除理解和未解决问题留在 Explore issue。

## 收尾汇报

先用一句话讲清系统怎样工作，再讲本次真正展开了哪条路径、哪些影响已经确认、哪些仍未知，以及现有理解是否足以支持下一步。最后才给 issue、路径文章和证据位置。

## 应用场景

需要理解一块系统如何工作、修改代码前顺清逻辑和影响范围、Design 发现现状模型不足、多个 issue 反复依赖同一系统理解，或 Project Spec 需要稳定的机制说明时使用。

不适用：需求本身还在脑暴时切换到 Talk；未来方案已经明确只需实现设计时切换到 Design；已有行为明确偏离预期时由 Complain 建立反馈回路和根因链；规格已经确认只需写入时切换到 Spec；默认不做无边界的全仓库审计。
