# 快改：小改快速通道

小需求、用户要快时使用。不写 Design / 检查表 / 验收清单；**动手前轻扫 `.cs/`，动手后必留 `ff` issue**。可追溯，但不走完整受管理仪式。

触发示例：快速、快改、fastforward、别走流程、直接开干、小改一下（且规模配得上）。

## 原则

**不改变“小功能会直接动手”。** 只加两层保护：改前指到制度记忆；改后留下最简 `ff` 痕迹。

**入口轻，出口有纪律。** 禁止动手前建空壳 issue；代码与验证完成后再写 `ff`（同会话做完可直接落 `NNN-x-ff-名称.md`）。

**命中的 spec/decision 是约束，不是参考。** 违反要么先改 spec（须授权），要么别按违反方式实现。

**复杂就跳车。** 硬推快改的成本高于切回 Design / 常规 issue。

## 命名与序号

见 `SKILL.md` Issue 约定：

- 进行中：`.cs/issues/{NNN}-o-ff-{名称}.md`
- 已完成：`.cs/issues/{NNN}-x-ff-{名称}.md`
- `NNN` = 已有最大序号 + 1（至少三位）
- 模板：`templates/entities/ff-issue.md`，`type: ff`

## 行动指南

### 1. 改前两问（轻量 `.cs/` 扫描）

对整个 `.cs/` 做路径浏览 + 关键词 grep，再按命中深读：

1. **这块以前栽过吗？** → `notes/`、issues（含 `-x-` 与历史 `ff`）
2. **有没有已拍板的写法/边界？** → **project spec**（重中之重）+ 相关 epic spec

本会话同主题已扫过且无新写入可复用。没命中按自己判断写很正常。

快改不要求完整现状说明；一句话能说清触发→结果即可。说不清或跨多边界 → 跳出快改。

### 2. 实现时守住的红线

- **先想放在哪儿**：优先扩展正确归属的现有模块；别往顺手文件里塞。
- **只写用户明确要的**：不提前抽象、配置开关、工厂、无关兜底。
- **只动该动的**：同文件无关丑代码不顺手改；发现记到「顺手发现」。
- **默认最少代码**：细节见 [economy](economy.md)。
- **反射信号就停**（任一条出现就评估是否跳出）：改动跨 3+ 能力边界；要引入新术语或冲突统一语言；要动 project/epic 已定模块/产品边界；范围翻倍；文件/函数已明显过载还在硬塞。

### 3. 验证

跑最相关检查；不能跑则说明手测步骤。用户确认效果 OK 后再落 `ff` 痕迹（用户明确说不用确认则验证通过即可落盘）。

### 4. 回写 `ff` issue（必做）

从用户请求抽 2–4 词英文短横线作 `{名称}`。取下一 `NNN`。

同会话已完成：直接写 `.cs/issues/{NNN}-x-ff-{名称}.md`，`status: closed`。  
需跨会话：先 `NNN-o-ff-...`，完成后再改名为 `NNN-x-ff-...`。

正文只填：

- 做了什么
- 改了哪些
- 怎么验证的
- **对 `.cs/` 的影响**（必选其一：无影响 / 已同步 spec / 已知漂移 / 应升级流程）
- 顺手发现（可选）

写得轻：总共十几行通常够。不要迷你 Design。

真相失效且确定 → 同步相关 project spec 小节，并在「对 `.cs/` 的影响」写路径。不确定 → 标漂移并问用户。不自动改 Vision。

### 5. 收尾

告知用户 `ff` 路径已落盘。默认不 commit；用户同意时可 scoped 提交：本次代码 + 该 `ff` issue +（若有）spec 同步。

可选一句提示（用户说不用则跳过）：是否把长期约束升进 project/epic spec；是否代为 commit。

## 什么时候跳出快改

任一情况停下，建议切 Design 或常规 issue：

- 改动涉及 3 个以上子系统/能力边界
- 需要引入新术语或与统一语言冲突
- 要改 project / epic 已定边界
- 用户追加需求使范围翻倍
- 需要多轮验收或正式质量承诺清单

已写代码在后续 Design/issue 中标「已部分实现」即可。

## 不做什么

- 不写 Design / 九项清单 / 完整验收表
- 不在动手前建空壳 `ff` issue
- 不把 `ff` 写成迷你 Design 或常规 feature issue
- 不因「要快」跳过 `.cs/` 轻检索与验证
- 不自动 push / 部署

## 收尾汇报

做了什么、怎么验的、`ff` 路径、对 `.cs/` 的影响、是否建议 commit。顺手发现单独列出。

## 应用场景

小功能、小修复、布局微调、用户明确不要流程。

不适用：模糊需求 → Talk；明确要设计 → Design；bug 走 Complain（默认定位试修，修不好再升级）；大边界变化 → Epic / 常规 issue。
