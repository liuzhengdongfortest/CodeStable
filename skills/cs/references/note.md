# Note：记录可复用知识

## 背景

把“这事以后还会用到”放到未来真正会读取的位置，但不要把 `.cs/` 或 Agent 指令写成垃圾桶。

两条去处，没有第三条：

| 去处 | 放什么 |
|---|---|
| **`AGENTS.md` / `CLAUDE.md`**（项目根，框架自动注入） | 短而稳、几乎每次启动都影响判断的规则 |
| **`.cs/notes/`** | 需要背景、证据、步骤的可复用知识 |

**不要**再建 `.cs/facts.md` 或 CodeStable 自创的“启动事实”文件。跟各 Agent 框架走：跨工具优先 `AGENTS.md`，只给 Claude 的用 `CLAUDE.md`。

## 原则

先做“六个月测试”：半年后仍会影响判断或操作，而且很难只从代码、git diff、README 重新看出来，才值得写。代码已明说的细节、一次性流水、临时情绪，不要沉淀。

Agent 指令不是知识库：只放短规则；复杂内容写 notes，指令里最多一行“做 X 前先读某 note”。

拿不准时默认写 notes，不要污染 Agent 指令。项目主线真相进 project spec；变更历史留 issue/epic；未知流程走 [maketools](maketools.md)。

notes 按 [docs](docs.md) 组织：服务复用，不复刻聊天流水。

## 行动指南

先复用框架已注入的指令与当前上下文；查重时搜 `.cs/notes/`、相关 spec 与 issue/epic。已有相近 notes 则更新原文件（文末可加日期小节），不另开一条。

写启动规则：

1. 看项目根已有哪个：`AGENTS.md`、`CLAUDE.md`（或框架约定的等价文件）。
2. 跨 Agent → `AGENTS.md`；仅 Claude → `CLAUDE.md`；都有则不重复写同一条。
3. 都没有 → 向用户说明建议建 `AGENTS.md`（或按其常用框架），确认后再建。
4. 写入前查重；只追加/更新启动必需短规则，不改无关段落。

适合进指令的例子：固定测试/构建命令、环境陷阱、目录禁区、长期命名约定、指向某 note 的一行索引。

写 notes：`.cs/notes/{NNN}-{名称}.md`（扁平序号，见 `SKILL.md`）。标题和正文保留可搜关键词。

## 产物契约

`AGENTS.md` / `CLAUDE.md`：仅追加或更新启动短规则；查重；不搞第二套事实文件。

`.cs/notes/{NNN}-{名称}.md` 推荐结构：

```markdown
# {知识点}

## 结论

{以后最应该复用的判断或做法}

## 触发场景

{这条知识在哪类任务、错误、流程或模块里适用}

## 证据和细节

{代码位置、命令、日志、截图描述、操作步骤、失败分支、危险边界}

## 相关位置

- {issue / epic / spec / 文件路径}
```

一次处理一个主题；多个无关主题拆成多条 notes 或独立的 Agent 指令更新。

## 收尾汇报

告诉用户写到了 `.cs/notes/` 还是 `AGENTS.md` / `CLAUDE.md`、为什么选择这个去处、是新增还是更新已有文件。不要把文件清单当主要汇报，先讲这条知识未来会怎么帮上忙。

## 应用场景

用户说“记下来”“沉淀一下”“留个 note”“以后每次都要知道”，或一次工作暴露出可复用坑点、操作经验、调研结论、长期约束、命令/路径/环境陷阱时使用。

不替代 Close、Make Tools 或 Spec 模式。
