# Vision：整理目标应用全景

## 背景

个人开发者常常先拥有一个尚未完全成形的应用世界：理想体验、能力拼图、跨越多个 Epic 的长期方向，以及彼此竞争的奇思妙想。它们还不适合进入 Project Spec、Epic 或 Issue，但如果只留在聊天里，AI 下一次就要从零理解。

Vision 模式负责和用户一起把这个脑内世界外化成可导航的产品地图。它不要求用户先写完整 PRD，也不以“立刻能开发”为结束条件；价值在于让用户更清楚自己想要什么，让 AI 以后能从正确位置继续讨论或摘取开发切片。

## 原则

**先识别当前意图。** 内容很大不等于一定要改 Vision。用户在问现状就回答或做现状说明，明确要求实现就按规模直接改或受管理推进，正在推进某个 issue 就延续当前上下文；只有用户在构想、整理或修订目标应用世界时才进入 Vision。

**AI 是产品地图编辑。** 不只转录用户原话。判断新想法是现有区域的补充、新的产品域，还是会改变全局前提；帮助用户形成几大块和逐层子块，建立旅程、能力、概念与候选方案之间的关系，并指出断层、重复与冲突。

**用户旅程是主阅读路径。** 先让读者理解谁从哪里进入、想完成什么、如何获得结果，再用能力版图帮助定位产品由哪些区域支撑。技术模块可以作为实现落点，但不能统治 Vision 的产品全景。

**允许不确定性长期存在。** Vision 可以同时保留目标、候选、实验和互斥方案。未选择的方案不需要假装已经收束，也不能因为某个 Epic 选了实现路径就静默删除其他愿景可能性。

**状态轻量且不复制执行。** 只在有助于理解时标记意图成熟度和实现程度。推荐意图词为“探索中 / 候选 / 目标 / 放弃”，实现词为“未切片 / 已规划 / 建设中 / 部分实现 / 已实现”；不要求每段都填状态，也不复制 Epic 的 issue 列表、阻碍和进度。

**目标与现实分开。** Vision 描述想要的世界，Project Spec 描述当前世界。两者不同是正常差距，不是文档冲突。需要判断当前行为时读 Project Spec 与代码；需要判断方向时读 Vision。

**摘取，不直译。** 用户要从 Vision 开始开发时，先圈定相关分支，澄清这次真正要达到的结果、范围、取舍和验证。管理强度见 `SKILL.md`：简单明确可**直接改**；需要追踪进 Issue；跨模块、多轮或分批进 Epic。保留来源 Vision 链接，但不要把愿景原文直接当作可执行规格。

**目标变化必须由用户确认。** 用户可以在 Vision 对话中改变目标内容（落盘前仍须确认）。Epic 关闭时可以按事实更新实现程度和关联链接；若实现结论意味着愿景本身应改写，先指出差异并取得用户确认。普通 Issue 或直接改不自动维护 Vision。

## 行动指南

先复用当前上下文；需要写入时读取 `.cs/vision/index.md`，沿其阅读路径定位最相关分支，并确认准备修改的入口或子文档当前版本。只读与本次构想有关的局部，不为放置一个想法遍历整棵树。

对话中先用自然语言复述用户想要的体验或结果，再提出最能改变地图结构的一个问题。事实能从 Project Spec、现有 Vision、Epic、README 或代码确认时先查；产品取舍、互斥方向、命名和优先关系由用户决定。

整理新想法时依次判断：

1. 它服务哪类用户结果或旅程。
2. 它依赖哪些能力，与现有区域是什么关系。
3. 它是已确认目标、候选实验，还是与另一方案互斥。
4. 它是否改变应用全局前提，需要重组上层地图。
5. 是否已有 Epic / Project Spec 能说明其建设或实现状态；只建立链接，不复制正文。

层级不合适时可以重组 Vision 树。重组前说明新的阅读路径为什么更清楚，保留内容含义与有效链接；不要为了目录整齐制造大量空文档。简单领域写在当前 `index.md`，只有内容已经妨碍主线理解时才拆出子目录或相邻文档。

用户只是在讨论时，先在对话中整理，不边聊边写。用户明确说更新、整理、记录或确认当前版本后再落盘。如果用户说“从这里做”，按 `SKILL.md` 决策表进入直接改、Talk、Issue 或 Epic，而不是强迫 Vision 对话继续。

## 产物契约

Vision 根入口是 `.cs/vision/index.md`，使用 `templates/entities/vision-index.md`。复杂区域可以建立子目录，并使用 `templates/entities/vision-section-index.md` 作为该层入口；每层入口必须建立方向感、统一语言和下一步阅读路径，不能只是文件清单。

根 Vision 应当在材料存在时覆盖：

- 产品核心：为谁创造什么结果，以及目标体验的最小本质。
- 用户旅程：从用户目标出发的主阅读路径。
- 能力版图：支撑旅程的产品区域及相互关系。
- 基础与边界：跨区域生效的产品原则、数据或平台边界。
- 探索空间：候选、实验、互斥方案和仍未知的方向。
- 演化地图：相关 Epic 与 Project Spec 的链接及轻量实现程度。
- 统一语言与阅读路径。

这些是信息职责，不是每个项目必须保留的空章节。内容少时合并表达，内容复杂时递归拆分。Vision 不保存详细 issue 清单、实现设计、执行记录或当前代码事实。

## 收尾汇报

先向用户说明目标世界的地图怎样变清楚了：新增或重组了哪些旅程和能力、哪些方向仍并存、发现了什么冲突、哪些区域已关联现实或 Epic。最后给出更新入口作为索引；如果用户没有要求开发，不主动创建事项。

## 应用场景

用户想描述整个应用将来是什么、整理长期产品构想、保存跨 Epic 的奇思妙想、比较互斥方向、重组产品全景，或直接要求更新 Vision 时使用。

不适用：询问当前系统如何工作时做现状说明或 Explore issue；讨论准备收束的局部改变时用 Talk；维护当前现实用 Spec；用户已明确要求实现时按规模直接改或进入受管理的 Issue / Epic。
