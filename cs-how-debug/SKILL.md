# cs-how-debug

debug 的核心不是"把报错消掉"，而是找到行为为什么偏离预期。没有因果链的修复，只是在赌。

## 背景

bug 最容易被修坏，不是因为代码难，而是因为压力会诱导人跳过理解：看到一个症状，脑子里冒出一个可能原因，于是顺手改一处，再跑一次；不行就再改一处。这样看起来很快，实际是在把系统变成一堆猜测的残留物。

CodeStable 要的是可关闭的事项：问题被定位，根因被解释，修复能验证，回归能挡住。debug 这件事的中心因此不是"尝试方案"，而是**建立证据链**：从触发条件开始，沿着数据、状态、调用和环境一路追到症状出现的位置，解释中间每一步为什么会发生。

## 原则

### 先把问题钉住

不要从修复开始。先把三件事说清：

- **预期行为是什么。**
- **实际行为是什么。**
- **二者在哪个最小场景里分叉。**

能复现就复现，不能稳定复现就收集更多证据，不要猜。错误信息、堆栈、日志、截图、网络请求、数据库状态、最近变更，都是证据。读错误要读完整，不要只抓一个看起来熟的关键词。

### 追因果链，不追症状

从症状位置往回追：这个坏值从哪来？谁调用了这里？状态在哪里第一次变坏？配置在哪一层丢了？环境差异从哪进入系统？

根因不是"这里报错了"，而是**有效状态第一次变成无效状态的地方**。如果只能说"可能是 X 导致 Y"，中间还有"反正大概会"这种空洞连接，就还没找到根因。链条必须能从触发条件一路讲到观察到的症状。

多组件系统尤其要在边界上取证：前端到 API、API 到服务、服务到数据库、CI 到构建脚本、构建脚本到发布环境。每个边界都问：进来的是什么，出去的是什么，配置和状态有没有按预期传过去。先定位是哪一层断了，再进那一层细查。

### 假设要能被测试

假设必须具体到可以被证伪："我认为根因是 X，因为 A 到 B 的链条里，C 在这里变坏。" 如果链条里有不确定环节，给它一个预测：如果这个解释是真的，另一个路径、日志或数据状态应该也会呈现什么现象。

测试假设时一次只动一个变量。最小实验可以是加一条诊断日志、跑一个更窄的测试、构造一个最小输入、对比一个能工作的相邻实现。不要把多个改动叠在一起"看看会不会好"；那样即使绿了，也不知道哪个解释成立。

### 修根因，不补表象

确认根因以后再修。修复要直接消除因果链里的断点，而不是在症状处加特殊分支。

能写测试就先写一个失败测试，至少要有能重复触发问题的脚本或手动步骤。修复只做一件事：让这个失败场景通过，并确认相关测试没有被打坏。不要把"顺手重构"、"顺手改命名"、"顺手优化"塞进同一次 debug。

如果一个修复看起来有效，但前面的预测不成立，停下来。那不是根因被证明了，而是另一个症状被遮住了。

### 修不动时，怀疑结构

连续两三次假设被推翻，不要继续硬试。先诊断为什么卡住：

- 证据互相矛盾：可能是对代码路径的心智模型错了，回去重读路径。
- 本地好、CI 或线上坏：优先查环境、配置、依赖、时序和数据差异。
- 每个修复都要加新分支：可能不是局部 bug，而是责任边界或接口设计错了。
- 每次修复都引出另一个地方的问题：可能是架构在用惯性硬撑。

当问题指向责任错位、接口错位或需求错位时，不要继续堆 if/else。回到设计层面，重新决定能力属于哪、接缝在哪、测试面在哪。

## 应用场景

- **测试失败、构建失败、线上异常、性能异常、集成问题**：先复现，再追因果链，再修复。
- **用户给出报错、堆栈、issue、失败截图或"为什么不行"**：把输入转成可验证的问题陈述。
- **已经尝试过修复但没有成功**：停止叠补丁，回到证据和根因。
- **问题越修越散**：检查是不是架构或责任边界错了。

不适用：

- **不是主动审计工具。** 不为了找问题而扫全仓库；debug 从一个已经出现的异常行为开始。
- **不替代需求澄清。** 如果还不知道什么叫"正确行为"，先用讨论把预期说清。
- **不鼓励临时绕过。** 可以为外部故障加重试、超时、错误提示，但前提是已经证明根因确实在外部或环境里。
